# Cool Team Docs

## 何时使用

- 新模块
- 新页面流程
- 新接口
- 新表、改表、字段调整、索引调整
- 初始化配置
- 多人协作任务
- 用户明确要求输出设计文档或开发文档

简单修复可以只输出简短说明；L2 及以上任务要使用模板化文档。
只要涉及“新功能”而不是单点修复，默认在编码前先出文档。

## 先做什么

1. 先判断这次是 L1 / L2 / L3 / L4 哪一级
2. 再决定最少要补哪些文档
3. 先把文档落点和命名定住
4. 文档够了再进入正式开发

## 文档风格

- 全部使用中文
- 优先写给初级程序员
- 少用抽象词，多用“做什么、为什么、放哪里、怎么做、注意什么”
- 一份文档只解决一类问题，不把所有内容混在一起

## 任务分级对应文档要求

### L1：小修复

常见情况：

- 改文案、样式、提示语
- 修一个明确的小 bug
- 调整一个小判断、小字段、小接口处理

文档要求：

- 不要求完整模板文档
- 但要补简短改动说明
- 要能让别人看懂你改了什么、怎么验证

### L2：普通功能

文档要求：

- 开发前至少补：需求说明、技术设计、开发实施
- 有接口变更：补 API 文档
- 有数据库变更：补数据库文档

### L3：复杂功能 / 多人协作任务

文档要求：

- 至少补：需求说明、产品设计、技术设计、开发实施、测试清单、交付说明
- 有接口变更：必须补 API 文档
- 有数据库变更：必须补数据库文档

### L4：系统级 / 跨端 / 高影响任务

文档要求：

- 必须补全套文档
- 必须写清楚跨端配合方式、依赖关系、联调顺序
- 只要有接口变化，必须补 API 文档
- 只要有数据结构或表结构变化，必须补数据库文档

## 文档输出位置

- 当前任务发生在模板仓库 `base` 目录内，默认输出到 `base/docs`
- 当前任务发生在具体业务项目中，默认输出到项目根目录 `docs`
- 文档目录结构与命名规则优先参考 `../cool-project-stack/references/doc-structure.md`
- 不要把所有内容都堆到一个总文档里，优先采用“系统长期文档 + 模块专项文档”结构

## 多期功能文档组织规范

只要一个功能会拆成两期及以上迭代，默认按“模块目录 + 模块 README + 各版本文档”组织，不要把每一期文档单独散落在 `docs/` 根目录，也不要只留下一个后来谁都看不懂的总文档。

### 1. 默认落点

多期功能文档默认放在：

- `docs/01-modules/模块名/`

不要这样做：

- 在 `docs/` 根目录平铺 `xx-v0.1.md`、`xx-v0.2.md`、`xx-v1.0.md`
- 每一期单独建一个互相没有导航关系的目录
- 只新增版本文档，不维护模块入口 README

### 2. 模块 README 是总入口，必须长期维护

模块目录下必须有 `README.md`，它不是可有可无的说明页，而是这个模块的长期总入口。

`README.md` 至少必须写清楚：

1. **模块背景**：这个模块是干什么的，解决什么问题
2. **版本规划**：现在有哪些版本，后面预计还有哪些阶段
3. **推荐阅读顺序**：第一次接手应该先看什么，再看什么
4. **当前版本入口**：当前正在开发或当前已上线的版本是哪个
5. **长期约束说明或总览入口**：哪些规则是跨版本长期有效的，或者跳到长期说明文档去看

只要模块继续演进，README 就必须同步更新，不能写完第一版后长期不管。

### 3. 哪些文档应该按版本拆

下面这些文档，默认按版本拆：

- 需求文档
- 技术设计文档
- 数据库文档
- 开发实施文档
- 如果版本差异明显，测试文档和交付文档也可以按版本拆

推荐做法是按版本建立独立文件，例如：

- `README.md`
- `v0.1-requirements.md`
- `v0.1-design.md`
- `v0.1-database.md`
- `v0.1-implementation.md`
- `v0.2-requirements.md`
- `v0.2-design.md`
- `v0.2-implementation.md`
- `v1.0-requirements.md`

如果团队更喜欢“版本子目录”方式，也可以统一放成：

- `docs/01-modules/模块名/README.md`
- `docs/01-modules/模块名/v0.1/01-requirements.md`
- `docs/01-modules/模块名/v0.1/02-design.md`
- `docs/01-modules/模块名/v0.2/01-requirements.md`

但一个模块内只能选一种版本组织方式，不要一半按平铺命名、一半按子目录命名。

### 4. 哪些文档应该保持模块级长期共用

下面这些内容，不要每个版本都复制一份，默认保持模块级长期共用：

- 模块 README
- 长期约束说明
- 稳定后的核心实体说明
- 模块总览文档

判断规则：

- **只对某一个版本成立**的内容，写进对应版本文档
- **跨多个版本都成立**的内容，写进模块级长期文档

如果一个核心实体、约束规则、模块边界已经稳定，后续版本继续引用它，不要每个版本都重新抄一遍。

### 5. 执行卡不是模块永久总文档

任务执行卡的定位是：

- 任务级文档
- 迭代级文档
- 当前阶段推进记录

它不是“这个模块永久只维护一张总执行卡”。

明确规则：

- 不同版本，要创建各自的执行卡
- 同一版本下，不同开发任务，也可以各自建执行卡
- 执行卡用于记录当前任务怎么推进，不用于替代模块 README 或长期文档

不要把模块的长期背景、所有历史版本、所有后续计划都塞进一张执行卡里。

### 6. README 必须负责串联版本文档

后续版本越来越多时，最容易乱的原因不是文档少，而是**没有统一入口**。

所以模块 README 必须承担串联责任，至少要包含：

- 版本列表
- 每个版本解决的问题
- 每个版本对应文档链接
- 建议先读哪一版，再读哪一版
- 当前版本和历史版本的区别

不允许出现这种情况：

- 版本文档平铺在目录里，但没人知道先看哪个
- 新人只能靠文件名猜顺序
- AI 只能自己扫描目录猜“最新版本”

### 7. 新版本基于旧版本增强时，禁止整份复制重写

如果后续版本是在上一版基础上增强，不要把上一版所有内容整份复制一遍再改几个字。

正确做法：

1. 在新版本文档开头明确写：**本版继承自哪个版本**
2. 明确列出：**本版新增了什么、调整了什么、废弃了什么**
3. 没变化的内容，直接引用上一版或模块级长期文档
4. 只有真正变化的部分，才在新版本里展开重写

推荐写法示例：

- `本版基于 v0.1 增强，未变更的角色定义、核心状态流转、基础数据结构，继续沿用 v0.1。`
- `本版新增 teacher 端审核节点，调整消息通知规则，其余接口约定不变。`

这样做的目的，是避免：

- 文档越来越重复
- 同一规则在不同版本里写出多个说法
- 后续维护时不知道该改哪一份

### 8. 推荐目录示例

```text
docs/
  01-modules/
    teacher-channel/
      README.md
      constraints.md
      entities.md
      v0.1-requirements.md
      v0.1-design.md
      v0.1-database.md
      v0.1-implementation.md
      v0.1-task-card.md
      v0.2-requirements.md
      v0.2-design.md
      v0.2-implementation.md
      v0.2-task-card.md
      v1.0-requirements.md
      v1.0-design.md
      v1.0-test-checklist.md
      v1.0-delivery.md
```

### 9. AI 执行硬规则

AI 在补多期功能文档时，默认必须遵守：

1. 先找到对应模块目录，没有就先在 `docs/01-modules/模块名/` 下建立模块入口
2. 先更新模块 `README.md`，再新增本期版本文档
3. 先判断哪些内容是版本专属，哪些内容应该放模块长期文档
4. 新版本如果继承旧版本，必须写清继承关系和差异点
5. 任务推进记录写进对应版本或对应任务的执行卡，不要把执行卡当模块总说明
6. 写完后检查 README 是否已经把所有版本文档串起来

只要 README 没有把版本关系讲清楚，就不能算“文档组织完成”。

## 默认输出顺序

推荐按这个顺序产出：

1. 文档导航 README（首次建立文档体系时）
2. 系统说明（首次建项目文档或系统结构明显变化时）
3. 需求说明
4. 产品设计
5. 技术设计
6. 数据库设计
7. 接口或数据结构说明
8. 开发实施
9. 联调与测试清单
10. 交付与变更说明
11. 系统功能说明（功能完成后补齐或更新）

如果任务较小，可以合并成 2 到 3 份文档，但结构不要乱。

## 开工前最少文档门禁

开始编码前，最少需要满足以下条件：

- L1：有简短改动说明或明确任务记录
- L2：至少有需求说明、技术设计、开发实施
- L3 / L4：至少有需求说明、产品设计、技术设计、开发实施
- 涉及接口联调，必须再补：接口与数据结构说明
- 涉及数据库变更，必须再补：数据库设计

如果这些文档还没有写出来，不应该直接进入正式编码阶段。

## 必须写清楚的内容

- 本次要做什么
- 为什么这样做
- 文件放哪里
- 关键接口或关键页面是什么
- 数据库怎么设计，字段是什么意思
- 如何联调
- 如何验收
- 还有哪些风险

## 开发完成后必须补的文档动作

- 更新 `交付与变更说明`
- 更新 `联调与测试清单`
- 更新 `docs/README.md` 中的目录导航、阅读顺序或新增模块入口
- 更新 `系统功能说明文档` 中本次新增或变更功能
- 如果本次改动影响系统模块边界、角色权限、部署配置、外部依赖或页面入口，继续更新 `系统说明文档`

## 完成标准

- 看文档的人不需要猜目录和落点
- 初级开发者可以按文档继续开发
- 文档和当前项目结构一致，不是空泛模板
- 文档顺序能支撑“先设计、再开发、再联调、再交付、再回写系统文档”
