# Cool Project Stack

## 适用范围

- 团队新项目初始化、功能开发、模块扩展、接口联调、设计文档、开发文档、交付说明。
- 模板仓库内存在 `cool-admin-vue`、`cool-admin-midway`、`cool-uni` 作为初始化基线时。
- 需求同时涉及规范判断、代码落点判断、文档输出、阶段化推进时。
- 项目已经进入长期维护阶段，需要先恢复项目理解、再决定是否开发时。
- 需要 Human-in-the-loop 协作，而不是 AI 一口气自动做完全部工作时。

## 这个 skill 现在的默认工作方式

这个 skill 的默认目标，不再只是“接需求后补文档再开发”，而是把整个任务改造成**阶段化执行总控流程**。

默认流程是：

1. 需求分析与任务分类阶段
2. 开发前文档产出阶段
3. 文档审核与迭代阶段
4. 分步骤开发阶段
5. 每步验收与纠偏阶段
6. 收尾补齐文档阶段

这 6 个阶段默认按顺序推进，**前一阶段没有完成，不能进入下一阶段**。

其中有些阶段必须等待人工确认：

- 阶段 1 结束后：如果任务不是 L1 小修复，默认等待用户确认任务理解、分级和影响范围
- 阶段 2 结束后：默认等待用户审核开发前文档
- 阶段 3 中：每次文档被修改后，默认继续等待用户确认，没确认前不能进入开发
- 阶段 4 中：每一个开发步骤完成后，默认进入阶段 5 等待人工验收或继续指令
- 阶段 5 中：如果该步存在理解偏差、实现偏差、范围偏差，先回改文档，再继续下一步开发
- 阶段 6 结束后：等待用户确认交付内容是否完整

一句话：**默认是“文档先对齐、步骤分开做、每步可检查、出错先回文档”，不是无人审核的一次性生成。**

## 新人先做什么

激活后先完成 6 件事：

1. 判断当前需求属于 `cool-admin-vue`、`cool-admin-midway`、`cool-uni` 中的哪一端，或是否为跨端任务。
2. 先看当前项目已有结构，再看 `base` 模板，不允许直接跳到“通用最佳实践”。
3. 如果项目已经进入长期维护阶段，或任务属于复杂功能、高影响修改、核心模块修改，先恢复项目理解：
   - 先看 `docs/README.md`
   - 再看系统说明相关文档
   - 再看“项目与 base 差异”相关文档
   - 再看核心模块或当前模块相关文档
4. 按目标端继续读取相邻子 skill：
   - Vue：`../cool-admin-vue-dev/SKILL.md`
   - Midway：`../cool-admin-midway-dev/SKILL.md`
   - Uni：`../cool-uni-dev/SKILL.md`
5. 只要任务不是 L1 小修复，就继续读取 `../cool-team-docs/SKILL.md`
6. 如果任务涉及核心模块、高风险流程、公共结构，不要直接开发，先做检查，再决定是否进入编码。

如果任务属于“新功能、新模块、新页面流程、新接口、数据库调整、多人协作任务”中的任意一种，默认视为需要前期文档，不能直接跳过文档阶段开始编码。

## 常见任务怎么选 skill

先看你现在要做什么：

| 任务类型 | 先看哪个 skill | 下一步做什么 |
| --- | --- | --- |
| 后端 / 接口 / 实体 / 服务 | `../cool-admin-midway-dev/SKILL.md` | 先找同类模块，再决定是否补 API / 数据库文档 |
| 后台 / CRUD / 表单 / 表格 / 页面 | `../cool-admin-vue-dev/SKILL.md` | 先找同类页面，再决定是否补需求 / 技术 / 实施 |
| Uni / 移动端页面 / 路由 / 登录态 | `../cool-uni-dev/SKILL.md` | 先看页面入口和接口接入方式 |
| 小修复 | 对应端 skill | 先看同类代码，改完走检查清单 |
| 新功能 | 对应端 skill + `../cool-team-docs/SKILL.md` | 先补需求、技术、实施，再进入阶段化开发 |
| 跨端任务 | 先看本文件，再看各端 skill | 默认顺序：Midway → Uni → Vue |
| 不知道还要不要装别的外部 skill | `./references/external-skills.md` | 先看“必装 / 推荐 / 按需” |

一句话判断：

- 只是改一点点：先找对应端 skill
- 只要是新功能：加上文档 skill
- 只要跨两个及以上端：先按跨端任务处理

## 任务分级规则

### L1：小修复

常见情况：

- 改文案、样式、提示语
- 修一个明确的小 bug
- 调整一个小判断、小字段、小接口处理
- 不新增模块，不改主流程，不改数据库

要求：

- 可以直接改代码
- 但也要写清楚改了什么
- 提交前做最小自测
- 如果用户明确要求走阶段化流程，也可以按阶段执行，但通常不强制等待多轮确认

### L2：普通功能

常见情况：

- 新增一个普通页面、表单、列表
- 在一个模块内新增一段业务逻辑
- 新增或调整接口，但范围可控

要求：

- 开发前至少补：需求、技术、实施
- 涉及接口变更：补 API 文档
- 涉及数据库变更：补数据库文档
- 默认先完成文档审核，再开始开发

### L3：复杂功能 / 多人协作任务

常见情况：

- 新模块、新流程、跨角色流程
- 前后端联动明显
- 多人协作、分阶段交付
- 影响多个目录、多个页面、多个接口

要求：

- 需要较完整文档链路
- 至少包括：需求、产品、技术、实施、测试、交付
- 涉及接口必须补 API 文档
- 涉及数据库必须补数据库文档
- 默认拆成多个开发步骤，每步都要可验收

### L4：系统级 / 跨端 / 高影响任务

常见情况：

- 跨端任务
- 登录、权限、核心流程、主数据调整
- 系统结构调整
- 同时影响后端、后台、Uni 中两个及以上端

要求：

- 必须补全套文档
- 必须写清楚各端怎么配合
- 只要有接口变化，必须补 API 文档
- 只要有数据结构或表结构变化，必须补数据库文档
- 交付前必须补联调和交付说明
- 开发前先恢复项目理解，不能只根据当前需求直接改代码
- 如果涉及核心模块、高风险流程、公共结构，先完成检查，再决定是否开发
- 默认强制走“阶段审核 + 分步实施 + 每步验收 + 偏差回改文档”流程

## 长期维护项目附加规则

只要项目已经进入长期维护阶段，处理复杂功能或高影响修改时，必须额外执行下面几条：

1. 不能只看当前需求，必须先恢复项目理解。
2. 恢复项目理解时，优先顺序是：
   - `docs/README.md`
   - 系统说明相关文档
   - 项目与 `base` 差异相关文档
   - 核心模块或当前模块相关文档
3. 如果任务涉及核心模块、高风险流程、公共结构，先做检查，再决定是否开发。
4. 如果当前项目已经明显偏离 `base`，优先以当前项目为准，不要机械按 `base` 回退。
5. 回答问题、给方案、开始开发前，都要先恢复项目理解，不要只靠临时上下文。
6. 恢复项目理解完成前，不要进入正式开发阶段。

## 跨端任务处理顺序

默认顺序：

1. 服务端 `cool-admin-midway`
2. 用户端 `cool-uni`
3. 后台端 `cool-admin-vue`

原因：

- 服务端接口和数据结构要先稳定
- 移动端通常先依赖服务端接口
- 后台端更多是管理、运营、配置与验证入口

如果任务只涉及单端，则只读取对应子 skill，不要一次性加载全部细节。

## 团队首次使用初始化

如果是团队第一次使用这套基座，或者当前环境还没装团队约定的外部 skill，先做下面几步：

1. 读取 `./references/external-skills.md`
2. 先安装“必装”部分
3. 确实需要对应能力时，再安装“推荐”或“按需”部分
4. 如果要一次性安装默认清单，再执行 `bash skills/cool-project-stack/scripts/install-team-skills.sh`
5. 重启 Codex，让新安装的 skill 生效
6. 如果后续真要用 `browser-use`，再执行 `browser-use doctor`
7. 如果后续真要用 `notebooklm`，再按其 skill 说明完成认证和 notebook 配置

注意：

- 这些外部 skill 默认安装到个人 AI 环境中，不安装到每个项目仓库里
- `base/skills` 负责约定“什么时候该调用什么 skill”，不是把第三方 skill 复制进项目
- 不要给新人“第一次就全部装完”的压力

## 外部 Skill 调用时机

完整安装说明和触发矩阵见 `./references/external-skills.md`，常用规则如下：

- 需要搜索、补装、升级其他 skill 时，调用 `find-skills`
- 需求复杂、跨端、多文档、多阶段、多人协作时，优先调用 `planning-with-files`
- 需要做品牌页、专题页、落地页、强视觉页面，且希望去掉 AI 塑料感时，调用 `frontend-design`
- 需要更强的视觉提案、品牌调性、UI/UX 打磨时，调用 `ui-ux-pro-max`
- 需要浏览器自动化、录制流程、截图验收、多步骤联调时，调用 `browser-use`
- 需要从大量文档、资料、知识库中做带来源的研究时，调用 `notebooklm`
- 需要从顶级专家视角做产品、架构、策略破局时，调用 `best-minds`

例外规则：

- 现有项目已经有稳定设计系统或后台风格时，不默认调用 `frontend-design` 和 `ui-ux-pro-max` 去重做视觉语言
- 标准后台 CRUD、普通表单页、简单修复任务，不因为“能设计”就强行触发高视觉 skill

## 规范优先级

规范冲突时，必须按下面的顺序判断，并在回答中说明“本次依据是什么”：

1. 用户当前明确提出的要求
2. 当前项目中已经存在的代码结构、命名方式、规则文件、实现方式
3. `base` 模板中的同类实现
4. 官方文档
   - `https://vue.cool-admin.com`
   - `https://node.cool-admin.com`
   - `https://uni-docs.cool-js.com`
5. 团队级参考 `./references/base-repo-conventions.md`
6. 通用工程最佳实践

注意：

- 团队级参考是补充说明，不覆盖当前项目已有规则。
- `cool-uni` 没有本地 `.cursor/rules` 时，优先级变成：用户要求 > 当前项目代码 > `cool-uni` 模板 > 官方文档 > 团队级参考 > 最佳实践。
- 命名规则优先参考 `./references/naming-conventions.md`
- 代码组织、模块复用、中文注释要求优先参考 `./references/code-organization.md`

## base 基线

把下面三套模板视为默认初始化基线：

- Vue 基线：`../../cool-admin-vue`
- Midway 基线：`../../cool-admin-midway`
- Uni 基线：`../../cool-uni`

处理新项目时：

- 如果当前项目和 `base` 一致，直接沿用 `base` 风格继续扩展。
- 如果当前项目已经偏离 `base`，优先尊重当前项目已有实现，不强行改回模板。
- 如果当前项目某类功能还没实现，优先参考 `base` 中最接近的结构和写法。
- 只要发现当前项目与 `base` 存在明显差异，先确认差异原因和当前项目约束，再决定是否参考 `base`。

## 默认阶段化工作流

下面这部分是本 skill 的核心流程。默认按阶段推进，不要跳步。

### 阶段 1：需求分析与任务分类

先输出以下内容，再决定要不要继续：

- 需求理解
- 目标端识别
- 任务等级（L1 / L2 / L3 / L4）
- 影响范围
- 模块落点
- 关键文件路径
- 依赖的现有模块或页面
- 风险点和待确认点
- 当前项目是否已经偏离 `base`
- 本次先依据了哪些系统说明、差异记录、模块文档

阶段动作：

1. 先判断任务落在哪些端。
2. 先看当前项目已有结构，再看 `base`。
3. 长期维护项目先恢复项目理解。
4. 给出任务分级和原因。
5. 列出是否需要开发前文档。
6. 只要任务需要走阶段化流程，就创建或初始化任务执行卡，先记录任务名称、当前阶段、任务等级、涉及端、需求摘要、待确认点、下一步动作。

默认确认规则：

- L1 小修复：如果需求非常明确，可直接进入开发或最小文档补充
- L2 / L3 / L4：默认等待用户确认需求理解、任务分级、影响范围后，再进入阶段 2
- 已创建任务执行卡的任务，在等待确认时要把执行卡状态更新为“待确认”

### 阶段 2：开发前文档产出

先确认文档根目录：

- 当前任务发生在模板仓库 `base` 目录内，默认输出到 `base/docs`
- 当前任务发生在具体业务项目中，默认输出到项目根目录 `docs`
- 如果任务是多期迭代功能，文档默认收敛到同一个模块目录 `docs/01-modules/模块名/`，由模块 `README.md` 统一维护版本规划、阅读顺序、当前版本入口和长期约束入口

项目级长期文档处理规则：

- 如果文档导航 `docs/README.md` 不存在，先补
- 如果项目级 `系统说明文档` 不存在，先补
- 如果项目已经进入长期维护阶段，复杂功能或高影响修改开始前，先看系统说明文档、项目与 `base` 差异文档、核心模块相关文档
- 如果任务涉及核心模块、高风险流程、公共结构，开发前先完成检查，不要直接进入编码
- 如果本次新增了模块、页面入口、角色流程、外部依赖或部署约定，开发完成后必须同步更新 `系统说明文档`
- 如果本次新增、确认或扩大了项目与 `base` 的差异，开发完成后必须同步更新差异记录
- 如果本次新增或变更了可被业务、测试、运营直接使用的功能，开发完成后必须同步更新 `系统功能说明文档`
- 如果本次新增了模块，或明显扩展了模块现有能力，开发完成后必须同步更新对应模块 `README.md`、模块能力说明或模块清单，让后续人和 AI 能快速知道这个模块已经具备什么能力、哪些文件可以复用

以下文档是开发前的标准产物，不要等到开发完成后才补：

- L2：至少补 `需求说明文档`、`技术设计文档`、`开发实施文档`
- L3 / L4：至少补 `需求说明文档`、`产品设计文档`、`技术设计文档`、`开发实施文档`

按场景追加：

- 涉及新表、改表、字段变更、索引、状态字段、数据迁移：必须补 `数据库设计文档`
- 涉及前后端联调、新接口、接口字段调整：必须补 `接口与数据结构说明`
- 涉及复杂页面流程、角色流程、状态流转：在技术设计或开发实施文档中写清页面流程与状态流转

阶段动作：

1. 先产出开发前文档。
2. 再拆出推荐开发步骤。
3. 标记每一步的目标、边界、验收点。
4. 写清楚哪些点仍待用户确认。
5. 把开发前文档清单、当前结论、待确认问题、下一步动作同步到任务执行卡。

默认确认规则：

- 阶段 2 结束后，默认必须等待用户审核文档
- 用户没有确认前，不能进入阶段 4 开发
- 等待审核时，要把任务执行卡状态更新为“待确认”

### 阶段 3：文档审核与迭代

这个阶段的目标不是“快速跳过”，而是让文档真正可执行。

阶段动作：

1. 用户审核需求、方案、步骤拆分。
2. AI 根据问题或反馈回改文档。
3. 每次改完都重新同步受影响文档。
4. 确认“文档版本”和“接下来要开发的步骤”已经一致。
5. 每次文档有回改时，同步更新任务执行卡里的当前结论、待确认问题、下一步动作、相关文档清单。

默认确认规则：

- 文档每次被修改后，默认继续等待用户确认
- 没有明确确认“可以开发”，不能进入阶段 4
- 等待再次确认期间，任务执行卡状态保持为“待确认”

文档回改要求：

- 如果需求理解变了，要回改：需求说明文档、产品设计文档、开发实施文档
- 如果接口或字段变了，要回改：技术设计文档、接口与数据结构说明、数据库设计文档（如有）
- 如果步骤顺序变了，要回改：开发实施文档、测试检查项、交付说明草稿
- 如果影响范围扩大或缩小，要回改：需求说明文档、技术设计文档、系统说明文档相关章节（如已写）

### 阶段 4：分步骤开发

满足以下条件后才能进入编码阶段：

- 已明确本次功能属于哪一端
- 已完成任务分级
- 已核对当前项目与 `base` 中的同类实现
- 长期维护项目下，已恢复系统说明、项目与 `base` 差异、核心模块相关文档这三类理解
- 已完成最少必需文档
- 关键接口、关键字段、关键页面入口没有悬空
- 文档已获用户确认
- 如果涉及核心模块、高风险流程、公共结构，已先完成检查

如果文档仍有明显空白项，只能继续补文档或补澄清，不能直接开始写代码。

开发约束：

- 不发明仓库里不存在的目录体系。
- 先检查命名是否符合 `./references/naming-conventions.md`
- 代码组织、模块拆分、模块复用、中文注释要求，先检查是否符合 `./references/code-organization.md`
- 新增代码前先检查当前项目和 `base` 是否已有同类模块、相近页面、可复用组件、可复用 service、可复用 hooks、可复用工具方法，能复用优先复用，不能复用再新增。
- 新增模块或明显扩展模块能力时，开发完成后必须同步更新模块 README、功能说明或模块清单，让后续 AI 能快速知道现有能力和可复用入口。
- 关键业务逻辑、状态流转判断、核心字段含义、特殊兼容处理、复用关系等位置，必须补清晰明了的中文注释，让初级程序员能看懂“这里为什么这么写”。
- 不把页面逻辑、业务逻辑、接口调用、类型定义、配置常量等大量逻辑堆进单个文件；页面 / service / hooks / utils / types / constants 等职责应按模块和职责拆分。
- 后端也不能把全部逻辑塞进单个 service 或 controller；controller 负责请求入口，service 负责业务编排，复杂规则、状态处理、公共能力要按职责拆分。
- 不把其他技术栈的习惯硬套进当前端。
- 不为了“看起来高级”而过度抽象。
- 缺少必要信息时，只问会影响落地的最小问题。
- 尽量复用 `base` 和当前项目已有模块，不轻易新增第二套模式。
- 必须按文档中约定的步骤开发，不要一口气做完整包。

开发方式：

1. 每次只推进一个明确步骤。
2. 每个步骤完成后，输出本步改动、影响范围、待验收点。
3. 同步更新任务执行卡中的已完成内容、当前阶段、当前状态、下一步动作。
4. 默认进入阶段 5，等待人工验收或继续指令。

### 阶段 5：每步验收与纠偏

这个阶段用于把“写完一段代码”变成“确认这一段做对了”。

阶段动作：

1. 对照本步文档目标检查实现是否一致。
2. 对照验收点检查代码、接口、页面、状态流转。
3. 由人工确认是否通过、是否继续下一步。
4. 如果发现理解错误、范围错误、步骤错误，先回改文档，再改代码。

默认确认规则：

- 每个开发步骤完成后，默认等待用户验收
- 用户未确认通过前，不进入下一步开发
- 等待验收时，任务执行卡状态更新为“待确认”；验收通过后，再更新为下一阶段对应状态

纠偏规则：

- 如果只是实现细节偏差：先回改当前步骤相关的开发实施文档，再调整代码
- 如果是需求理解偏差：回到阶段 3，先回改需求与设计文档
- 如果是接口、字段、状态定义偏差：先回改技术设计、API、数据库文档，再调整代码
- 如果某一步被推翻：至少回改该步对应的开发实施文档、测试检查项、交付说明，并检查前序文档是否也被影响，同时在任务执行卡里补齐“被推翻原因、已回改文档、下一步动作”

一致性要求：

- 文档和代码要始终保持一致
- 不能出现“文档还是旧的，代码已经改成新的”
- 不能出现“代码回滚了，但文档还保留旧结论”

### 阶段 6：收尾补齐文档

开发结束后，不是直接宣布完成，而是进入收尾阶段。

阶段动作：

1. 补齐测试、联调、交付文档。
2. 同步更新 `docs/README.md` 中新增模块、阅读顺序或文档入口。
3. 同步更新 `系统说明文档` 中受影响的章节。
4. 同步更新 `系统功能说明文档` 中新增或变更功能的入口、流程、限制条件、权限要求。
5. 检查开发前文档、代码实现、测试记录、交付说明是否一致。
6. 把最终结论、最终产物、收尾结果同步到任务执行卡，并把状态更新为“已完成”或“已暂停”。

默认确认规则：

- 阶段 6 结束后，等待用户确认交付内容是否完整

收尾产物：

- 改动说明
- 测试与联调记录
- 已知风险
- 后续优化项
- 更新后的文档入口

文档输出规则与模板由 `../cool-team-docs/SKILL.md` 负责。

## 阶段切换硬规则

为了避免 AI 直接跳步，默认遵守下面规则：

1. 没做任务分级，不进入开发前文档阶段。
2. 该补的开发前文档没补齐，不进入开发阶段。
3. 用户没确认文档，不进入开发阶段。
4. 当前步骤没验收，不进入下一开发步骤。
5. 需求、接口、字段、流程被推翻时，先回改文档，再继续开发。
6. 文档和代码不一致时，优先先修一致性，再继续推进。
7. 长期维护项目里，没恢复项目理解前，不进入正式开发。

## 输出风格

- 全部使用中文。
- 优先写给初级程序员看懂。
- 少用空泛术语，多用“先做什么、再做什么、放哪里、怎么做、注意什么”。
- 文件路径、模块名、页面名、接口名尽量写清楚。
- 说明阶段时，优先给出“当前在哪一阶段、下一步要等什么确认”。

## 完成标准

一次任务完成时，至少满足以下条件：

- 已明确本次采用的规范来源。
- 已明确任务等级和对应文档要求。
- 代码落点和结构与当前项目或 `base` 保持一致。
- 开发前文档阶段没有被跳过。
- 重要设计决策有说明。
- L2 及以上任务已包含对应的前期文档、联调文档和交付说明。
- 每个关键开发步骤都具备可验收结果。
- 如果某一步被推翻，相关文档已经同步回改。
- 文档与代码保持一致。
- 如果存在阻塞，已明确阻塞点、影响范围和下一步最小动作。
