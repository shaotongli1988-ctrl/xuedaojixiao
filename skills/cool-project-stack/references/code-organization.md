# 代码组织与模块复用规范

这份规则专门回答一件事：

**代码到底应该怎么组织，才既适合初级程序员长期阅读，也方便后续 AI 快速理解、继续复用和维护。**

如果只追求“先跑起来”，很容易出现下面这些问题：

- 一个文件越写越大，后来没人敢动
- 页面里混着接口、状态、计算、常量、类型，读起来很乱
- 明明项目里已经有类似模块，又重新写了一套
- 新增能力做完以后，没有地方记录，后面的人和 AI 根本不知道可以复用
- 关键逻辑没有说明，接手的人只能靠猜

所以这份规则不是“锦上添花”，而是默认基础要求。

---

## 一、总原则：代码可读性优先

先记住一句话：

**生成或修改代码时，第一优先级不是“看起来高级”，而是“让初级程序员能看懂、能接着改、改错概率低”。**

具体要求：

1. 命名要直接表达含义
   - 变量名、函数名、模块名，要让人看到名字就知道大概是干什么的
   - 不要为了简短大量使用只有作者自己懂的缩写
   - 不要把多个意思混在一个名字里

2. 一个文件、一个函数、一个模块，都要尽量职责清楚
   - 读代码的人应该能快速判断：这个文件主要负责什么，不负责什么
   - 如果一个文件同时承担页面展示、状态拼装、接口请求、字段转换、权限判断，通常就是拆分不够

3. 少做过度抽象
   - 如果只是一个普通业务场景，不要为了“显得高级”强行引入难理解的模式
   - 三五处以内的相似逻辑，优先先保持直白；只有在重复已经明显影响维护时，再抽公共层
   - 不要把简单流程拆成一串难追踪的包装函数

4. 代码结构要有“顺着读就能懂”的感觉
   - 页面入口先放页面主逻辑
   - 业务规则放到业务层
   - 工具函数放到工具层
   - 类型、常量、配置单独放清楚

一句话判断标准：

**一个刚接手项目的初级程序员，能不能在较短时间内看懂“这段代码放哪、为什么这么写、接下来应该去哪里改”。**

如果答案是否定的，就说明代码组织还不够好。

---

## 二、禁止把一个功能的大量逻辑堆进单个文件

这是强规则。

**不允许为了省事，把一个功能的大量逻辑全部塞进单个 `view`、`service`、`controller`、`page` 或其他单一文件。**

常见错误写法：

- 一个页面文件里同时写：页面结构、表单校验、接口请求、字段转换、状态映射、常量、类型定义、权限判断
- 一个后端 `service` 文件里同时写：参数校验、业务编排、数据库查询、状态流转、通知发送、日志拼装、兼容处理
- 一个 `controller` 文件里同时承担：入参处理、业务规则、数据库操作、响应格式转换

这样做的问题很直接：

- 文件会越来越大，后面很难定位问题
- 改一个点容易误伤别的逻辑
- 新人读代码时不知道从哪里开始
- AI 下一次接手时，也很难快速识别哪些逻辑应该复用、哪些逻辑应该独立修改

### 正确做法

一个功能里，下面这些内容应该按职责拆开，而不是混在一起：

- 页面展示逻辑
- 业务逻辑
- 接口调用
- 类型定义
- 配置常量
- 可复用 hooks
- 公共工具方法
- 状态映射或枚举定义
- 特殊兼容处理

### 判断是否该拆分的简单标准

出现下面任意几种情况，通常就该拆分了：

- 同一个文件已经同时处理 3 类以上职责
- 同一个文件已经超过“新人能一眼看明白主流程”的范围
- 同一段逻辑在别的页面、别的模块也可能复用
- 修改一个字段或一个判断，需要在同一个大文件里来回跳很多段代码
- 页面文件里已经出现大量与 UI 无关的计算和数据整理
- `service` 或 `controller` 已经变成“什么都往里塞”的总入口

## 三、模块化拆分规则

默认优先按模块组织代码，而不是按“随手放到哪里方便”组织。

### 1. 先按模块，再按职责拆分

推荐思路：

- 先确定这是哪个业务模块
- 再在模块内部按职责拆分页面、服务、hooks、utils、types、constants 等内容

也就是说，先回答：

- 这段代码属于哪个模块？
- 它在这个模块里承担什么职责？

而不是先随手写一个大文件，再想以后怎么补结构。

### 2. 前端推荐拆分思路

前端一个模块中，常见职责应尽量分清：

- `pages` / `views`：页面结构、交互入口、页面级状态组织
- `components`：模块内可复用组件
- `services` / `api`：接口调用
- `hooks` / `composables`：可复用状态逻辑、交互逻辑
- `types`：类型定义、字段结构
- `constants`：常量、枚举映射、固定配置
- `utils`：纯工具函数、格式化处理

注意：

- 页面文件负责“把这些能力组织起来”，不要自己吞掉全部细节
- 纯展示组件不要承载过多业务决策
- 纯工具函数不要反过来依赖页面上下文

### 3. 后端推荐拆分思路

后端也一样，不能把所有内容堆到一个 `service` 或 `controller`。

常见职责应尽量分清：

- `controller`：接请求、收参数、返回结果
- `service`：业务编排、规则处理
- `entity` / `model`：数据实体定义
- `dto` / `types`：入参出参结构、类型说明
- `constants` / `enums`：状态常量、枚举定义
- `utils`：无业务上下文依赖的通用方法
- 模块内可复用子服务：把复杂规则拆出去，而不是全塞主 service

注意：

- `controller` 不要直接承担大量业务逻辑
- `service` 不要既做流程编排，又做全部底层细节，还做全部兼容分支
- 数据库访问、状态流转、消息触发、外部调用，如果已经明显是不同职责，应适度拆开

### 4. 公共能力要抽到公共层，但不要乱抽

能确定会跨模块复用的能力，应该抽到公共层。

例如：

- 多个模块都共用的组件
- 多个页面都会用到的 hooks
- 多个业务都使用的 service 能力
- 稳定通用的工具方法
- 统一的状态枚举、字段映射、格式化规则

但要注意：

- 只有当前模块自己用的一次性逻辑，不要为了“可能以后会复用”就提前抽成公共层
- 公共层一旦建立，就要保证命名、入口、用途清楚
- 抽公共层的目标是减少重复和降低理解成本，不是增加跳转成本

## 四、优先复用已有模块，不重复造轮子

开发前必须先做一件事：

**检查项目里是否已经有同类模块、相近页面、可复用组件、可复用 service、可复用 hooks、可复用工具方法。**

这是默认动作，不是可选动作。

### 开发前至少要检查这些内容

1. 有没有相似页面或相似模块
   - 看页面结构是不是已经有现成模式
   - 看列表、表单、详情、弹窗是不是已有统一写法

2. 有没有可复用组件
   - 不要因为“这个组件我自己再写一个更顺手”就重复造

3. 有没有现成 service 或接口封装
   - 同类请求处理、字段整理、状态转换，优先沿用已有写法

4. 有没有可复用 hooks / composables
   - 尤其是查询条件、列表状态、弹窗状态、表单提交流程等常见逻辑

5. 有没有现成工具方法和常量定义
   - 状态映射、时间格式化、金额处理、字段转换等优先复用

### 复用规则

- 能直接复用：优先直接复用
- 不能直接复用但只差一点：优先在原能力上扩展
- 确实不适合复用：再新增

### 禁止的做法

- 为了“更优雅”重复造一套功能差不多的组件或工具
- 明明已有同类 service，却重新起一套名字和一套调用方式
- 明明已有同类页面结构，却为了个人习惯再写一套组织方式

一句话：

**先复用已有模式，再新增新模式。新增必须有理由。**

## 五、已实现模块必须能被后续 AI 快速找到

如果模块做出来了，但文档没更新，后续人和 AI 就会以为“项目里还没有这项能力”，然后继续重复建设。

所以这里有一条非常关键的规则：

**只要新增了模块，或者明显扩展了某个模块的能力，就必须同步更新对应的模块 README、功能说明或模块清单。**

### 至少要补哪些信息

应让后续接手的人和 AI 能快速知道：

- 这个模块叫什么
- 解决什么问题
- 放在哪个目录
- 现在已经具备哪些能力
- 哪些页面、组件、service、hooks 可以复用
- 当前推荐先看哪几个文件
- 如果是多期模块，现在最新做到哪一版

### 推荐更新位置

根据项目现有文档结构，优先更新这些地方：

- 模块目录下的 `README.md`
- 模块功能说明文档
- `docs/README.md` 中的文档入口或阅读顺序
- 系统功能说明 / 模块清单 / 系统总览中与该模块有关的部分

### 为什么这条规则很重要

因为后续 AI 是否能复用已有能力，很多时候不只取决于代码在不在，还取决于：

- 入口清不清楚
- 模块边界清不清楚
- 现有能力有没有被写出来
- 推荐阅读顺序有没有写出来

如果这些信息没有沉淀，后续维护者就只能一遍遍重新搜索、重新猜。

## 六、中文注释要求

这是强规则，而且非常重要。

**关键业务逻辑、状态流转判断、核心字段含义、特殊兼容处理、复用关系等位置，必须有清晰明了的中文注释。**

### 注释不是越多越好，而是关键处必须说清楚

不是要求每一行代码都加注释。

而是要求在下面这些地方，不能省略说明：

1. 关键业务规则
   - 例如：为什么这个角色只能看到这类数据
   - 例如：为什么这个状态下允许编辑，另一个状态下不允许

2. 状态流转判断
   - 例如：订单从哪个状态可以流转到哪个状态
   - 例如：审核驳回后为什么要保留某个字段

3. 核心字段含义
   - 例如：某个状态码、标记位、来源字段到底表示什么
   - 例如：看起来相似的两个字段，实际业务区别是什么

4. 特殊兼容处理
   - 例如：为什么这里要兼容旧数据
   - 例如：为什么这里不能直接删掉某个分支

5. 复用关系或依赖关系
   - 例如：这个方法为什么必须和另一个模块保持同样规则
   - 例如：这个字段为什么要和后台配置保持一致

### 中文注释要写什么

注释要优先帮助初级程序员理解下面两件事：

- **这里在做什么**
- **这里为什么要这么做**

尤其第二点更重要。

如果只是把代码字面意思再翻译一遍，这种注释价值很低。

### 推荐注释风格

推荐写成这种风格：

- 先写业务背景
- 再写为什么这么判断
- 必要时补充上下游依赖

例如：

```ts
// 这里必须在提交前再次判断课程状态，
// 因为课程可能在用户停留当前页面期间被后台下线。
// 如果不二次校验，前端会把已经失效的课程继续提交到订单流程里。
```

再例如：

```ts
// `sourceType` 不是页面来源，而是线索归因来源。
// 后端后续会根据这个字段决定是否进入渠道分配流程，
// 所以这里不能随意改成前端自己的临时枚举值。
```

### 禁止的注释写法

- 只留模糊英文缩写，让新人看不懂
- 只写“特殊处理”“兼容一下”“这里不要动”这种没有上下文的话
- 注释里不说明原因，只给结论
- 关键逻辑完全没有注释，只能靠猜

### 注释和代码要一起维护

- 代码改了，注释也要一起改
- 注释过期比没有注释更危险
- 如果一段逻辑已经删掉，对应旧注释也要同步清理

## 七、适合团队和 AI 协作的落地要求

为了让团队成员和 AI 都能持续接手，默认再补 4 条落地要求。

### 1. 新增代码前，先判断应该放进现有模块还是新建模块

不要一上来就新建目录。

先判断：

- 这是现有模块的扩展吗
- 这是现有流程上的增强吗
- 这只是某个模块里的一个子能力吗

如果本质上还是同一个模块，就优先放回原模块继续维护。

### 2. 新增模块时，目录名、入口文件、README 要一起补

不要只建代码目录，不补说明。

至少让后续接手者知道：

- 模块入口在哪
- 模块里有哪些子能力
- 当前推荐从哪里开始看

### 3. 明显扩展模块能力时，要回写文档

例如新增了：

- 一个新的页面能力
- 一个新的 service 能力
- 一组新的状态流转
- 一类新的复用 hooks / 组件

都应该更新模块 README 或功能说明，不要只改代码不留入口。

### 4. 回答和交付时，要明确说明复用了什么、新增了什么

这样后续维护者才知道：

- 哪些是沿用已有模块
- 哪些是本次新增
- 下次类似需求该优先看哪里

## 八、开发时可直接执行的检查清单

开始写代码前，至少快速检查下面这些问题：

1. 这个需求能不能放进现有模块，而不是新开一套
2. 项目里有没有同类页面、同类组件、同类 service、同类 hooks
3. 这段逻辑是页面职责、业务职责、接口职责，还是公共职责
4. 有没有把类型、常量、接口调用、工具方法全塞进同一个文件的倾向
5. 哪些关键业务规则需要补中文注释
6. 如果这次新增了模块或扩展了能力，开发完成后要更新哪个 README 或功能说明

提交前再检查：

1. 初级程序员能不能顺着目录和命名快速看懂
2. 单个文件是否承担了过多职责
3. 是否优先复用了已有能力，而不是重复造轮子
4. 关键业务判断、状态流转、字段含义、兼容逻辑，是否已经补了清晰中文注释
5. 模块文档、README、功能说明是否已经同步更新

---

## 九、一句话记住

**代码不是只要能跑就行，还要让初级程序员看得懂、让团队接得住、让后续 AI 找得到、复用得上、继续改得动。**
