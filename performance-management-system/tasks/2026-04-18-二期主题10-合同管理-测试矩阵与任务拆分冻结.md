## 已冻结验证门槛

1. 后端 `jest` 覆盖合同状态流、删除限制和权限拒绝
2. 前端必须通过 `type-check` 和 `build`
3. smoke 至少覆盖 `HR` 主链成功、`draft` 删除成功、经理 / 员工拒绝

## 最小测试矩阵

| 类型 | 最少覆盖 |
| --- | --- |
| 正常 | `HR` 可分页、查看详情、创建、编辑，且 `draft` 删除成功 |
| 异常 | 经理 / 员工访问 `page / info / add / update / delete` 被拒绝；非法 `type / status`、`endDate <= startDate`、无效 `employeeId / departmentId` 被拒绝 |
| 边界 | `title / contractNumber / position` 长度边界、`probationPeriod` 整数边界、`salary` 数值 / 空值边界、分页参数边界、可空字段缺失边界 |
| 权限 | 首批仅 `HR` 有入口与动作权限；经理 / 员工无入口；`salary` 仅 `HR` 明文 |
| 状态 | `draft -> active -> expired / terminated` 合法；`active` 不可删除；`expired / terminated` 不可编辑、不可删除、不可恢复；不接受 `signing / signed / approving` 等越界状态 |
| 数据 | 列表 / 详情不得返回附件全文、PDF 预览流、签名图片、签署轨迹、审批历史、证照号、银行账号和员工额外联系方式 |

## 后续实现拆分顺序

1. 是否拆成后端、前端、联调 3 个窗口：`是`
2. 推荐顺序：
   - 窗口 A 后端：`contract` entity / service / controller / 最小 `jest`
   - 窗口 B 前端：类型、service、列表 / 详情 / 创建 / 编辑页与 `HR` 权限显隐
   - 窗口 C 联调：菜单 / 权限 / 运行时初始化、真实 smoke、文档收口
3. 串并行约束：
   - A/B 可并行
   - C 必须在 A/B 完成后收口真实联调

## 窗口A/B交叉核验

1. 已一致部分：
   - 首批主题名固定为 `合同管理`
   - 首批业务域固定收窄为“合同台账主链”
   - 首批不纳入 `PDF 预览 / 电子签 / 签名板 / 自动审批流 / 归档下载`
   - 权限固定为首批仅 `HR`
   - 状态固定为 `draft / active / expired / terminated`
2. 用户已明确澄清首批包含 `delete`，因此窗口C测试矩阵按“`draft` 可删、非 `draft` 不可删”冻结。
