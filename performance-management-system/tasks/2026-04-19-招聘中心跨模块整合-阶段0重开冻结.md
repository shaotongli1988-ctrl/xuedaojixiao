## 背景

1. 当前招聘中心已按独立主题冻结并实现：
   - 主题 `8`：面试 `interview`
   - 主题 `12`：人才资产 `talentAsset`
   - 主题 `15`：简历池 `resumePool`
   - 主题 `16`：招聘计划 `recruitPlan`
   - 主题 `17`：职位标准 `jobStandard`
   - 主题 `18`：录用管理 `hiring`
2. 现状优点是每个主题边界清晰、权限和状态机独立。
3. 现状缺口是：
   - 缺统一入口
   - 缺清晰的跨模块引用规则
   - 缺“来源跳转 / 来源摘要 / 来源快照”的统一口径
4. 若不先重开阶段0冻结，直接做跨模块引用，容易出现：
   - 状态机互相写穿
   - 一个主题变成招聘全流程超级主表
   - 前后端契约和权限范围失控

## 非目标

1. 不在本卡直接修改任何前后端代码。
2. 不在本卡直接改写主题 `8 / 12 / 15 / 16 / 17 / 18` 既有阶段0主卡或最终验收卡。
3. 不把招聘中心改成单表或单资源超级域。
4. 不引入审批流、编排引擎或跨主题自动批处理。

   - `recruitPlan` 改状态不自动改 `resumePool / interview / hiring`
   - `hiring` 改状态不自动改 `resumePool / talentAsset / interview`
6. 不扩到员工入职、合同、薪资、人事档案自动创建。

## 统一入口冻结

1. 一级业务域名称继续固定为：`招聘中心`
2. 统一入口建议固定为：
   - 一级入口语义：`recruitment-center`
   - 建议聚合路由：`/performance/recruitment-center`
3. 上述统一入口仅承担：
   - 领域导航
   - 聚合视图
   - 跨模块跳转
   - 来源引用查看
4. 上述统一入口不承担：
   - 替代各子模块原始资源接口
   - 合并各子模块状态机
   - 变成新的写入超级 API
5. 各子模块原始资源名继续保持独立，不变更为中心统一资源：
   - `talentAsset`
   - `recruitPlan`
   - `jobStandard`
   - `resumePool`
   - `interview`
   - `hiring`

## 子模块边界冻结

1. `recruitPlan`
   - 负责招聘计划备案主链
   - 不负责承接简历全文、面试评语全文或录用决策全文
2. `jobStandard`
   - 负责岗位标准与画像摘要
   - 不负责承接招聘计划状态或录用状态
3. `resumePool`
   - 负责简历全文、联系方式、附件摘要和来源投递材料
   - 不负责承接面试评语全文或录用决策全文
4. `interview`
   - 负责面试安排、面试评语全文和结果摘要
   - 不负责承接简历全文或录用决策全文
5. `hiring`
   - 负责录用决策主链和来源快照
   - 不负责承接简历全文、附件全文或面试评语全文
6. `talentAsset`
   - 负责招聘侧优秀人才摘要池
   - 首批仍不升级为招聘中心全文主数据拥有者

## 跨模块引用冻结

### 首批允许的引用关系

1. `recruitPlan -> jobStandard`
   - 字段语义：`jobStandardId`
   - 用途：招聘计划引用目标岗位标准
2. `resumePool -> recruitPlan`
   - 字段语义：`recruitPlanId`
   - 用途：简历归属到某个招聘计划
3. `resumePool -> jobStandard`
   - 字段语义：`jobStandardId`
   - 用途：简历与岗位标准建立来源对应
4. `interview -> resumePool`
   - 字段语义：`resumePoolId`
   - 用途：面试引用候选人简历主记录
5. `interview -> recruitPlan`
   - 字段语义：`recruitPlanId`
   - 用途：面试能够回溯招聘计划来源
6. `hiring -> interview`
   - 字段语义：`interviewId`
   - 用途：录用单引用来源面试
7. `hiring -> resumePool`
   - 字段语义：`resumePoolId`
   - 用途：录用单引用来源简历
8. `hiring -> recruitPlan`
   - 字段语义：`recruitPlanId`
   - 用途：录用单能够回溯招聘计划来源

### 首批不强制进入主链的模块

1. `talentAsset`
   - 当前保留在招聘中心统一入口中展示和跳转
   - 但首批不强制冻结为招聘中心主链的必经强引用节点
2. 如后续要增加：
   - `talentAsset -> resumePool`
   - `resumePool -> talentAsset`
   - `hiring -> talentAsset`
   必须单独评审并重开阶段0对应边界卡。

## 引用方式冻结

1. 首批统一采用：`引用 ID + 来源摘要 + 来源快照`
2. 引用 ID 负责：
   - 跳转
   - 定位来源对象
3. 来源摘要负责：
   - 列表展示
   - 搜索过滤
   - 不跨模块查全文时的轻量展示
4. 来源快照负责：
   - 锁定创建当时的业务上下文
   - 防止上游名称变化导致下游历史记录失真
5. 上游对象发生修改时：
   - 下游保留既有快照
   - 不要求反向批量刷新历史记录

## 全文 / 摘要 / 快照职责冻结

1. `resumePool`
   - 全文主数据拥有者：简历全文、联系方式、附件摘要
2. `interview`
   - 全文主数据拥有者：面试评语全文、面试安排记录
3. `hiring`
   - 全文主数据拥有者：录用决策全文、录用结果时间线
4. `recruitPlan`
   - 摘要主数据拥有者：招聘计划主链与计划摘要
5. `jobStandard`
   - 摘要主数据拥有者：岗位标准摘要与模板信息
6. `talentAsset`
   - 摘要主数据拥有者：优秀人才池跟进摘要
7. 招聘中心统一入口
   - 只负责聚合和跳转
   - 不接管任何全文字段主所有权

## 自动联动禁区冻结

1. 禁止 `recruitPlan submit / close / void / reopen` 自动改写：
   - `resumePool.status`
   - `interview.status`
   - `hiring.status`
2. 禁止 `resumePool` 自动创建或自动关闭 `hiring`
3. 禁止 `interview completed` 自动创建 `hiring`
4. 禁止 `hiring accepted / rejected / closed` 自动改写：
   - `resumePool`
   - `talentAsset`
   - `interview`
5. 禁止招聘中心统一入口直接调用多个子模块写接口做事务编排。

## 推荐推进顺序冻结

1. 先做统一入口和来源跳转。
2. 再做首批强引用字段与只读来源摘要。
3. 再做创建页“带出上游摘要字段”的轻量联动。
4. 最后才评审是否需要更强的状态约束。

## 验收口径

1. 招聘中心整合后，仍能明确回答每个模块“谁拥有全文、谁拥有摘要、谁只保留快照”。
2. 任一子模块都不能因为整合而失去独立资源名和独立状态机。
3. 首批跨模块能力只允许：
   - 跳转
   - 引用
   - 摘要展示
   - 快照留存
4. 首批不允许：
   - 自动状态级联
   - 跨主题批量写
   - 隐式反写
5. 后续若进入实现，必须先把共享 docs 与对应主题卡同步到一致事实源。
