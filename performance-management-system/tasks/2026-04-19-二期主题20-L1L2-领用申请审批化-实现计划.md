## 分阶段计划

### 阶段 1：后端模型与审批对象接入

目标：

1. 新增 `assetAssignmentRequest` 实体、控制器和 `asset-domain` 主链方法。
2. 把 `assetAssignmentRequest` 纳入 `approval-flow` 的对象类型、配置、实例和回写链路。
3. 冻结 `submit -> inApproval`、`approve -> approvedPendingAssignment`、`assign -> issued` 的服务端真实语义。

交付物：

1. 新实体与 `src/entities.ts` 注册。
2. 新控制器与 `asset-domain` 方法。
3. `approval-flow` 对象类型扩展与回写适配。

门禁：

1. 请求单不污染现有 `assetAssignment` 状态语义。
2. `submit` 失败不留下半提交状态。
3. `approval-flow` 白名单、配置查询和实例详情能识别新对象。

### 阶段 2：权限、菜单、seed 与后端验证

目标：

1. 扩展 `menu.json`、角色授权、seed 和对象级数据范围控制。
2. 补齐后端定向测试、smoke 脚本所需的账号/权限/审批配置准备。

交付物：

1. 新权限键和菜单节点。
2. `seed-stage2-performance.mjs` 中的权限与配置准备。
3. 后端测试与 smoke 前置脚本更新。

门禁：

1. 部门负责人、资产管理员、管理层、员工、HR 的动作边界都有自动化证据。
2. `L0` 现有授权不被意外回收。

### 阶段 3：前端请求页与待配发页

目标：

1. 新增员工“我的领用申请”和资产管理员“待配发申请”页面或最小视图承载。
2. 接入新 service、类型、动作显隐和审批流入口。
3. 保持现有 `asset/assignment.vue` 继续承载执行结果，不混入申请草稿。

交付物：

1. 新视图与 service。
2. 类型与菜单分组更新。
3. 前端定向测试与基础构建验证。

门禁：

1. 页面能区分“申请过程”和“执行结果”。
2. 员工不可选具体资产编号。
3. 审批态和待配发态展示与权限一致。

### 阶段 4：联调、smoke、GUI 与文档回写

目标：

1. 打通员工申请 -> 审批 -> 配发 -> 正式领用记录闭环。
2. 用真实运行结果回写共享事实源、Theme20 重开卡和最终验收卡。
3. 在无未关闭 `P0 / P1` 的前提下收口。

交付物：

1. fresh runtime smoke 结果。
2. GUI 点测留痕。
3. 文档/任务卡回写。

门禁：

1. 无未关闭 `P0 / P1`。
2. 只按真实执行结果写回，不虚写“通过”。

## 最小测试矩阵

- Normal path:
  - 员工新建 `L1` 请求单 -> 提交 -> 部门负责人审批通过 -> 资产管理员配发 -> 生成 `assetAssignment`。
  - 员工新建 `L2` 敏感资产请求单 -> 部门负责人审批 -> 资产管理员确认 -> 管理层确认 -> 配发成功。
- Error path:
  - 员工提交时写入 `assignedAssetId` 被拒绝。
  - 审批未通过时执行 `assign` 被拒绝。
  - 非当前审批人调用 `approval-flow/approve` 被拒绝。
  - 审批实例创建失败时，`submit` 回滚到 `draft`。
- Boundary case:
  - 命中多规则时按最高层级。
  - 敏感资产即使 `< 1000` 仍走 `L1/L2`。
  - 已 `issued` 请求单不可撤回。
  - `manualPending` 下不可再次 `submit`。
- Permission checks:
  - 员工仅本人请求单。
  - 部门负责人仅本人当前节点。
  - 资产管理员仅可审批命中节点、仅可配发 `approvedPendingAssignment`。
  - `HR` 才能 `fallback / terminate / cancel / resolve`。
- Contract checks:
  - 新资源请求/响应字段。
  - `approval-flow` 新对象类型兼容。
  - `assetAssignment` 创建后回写字段和状态。
