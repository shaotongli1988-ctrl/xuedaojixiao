## 当前已能确认的边界

### 窗口 A 已冻结范围

1. 自动审批流属于二期候选能力，不属于首期回补。
2. 首批覆盖对象只包含：
   - `assessment`
   - `promotion`
3. 首批不做全局统一审批流引擎，仍依附现有 `performance` 模块，在对象各自的状态机与权限链路内分别配置、分别接入。
4. 首批窗口 A 只冻结“按规则推进审批节点”的业务范围，不把通知、催办、超时处理扩写成独立通知平台或独立主题。
5. 当前仓库里，自动审批流已完成“开发前冻结 / 需求澄清”阶段；后续进入设计或开发时必须严格复用当前冻结包，不得跳过事实源直接扩 scope。

### 首批绝不进入自动审批流的对象

1. `PIP`
   - 当前状态机只有 `draft / active / completed / closed`，没有审批节点
2. `goal`
   - 当前只有目标创建、编辑、进度推进，不存在审批主链
3. `feedback`
   - 当前是环评任务与反馈提交，不是审批对象
4. `salary`
   - 当前是 `HR` 维护与确认 / 归档链路，不纳入共享审批流首批范围
5. `suggestion`
   - 自动建议引擎只负责建议，不负责审批流
6. 驾驶舱及任何跨模块对象
   - 当前仓库没有统一审批主链事实源，不允许借题扩 scope

### 明确非目标

1. 不开发前端、后端、数据库、脚本或共享基础层代码。
2. 不把自动审批流和自动建议引擎混写成同一主题。
3. 不把自动审批流扩展成跨模块驾驶舱、通知平台、催办平台、超时任务平台或新技术栈方案评审。
4. 不回改首期已关闭的手工审批结论。

## 当前已冻结的核心规则

1. 审批节点模型已固定为串行单链路、单当前审批人，不支持并行会签、条件分支、加签、减签。
2. 审批人解析规则已固定为：
   - `specified_user`
   - `applicant_direct_manager`
   - `employee_department_manager`
   - `department_tree_role`
   - `hr_manual_assign`
3. 与现有手工审批主链的兼容方案已冻结：
   - 发起语义复用 `assessment/submit` 与 `promotion/submit`
   - 自动审批动作统一走 `approval-flow` 资源
   - 回退到手工审批主链只允许 `HR` 执行，且实例记为 `terminated`
4. 存储、接口、权限、字段粒度和回退口径已分别在 `03 / 04 / 06 / 12 / 15` 落版。
