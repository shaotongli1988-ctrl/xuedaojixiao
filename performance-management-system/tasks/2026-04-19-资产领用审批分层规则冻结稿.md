## 当前代码与冻结现状

1. 主题20现有冻结口径中，领用是直接业务动作：
   - `available -> assigned`
2. 当前 `assetAssignment/add` 直接创建领用记录并把资产主状态改成 `assigned`。
3. 当前主题20没有领用审批实例层，也没有 `submitted / approved / rejected` 一类领用审批状态。
4. 当前有权发起领用的角色已经不是普通员工，而是：
   - 管理层
   - 超级管理员
   - 资产管理员
   - 部门负责人

这意味着：

1. 现状并不是“员工先申请，再审批”。
2. 现状更接近“有管理权限的人直接办理领用”。
3. 因此若未来要做领用审批，必须先区分哪些场景值得增加审批成本。

## 本稿适用目标

1. 让低风险领用继续高效处理。
2. 让高价值、敏感、跨部门和异常场景具备正式审批留痕。
3. 控制主题20范围，不把所有资产动作都升级成审批中心。

## 非目标

1. 不把所有领用统一改造成审批流。
2. 不在本稿中直接改写主题20既有接口和状态集合。
3. 不直接引入复杂并行会签、条件分支或外部 OA 联动。
4. 不扩到耗材仓储中心、采购计划、预算控制或财务凭证链。

## 分层方案

### `L0`：免审批直领

#### 适用场景

1. 低值、标准化、低风险物品。
2. 同部门内部的常规领用。
3. 由资产管理员或部门负责人直接发放即可闭环的场景。

#### 建议适用品类

1. 键盘
2. 鼠标
3. 普通拓展坞
4. 常规外设
5. 单价较低的标准办公配件

#### 建议口径

1. 仍沿用当前主题20既有模式：
   - 有权限角色直接新增领用记录
   - 资产直接进入 `assigned`
2. 不进入统一审批流。
3. 重点保证：
   - 领用人明确
   - 领用部门明确
   - 用途留痕
   - 归还 / 丢失闭环完整

### `L1`：单级审批

#### 适用场景

1. 中高价值个人设备。
2. 需要主管确认合理性的资产。
3. 同部门内部领用，但不适合完全直领的设备。

#### 建议适用品类

1. 笔记本电脑
2. 手机
3. 平板
4. 显示器
5. 打印设备
6. 单价较高但不属于敏感类的固定资产

#### 建议审批链

1. 申请人：
   - 员工本人或代办人
2. 审批人：
   - 部门负责人
3. 执行人：
   - 资产管理员

#### 建议结果

1. 审批通过后，由资产管理员执行领用发放。
2. 审批驳回后，申请回到可修改或取消状态。

### `L2`：加强审批

#### 适用场景

1. 高价值资产。
2. 敏感资产。
3. 跨部门借用。
4. 遗失后补领。
5. 异常补发。
6. 报废替换发放。

#### 建议适用品类

1. 高配笔记本
2. 高价值移动终端
3. 服务器类终端设备
4. 涉密、受控、专岗专配资产
5. 涉及跨部门资产所有权或责任交接的借用场景

#### 建议审批链

1. 申请人：
   - 员工本人或代办人
2. 第一审批人：
   - 部门负责人
3. 第二确认人：
   - 资产管理员
4. 如涉及敏感资产或异常补领：
   - 可追加管理层确认

#### 建议结果

1. 只有全部审批通过后，才允许发放。
2. 必须保留异常原因、审批意见和执行留痕。

## 建议分层规则表

| 层级 | 是否审批 | 典型场景 | 建议角色链路 | 是否建议接统一 approval-flow |
| --- | --- | --- | --- | --- |
| `L0` | 否 | 低值常规物品、同部门标准发放 | 部门负责人 / 资产管理员直接办理 | 否 |
| `L1` | 是 | 中高价值个人设备 | 员工申请 -> 部门负责人审批 -> 资产管理员发放 | 是 |
| `L2` | 是 | 高价值 / 敏感 / 跨部门 / 异常补领 | 员工申请 -> 部门负责人审批 -> 资产管理员确认 / 管理层补充确认 -> 发放 | 是 |

## 金额与类别建议阈值

### 说明

1. 当前仓库没有现成“领用审批金额阈值”事实源。
2. 以下阈值仅作为初版建议值，后续必须由业务 / 财务 / 资产 owner 明确认领后，才能写入正式冻结正文。

### 初版建议值

1. `L0`
   - 建议：单价 `< 1000`
   - 且不属于敏感资产
2. `L1`
   - 建议：单价 `1000 - 5000`
   - 或属于笔记本 / 手机 / 平板 / 显示器等中高价值个人设备
3. `L2`
   - 建议：单价 `>= 5000`
   - 或命中任一加强条件：
     - 敏感资产
     - 跨部门借用
     - 遗失后补领
     - 异常补发
     - 报废替换发放

### 分类优先级规则

即使金额未达阈值，只要命中以下任一条件，也应直接升级审批层级：

1. 敏感资产
2. 跨部门借用
3. 遗失后补领
4. 异常补发
5. 特殊岗位专用设备

## 场景建议

### 不审批场景

1. 资产管理员给本部门标准办公位补发普通鼠标、键盘。
2. 部门负责人给本部门新增成员发放常规低值外设。
3. 同类低值资产的正常替换。

### 单级审批场景

1. 员工首次申请笔记本电脑。
2. 员工申请办公手机。
3. 员工申请单独显示器、平板等中高价值个人设备。

### 加强审批场景

1. 销售部门向技术部门借用高价值设备。
2. 员工遗失设备后申请补领。
3. 涉密岗位申请敏感终端。
4. 已报废设备的替换发放。

## 对主题20现状的兼容建议

### 当前不重开审批边界时

1. 继续沿用既有 `assetAssignment/add` 直领模式。
2. 只在制度和线下流程上区分哪些资产必须先经过审批确认。
3. 系统层先不扩状态机。

### 若未来重开审批边界

建议只为 `L1 / L2` 场景扩审批，不改 `L0`。

#### 推荐对象策略

1. `L0`
   - 继续保留在 `assetAssignment`
   - 不进入统一 `approval-flow`
2. `L1 / L2`
   - 作为新的“领用申请”对象进入审批实例层
   - 审批通过后再落正式 `assetAssignment`

## 若进入系统审批化，建议最小模型

### 推荐方案

不要直接把现有 `assetAssignment` 强改成审批对象，建议新增“领用申请层”。

#### 原因

1. 现有 `assetAssignment` 已承载“领用中 / 已归还 / 丢失”事实。
2. 它更像执行结果单，而不是审批申请单。
3. 若硬在现有对象上加 `submitted / approved / rejected`，会把“申请态”和“执行态”混在一起。

### 推荐资源拆分

1. `assetAssignmentRequest`
   - 负责申请、审批、驳回、撤回
2. `assetAssignment`
   - 负责审批通过后的真实领用结果记录

### 推荐申请状态

1. `draft`
2. `submitted`
3. `approved`
4. `rejected`
5. `cancelled`

### 推荐结果状态

1. `assigned`
2. `returned`
3. `lost`

## 与统一审批流的关系建议

### 不建议接入

1. `L0` 直领

### 建议接入

1. `L1` 单级审批领用申请
2. `L2` 加强审批领用申请

### 接入理由

1. 这些场景真正需要：
   - 当前审批人
   - 驳回意见
   - 待办
   - 转办
   - 人工恢复
   - 审计留痕
2. 统一审批流更适合“申请 -> 审批”层，而不是“发放 -> 归还 -> 丢失”执行层。

## 角色建议

### 申请角色

1. 员工本人
2. 代办人
3. 部门负责人

### 审批角色

1. 部门负责人
2. 资产管理员
3. 管理层（仅加强审批场景）

### 执行角色

1. 资产管理员

## 风险与注意事项

1. 如果不区分申请层和执行层，后续状态机会非常混乱。
2. 如果所有领用都强制审批，会明显增加低值物品流转成本。
3. 如果没有金额和类别双维度规则，后续容易出现：
   - 同价不同风险资产被错误归类
   - 敏感资产因为金额低而被绕过审批
4. 若后续真进入实现，必须同步重开：
   - 主题20接口 / 权限 / 状态冻结
   - 统一审批流对象白名单边界
