## 最小测试矩阵

### 正常路径

1. 当来源业务对象命中已冻结审批流范围、节点模型和审批人解析规则时，可生成审批实例并进入首个合法审批节点。
2. 合法审批动作完成后，审批实例状态、当前节点、下一节点和审计留痕必须一致。
3. 命中“回退到现有手工审批主链”的场景时，系统必须按已冻结兼容口径回退，不得同时保留两套生效主链。

### 异常路径

1. 节点模型不存在、审批人解析失败、或来源状态不允许进入审批流时，必须拒绝启动审批流。
2. 非当前节点审批人、超范围审批人或无权限角色执行审批动作时必须被拒绝。
3. 已结束审批实例不得重复推进、重复驳回或重复回退。
4. 缺少驳回原因、转办原因、回退原因或审计必填字段时必须拒绝写入。

### 边界路径

1. 单节点、双节点和多节点审批链都必须覆盖。
2. 审批人解析命中组织树边界、空节点、重复节点或多人候选时，结果必须符合已冻结口径。
3. 审批流运行中若来源业务对象状态发生变化，必须覆盖禁止继续推进、重新计算或回退到手工审批的既定结果。
4. 超时、转办、驳回、撤回等异常动作在边界条件下必须覆盖幂等与审计一致性。

### 权限路径

1. 只有已冻结审批节点上的合法审批人可执行当前节点动作。
2. `HR`、经理、员工以及后续若存在的其他角色都必须按已冻结口径验证可见范围和可执行动作。
3. 任何角色都不得因前端隐藏失效而绕过后端审批权限。
4. 审批记录、日志和审计结果不得暴露冻结范围外的敏感字段或跨模块数据。

### 状态路径

1. 审批实例启动后必须进入唯一合法初始态。
2. 每个节点动作完成后必须进入唯一下一态或终态。
3. 已结束状态不得继续推进；非法状态流转必须明确拒绝。
4. 在当前已冻结事实源下，回退到现有手工审批主链只能从被允许的节点 / 状态进入，且必须补齐审计留痕。

## 后续开发拆分顺序

1. 串行包 1：冻结首批适用业务对象、非目标和与现有手工审批主链的兼容边界。
2. 串行包 2：冻结审批节点模型、状态机、审批人解析、驳回 / 转办 / 超时 / 回退语义。
3. 串行包 3：冻结接口、字段、存储、审计、回滚和兼容迁移口径，并同步回写 `03 / 04 / 05 / 06 / 09 / 15`。
4. 串行包 4：实现审批配置模型、审批实例模型和基础校验骨架。
5. 并行包 5：
   - 后端只读链路：审批实例查询、节点详情、审计详情
   - 前端只读链路：审批流查看页、空态、无权限态、错误态
   - 自动化脚本草拟：`jest`、smoke、权限回归、兼容回退回归
6. 串行包 6：实现审批动作链路、兼容回退、转办 / 驳回 / 超时处理和审计写回。
7. 串行包 7：跑通权限回归、状态回归、兼容回退回归，并完成文档与交付留痕。

1. 主链兼容边界未冻结前，不允许并行做状态机、接口或实现拆分。
2. 节点模型 / 审批人解析未冻结前，不允许并行做动作接口或动作 UI。

4. 审批实例模型与兼容回退链路必须先稳定，之后才能实现动作链路，不能并行推进。

### 可并行

1. 在接口 / 字段 / 审计 / 回滚口径冻结完成后，只读后端、只读前端和自动化脚本草拟可以并行。
2. 在动作契约冻结完成后，动作前端交互和后端动作实现可以并行，但必须共用同一状态机、同一审批人解析口径和同一审计模型。
