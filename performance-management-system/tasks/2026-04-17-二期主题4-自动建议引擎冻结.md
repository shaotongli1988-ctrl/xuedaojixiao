## 当前已冻结边界

### 已能确认的范围

1. 自动建议引擎仍属于二期增强，不属于首期回补。
2. 首批建议对象只覆盖 `PIP` 与 `晋升` 两类现有业务对象，不覆盖薪资、环评、目标、驾驶舱或外部系统对象。
3. 自动建议当前只允许作为“建议能力”，不允许扩展为自动审批流、自动决策或自动创建正式业务单。
4. 当前建议能力只允许围绕现有评估结果域讨论，不能反向改写首期已关闭的 `PIP / 晋升` 手工创建主链。
5. 自动建议引擎的输出是“建议记录”而不是“正式单据”；是否采用仍由人工决定。

### 明确非目标

1. 不开发任何后端规则引擎、前端页面、定时任务或消息通知。
2. 不把“建议”写成“自动创建 PIP / 晋升单”。
3. 不把“建议”写成“自动提交、自动审批、自动流转状态”。
4. 不扩展到薪资建议、自动审批流、跨模块驾驶舱建议或任意 AI / 新技术栈。
5. 不修改首期已关闭的 `PIP`、晋升、评估主链、状态机、权限和接口语义。

### 建议对象与触发来源

1. 建议对象固定为两类：
   - `PIP 建议`
   - `晋升建议`
2. 首批触发来源只允许来自现有评估结果域的稳定结果字段：
   - `assessment.status`
   - `assessment.grade`
   - `assessment.totalScore`
3. 首批不直接使用以下来源触发建议：
   - 目标地图状态或进度
   - `360 环评` 汇总结果
   - 薪资结果
   - 驾驶舱聚合指标
   - `PIP` 跟进记录
   - 晋升评审记录

### 首批触发规则

1. 只允许在评估单已经进入稳定结果状态后触发建议；首批固定为 `assessment.status = approved`。
2. `PIP 建议` 触发规则固定为：
   - `assessment.grade = C`
   - 等价分数口径为 `assessment.totalScore < 70`
3. `晋升建议` 触发规则固定为：
   - `assessment.grade = S` 或 `assessment.grade = A`
   - 等价分数口径为 `assessment.totalScore >= 80`
4. `assessment.grade = B` 默认不触发任何建议。
5. 同一条评估结果在同一时点最多产出一条同类建议，不允许重复生成同类建议噪音。

### 建议语义与人工处理

1. 自动建议只产生“推荐”，不自动落正式 `PIP` 或 `晋升` 单据。
2. 为了后续人工处理留痕，允许系统自动落“建议记录”，但该记录不等于正式业务单。
3. 建议记录允许被人工：
   - 采用
   - 忽略
   - 驳回
   - 以人工新建正式单据方式覆盖
4. 手工创建 `PIP / 晋升` 的既有主链继续保留，不以存在建议为前置条件。

### 绝不触发建议的场景

1. 评估单不是 `approved`，绝不触发。
2. 评估结果缺少 `grade` 或 `totalScore` 任一稳定字段，绝不触发。
3. 同一员工、同一评估单下已存在同类未结束人工处理结果时，绝不重复触发同类建议。
4. 脱离评估单独立创建的 `PIP / 晋升` 历史记录，不反向生成建议。
5. 员工本人侧不触发、也不因为移动端或前端页面访问而触发建议。
6. 任何薪资、环评、目标、驾驶舱或外部系统数据变化，首批都绝不直接触发建议。

## 当前不在本窗口冻结的规则

1. 建议记录自身的完整状态机、合法流转和非法流转处理。
2. “采用 / 忽略 / 驳回 / 覆盖”四类动作的精确权限分配。
3. 误报、撤销、改判后的审计字段、责任归属和回归门槛。
4. 是否需要单独页面、消息通知、批量处理和报表统计。

## 待确认问题

1. 建议记录是否需要独立页面还是只嵌入 `PIP / 晋升` 现有页面。
2. 哪些角色可查看、采用、忽略、驳回、覆盖建议，员工本人是否可见。
3. 覆盖同类建议时，是否允许一键带入正式单据草稿，还是只保留手工创建入口。
4. 误报、漏报、撤销、改判的最小审计与测试矩阵如何定义。
