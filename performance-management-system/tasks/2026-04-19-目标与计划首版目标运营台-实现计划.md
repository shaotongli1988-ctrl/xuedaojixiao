# 2026-04-19 目标与计划首版目标运营台实现计划

## Before Starting
- Scope:
  - 在当前 `目标&计划 -> 目标&计划总览` 承载关系下，把现有 `/performance/goals` 从“目标地图列表页”演进为“目标运营台”首版。
  - 首版覆盖 `员工 + 主管 + HR/管理员` 三层角色的日闭环：规则配置、周/月目标、日目标下发、员工回填、自动统计、自动日报。
  - 实现必须满足 spec 中已经冻结的公平性约束：横向展示和日报必须能区分 `公共目标贡献` 与 `个人补充目标贡献`。
- Non-goals:
  - 不改一级导航 IA，不新增一级菜单。
  - 不合并 `班主任化` 页面，不改其二级目录。
  - 不做复杂审批、补填、申诉、人工修正流。
  - 不做跨部门对比、匿名榜单、岗位分组榜。
  - 不承诺本阶段接入完整外部群推送基础设施以外的增强能力。
- Affected modules:
  - `cool-admin-vue/src/modules/performance/views/goals/`
  - `cool-admin-vue/src/modules/performance/components/` 下与 goals 相关的新旧组件
  - `cool-admin-vue/src/modules/performance/service/goal*` 或相关请求封装
  - `cool-admin-midway/src/modules/performance/controller/admin/goal*`
  - `cool-admin-midway/src/modules/performance/service/goal*`
  - `cool-admin-midway/src/modules/performance/entity/goal*` 及可能新增的运营台相关实体
  - `cool-admin-midway/scripts/seed-stage2-performance.mjs`
  - `cool-admin-midway/scripts/smoke-stage2-performance*.mjs`
  - `performance-management-system/docs/`
  - `performance-management-system/tasks/`
- Acceptance criteria:
  - 主管可配置本部门时间节点并下发周/月/日目标。
  - 员工可查看目标、录入实际完成值、维护个人补充目标。
  - 截止未填按 `0` 入账。
  - 页面可展示完成率榜和绝对产出榜。
  - 页面和日报均可区分公共目标贡献与个人补充目标贡献。
  - 系统可自动生成部门日报并支持系统内留痕。
  - 存在自动发送与主管拦截/延期能力，至少形成可验证闭环。
- Impact surface:
  - 前端页面结构、角色显隐、状态文案、交互路径。
  - 后端目标资源、统计聚合、日报记录、时间节点配置。
  - 种子数据、smoke 脚本、GUI 场景。
  - 路由与菜单相关事实源、测试文档、版本记录。
- Source of truth:
  - 设计 spec：[2026-04-19-goals-plan-operations-design.md](/Users/shaotongli/Documents/xuedao/docs/superpowers/specs/2026-04-19-goals-plan-operations-design.md)
  - 导航 IA：[2026-04-19-navigation-ia-consolidation-design.md](/Users/shaotongli/Documents/xuedao/docs/superpowers/specs/2026-04-19-navigation-ia-consolidation-design.md)
  - 菜单/路由事实源：[10-路由与菜单映射.md](/Users/shaotongli/Documents/xuedao/performance-management-system/docs/10-路由与菜单映射.md)
- Requirement conflicts or unclear points that must be resolved before coding:
  - 是否完全复用现有 `goal` 主资源，还是拆分 `goalPlan / goalResult / goalSnapshot / goalDailyReport` 多资源。
  - 自动群推送的现有基础设施和失败重试策略。
  - 时间节点是否支持“跨日发送”或仅限同日时刻。
- High-risk surfaces:
  - 共享 `goal` 资源的契约扩展。
  - 角色与权限显隐。
  - 默认值与自动补零逻辑。
  - 榜单和日报的公平性解释。
  - 外部推送与失败回退。
- Required delivery evidence:
  - 后端目标运营台核心逻辑测试。
  - 前端 `type-check` / `build`。
  - 至少一条 fresh runtime smoke 覆盖主管下发、员工录入、统计和日报链路。
  - 至少一条 GUI 场景覆盖主管主入口和员工录入主入口。
  - 文档回写：版本记录、路由/权限变更、测试清单、任务卡。
- Required test matrix:
  - Normal path：主管下发 -> 员工填报 -> 自动统计 -> 自动日报。
  - Error path：截止未填 -> 自动记 0；无权限用户访问受限能力；推送失败回退到系统内留痕。
  - Boundary case：仅公共目标、仅个人目标、公共+个人混合、目标值为 0 或空、接近截止时间录入。
  - Permission checks：员工/主管/HR 管理员三层差异。
  - Contract checks：页面请求、返回结构、统计字段、日报字段。
- Key assumptions and constraints:
  - 当前仍以 `/performance/goals` 作为目标运营台承载路由。
  - 首版优先保证闭环，不追求一次性补齐所有运营细节。
  - 未关闭 `P1` 公平性风险前不得进入 handoff/release 结论。

## 分阶段计划

### 阶段 1：技术方案收口

目标：

1. 冻结后端资源模型和前端页面壳拆分方式。
2. 冻结权限键扩展方案和自动补零/统计快照的状态口径。
3. 产出实现写集，避免前后端边界漂移。

交付物：

1. 技术方案补充记录。
2. 明确的 API / 状态 / 权限扩展清单。
3. 确认是否需要新增实体或仅扩展现有 `goal` 资源。

门禁：

1. 源事实源与实现方案不冲突。
2. 能说清楚如何关闭 `P1` 公平性风险。

### 阶段 2：后端主链

目标：

1. 完成周/月/日目标承载、员工结果填报、统计聚合、日报记录。
2. 完成自动补零、榜单拆维度、日报生成主链。
3. 完成对应权限控制和最小种子数据。

交付物：

1. 后端实体、service、controller。
2. 后端单测 / 集成级验证。
3. fresh runtime smoke 脚本。

门禁：

1. 正常链路、补零异常链路和权限拒绝链路都有自动化证据。
2. 契约字段在调用方可消费。

### 阶段 3：前端主链

目标：

1. 把 `/performance/goals` 改造成目标运营台总览。
2. 完成主管总览、员工填报、目标对比、目标配置、日报汇报四个工作区。
3. 完成按角色裁剪展示和关键交互。

交付物：

1. 页面壳层和组件拆分。
2. 请求适配和状态处理。
3. type-check / build 通过。

门禁：

1. 页面能直观区分公共目标与个人补充目标贡献。
2. 不破坏现有导航 IA 和 `班主任化` 目录。

### 阶段 4：联调与验收

目标：

1. 打通主管与员工双端闭环。
2. 跑通 fresh runtime smoke 和 GUI。
3. 关闭 `P0 / P1` 阻塞后再允许 handoff。

交付物：

1. smoke 报告。
2. GUI 报告。
3. 任务卡与文档回写。

门禁：

1. 无未关闭 `P0 / P1`。
2. 公平性约束已在页面与日报层被真实验证。
