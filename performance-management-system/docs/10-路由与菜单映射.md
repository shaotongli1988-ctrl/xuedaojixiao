# 路由与菜单映射

## 文档职责

这份文档是“菜单、路由和页面目录映射”的唯一事实源。

它只负责定义：

1. 菜单分组
2. 页面路由
3. 页面名称
4. 对应的前端模块目录
5. 对应的权限键

## 不负责内容

这份文档不负责定义：

- 接口返回字段
- 图表统计口径
- 数据范围规则

## 基线约束

### 页面结构参考基线

首期页面结构统一参考 [18-页面结构与原型映射.md](/Users/shaotongli/Documents/xuedao/performance-management-system/docs/18-页面结构与原型映射.md)。

这里固定两条规则：

1. 菜单、路由和 `viewPath` 以本文件为准。
2. 页面内部结构、详情承载和组件拆分以 `18` 文档为准。

### 当前菜单实现基线

`xuedao` 当前后台菜单以 [menu.json](/Users/shaotongli/Documents/xuedao/cool-admin-midway/src/modules/base/menu.json) 的结构为基线，核心字段包括：

- `name`
- `router`
- `type`
- `icon`
- `viewPath`
- `keepAlive`
- `isShow`
- `childMenus`
- `perms`

类型约定：

- `type = 0`：菜单分组
- `type = 1`：页面菜单
- `type = 2`：按钮权限

### 当前前端路由装配基线

前端菜单和视图路由由 [menu.ts](/Users/shaotongli/Documents/xuedao/cool-admin-vue/src/modules/base/store/menu.ts) 动态装配，因此绩效模块需要满足下面几条：

1. `router` 必须是最终页面访问路径。
2. `viewPath` 必须能映射到 `cool-admin-vue/src/` 下的真实文件。
3. 同一页面菜单下的按钮权限通过 `type = 2` 子节点维护。
4. 页面组件内部仍通过 `v-permission` 和 `service.xxx.xxx.permission.*` 做显隐。

### 初始化方式基线

首期绩效模块菜单和权限初始化固定按下面方式执行：

1. 沿用现有 Cool 基座菜单导入链路。
2. 绩效模块菜单数据使用与 `base/menu.json` 相同的结构组织。
3. 按钮权限继续放在菜单树 `type = 2` 子节点中。
4. 不新增单独的本地硬编码菜单方案。
5. 不新增独立的绩效权限初始化平台或脚本体系。

## 页面与模块目录对应规则

1. `router` 与页面访问 URL 一一对应。
2. `viewPath` 一律指向 `modules/performance/views/...`，不混用 `cool/modules/...` 老路径。
3. 列表页统一落在各页面目录下的 `index.vue`。
4. 详情抽屉、表单弹窗、图表卡片等复用组件统一落在 `cool-admin-vue/src/modules/performance/components/`。
5. 评估模块三张列表页必须复用同一套详情和表单组件，不允许各自复制一份。
6. 360 环评只保留一个菜单页，任务列表、填写、汇总通过内部组件承载。
7. `jixiao` 的 `/performance` 导航首页和 `/performance/actions` 行动计划页不直接复制为首期菜单。
8. 当前 `modules/performance/` 只是技术挂载目录，不作为一级业务域命名依据。

## 实施注意事项

1. 菜单名称、权限键、接口资源名必须统一使用同一套英文资源名。
2. 如果后续新增详情独立页，本文件必须先更新，再补菜单和前端文件。
3. 菜单是否显示由 `isShow` 控制，权限是否可执行由 `perms` 控制，不能混用。
4. 首期菜单和权限数据通过现有菜单导入方式进入系统，不额外发明第二套初始化链路。
5. 后续若启用顶栏分组或“导航栏 + 侧栏”联动，一级导航分组必须以本文件定义的业务域归属为准。
