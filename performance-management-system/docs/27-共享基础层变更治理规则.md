# 共享基础层变更治理规则

## 文档职责

这份文档是“共享基础层变更如何治理”的唯一事实源。

它只负责定义：

1. 哪些内容属于共享基础层
2. 共享基础层变更的风险等级
3. 共享基础层变更必须补的验证和回写要求

## 不负责内容

这份文档不负责定义：

- 具体业务模块规则
- 单个页面交互细节
- 单个业务实体字段全文

## 共享基础层范围

本项目把以下内容视为共享基础层：

1. 登录链路
2. JWT 配置读取
3. 鉴权中间件
4. `ctx.admin` 运行上下文
5. 角色权限缓存
6. 部门权限缓存
7. 菜单权限获取
8. 动态菜单与权限键装配链路

当前仓库中的典型文件包括但不限于：

- `cool-admin-midway/src/modules/base/service/sys/login.ts`
- `cool-admin-midway/src/modules/base/middleware/authority.ts`
- `cool-admin-midway/src/modules/base/service/sys/perms.ts`
- `cool-admin-midway/src/modules/base/service/sys/menu.ts`
- `cool-admin-midway/src/modules/base/controller/admin/comm.ts`

## 风险等级

共享基础层变更默认视为高风险变更。

原因：

1. 它会同时影响多个业务模块
2. 它会影响登录、鉴权、权限显隐和数据范围
3. 单点修复容易引入系统性回归

因此，任何共享基础层改动都不得按普通业务文件处理。

## 强制验证要求

只要改动共享基础层，至少必须同时具备下面 3 类证据：

1. 单元或局部回归测试
   - 证明改动点本身逻辑正确
2. 受保护接口回归
   - 至少 1 个非 admin 账号访问真实受保护接口
3. 端到端 smoke
   - 至少覆盖真实登录与模块主接口

缺少任何一类，都不能视为完成。

## 最小回归矩阵

共享基础层改动后，最少覆盖：

1. `admin` 或等效高权限账号
2. 非 admin 普通业务账号
3. 菜单权限读取
4. 至少一个读取型业务接口
5. 至少一个带数据范围的业务接口

如果改动涉及 `ctx.admin`、权限缓存、角色解析或部门范围，还必须覆盖：

1. 角色权限正确
2. 数据范围正确
3. 越权不可见正确

## 禁止事项

共享基础层改动时，不允许：

1. 只修登录接口，不验证登录后受保护接口
2. 只补 mock 测试，不做真实接口回归
3. 只让 `admin` 通过，不验证普通账号
4. 只看页面是否能打开，不验证权限和数据范围
5. 不回写任务卡和版本记录就宣称已修复

## 放行条件

共享基础层改动只有同时满足下面条件才允许放行：

1. 最小回归矩阵通过
2. 当前阶段 smoke 通过
3. 任务卡已回写
4. 相关 `P0 / P1` 已关闭或转入明确 owner 的待办
