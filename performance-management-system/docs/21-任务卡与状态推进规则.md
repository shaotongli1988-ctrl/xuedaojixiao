# 任务卡与状态推进规则

## 文档职责

这份文档是“任务卡创建和状态推进”的唯一事实源。

它只负责定义：

1. 哪些任务必须建卡
2. 任务卡最小字段
3. 任务状态集合
4. 状态流转规则
5. 更新频率和回写要求

## 不负责内容

这份文档不负责定义：

- 业务需求本身
- 模块开发顺序
- 问题分级规则

这些内容分别以 `01`、`19`、`23` 文档为准。

## 建卡规则

1. 涉及模块开发的任务默认都要建卡。
2. 跨前后端、跨多个模块或需要用户多轮确认的任务，必须建卡。
3. 只读分析或一次性小修复可不单独建卡，但不能跨出当前会话范围。

## 最小字段

任务卡至少包含：

1. 任务名称
2. 当前阶段
3. 当前状态
4. 负责人
5. 需求摘要
6. 当前结论
7. 已完成内容
8. 待确认问题
9. 下一步动作
10. 受影响文档
11. 受影响代码目录
12. 自动化验证计划与结果
13. 回改范围

## 状态集合

| 状态 | 含义 |
| --- | --- |
| `draft` | 已建卡，尚未进入实质推进 |
| `ready` | 前置条件已满足，可开始执行 |
| `in-progress` | 正在推进 |
| `blocked` | 被外部依赖、确认项或缺陷阻塞 |
| `reviewing` | 当前阶段产物已完成，等待确认或复验 |
| `done` | 任务完成并关闭 |
| `cancelled` | 任务取消，不再继续 |

## 状态流转规则

| 当前状态 | 动作 | 下一状态 |
| --- | --- | --- |
| `draft` | 前置资料齐全 | `ready` |
| `ready` | 开始执行 | `in-progress` |
| `in-progress` | 被阻塞 | `blocked` |
| `blocked` | 阻塞解除 | `in-progress` |
| `in-progress` | 当前阶段提交确认 | `reviewing` |
| `reviewing` | 确认通过 | `done` 或回到 `in-progress` |
| 任意未关闭状态 | 任务取消 | `cancelled` |

补充规则：

1. `reviewing` 被打回后必须回到 `in-progress`，不能直接标 `done`。
2. `blocked` 必须写清楚阻塞原因和外部依赖人。
3. `done` 后如需求变更，开新任务卡，不直接复活旧卡。

## 更新频率

1. 每次阶段切换都必须更新任务卡。
2. 每次用户确认了关键规则，都必须更新任务卡中的“当前结论”和“下一步动作”。
3. 每次影响唯一事实源的文档回写，都必须更新任务卡中的“受影响文档”。
4. 至少每天同步一次状态；如果一天内无进展，也要写明阻塞原因。

## 回写规则

1. 需求规则变化，回写 `01`、`04`、`05`、`06`、`07`、`08`、`10`、`12` 中受影响文档。
2. 进度顺序变化，回写 `19`。
3. 流程门禁变化，回写 `20`。
4. 变更结论变化，回写 `13`。

## 关闭规则

任务卡只有在以下条件全部满足时才能关闭：

1. 对应模块达到最小完成定义。
2. 最小验证已执行。
3. 相关文档已回写。
4. 未关闭的 `P0 / P1` 不存在。
5. 如有遗留 `P2`，已记录 owner 和时点。
6. 涉及行为变化时，已补齐 [24-自动化测试策略与脚本规划.md](/Users/shaotongli/Documents/xuedao/performance-management-system/docs/24-自动化测试策略与脚本规划.md) 要求的自动化验证证据。
