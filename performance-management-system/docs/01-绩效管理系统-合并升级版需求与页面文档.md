# 绩效管理系统合并升级版需求与页面文档

## 1. 文档定位

本文档用于合并下面两份源文档，并输出一份可直接指导后续实现的统一基线文档：

1. `PRC-绩效管理系统需求文档.md`
2. `05-前端页面文档.md`

这份文档的目标不是简单拼接，而是做三件事：

1. 统一业务范围、角色、页面和路由。
2. 把需求文档的业务模块升级成可落地的后台页面结构。
3. 把所有实现约束固定在 `xuedao` 当前技术框架内。

## 2. 合并升级原则

### 2.1 文档优先级

本项目后续开发统一按下面优先级执行：

1. 用户当前确认的范围
2. 本合并升级文档
3. `xuedao` 仓库现有代码结构与技术框架
4. 两份原始文档

### 2.2 差异处理规则

| 冲突项 | 升级规则 |
| --- | --- |
| 模块范围 | 以 PRC 文档为准，纳入完整绩效域 |
| 页面粒度 | 以前端页面文档的列表、详情、创建、编辑粒度为参考 |
| 路由命名 | 对外菜单路由以 PRC 文档为准，内部实现可复用已有 CRUD 页面模式 |
| 技术栈描述 | 以 `xuedao` 实际存在的 `cool-admin-midway + cool-admin-vue` 为准 |
| 首期范围 | 只做 PC 后台和后端，不做 `cool-uni` |

### 2.3 首期实施边界

- 首期纳入完整绩效域：
  - 绩效驾驶舱
  - 我的考核
  - 指标库
  - 已发起考核
  - 待我审批
  - 360 环评
  - 目标地图
  - PIP 追踪
  - 薪资管理
  - 晋升管理
- 首期实现方式：
  - 按模块逐步实现
  - 不要求一次性全部编码完成
- 首期不纳入：
  - `cool-uni`
  - 人才中心、培训中心、合同、采购、会议等非绩效域代码开发

## 3. 项目定位与角色

### 3.1 项目定位

本项目是一个基于 Cool 基座实现的企业绩效管理后台系统，首期目标是在现有 `cool-admin-midway + cool-admin-vue` 基础上完成绩效域核心闭环，后续再扩展员工移动端。

### 3.2 核心角色

| 角色 | 主要职责 |
| --- | --- |
| HR 管理员 | 配置指标、发起考核、查看汇总、归档和全局管理、维护薪资和晋升流程 |
| 部门经理 | 查看团队绩效、审批下属考核、提交反馈、跟进 PIP、参与晋升评审 |
| 员工 | 查看个人考核、提交自评、查看目标、接收 PIP、查看个人薪资结果 |

## 4. 技术框架约束

### 4.1 后端框架

后端必须基于现有目录：

- `cool-admin-midway`

后端能力边界：

- Midway 3
- TypeORM
- MySQL2
- `@cool-midway/core`
- `@CoolController` + `service` + `entity` 模式

### 4.2 前端框架

前端必须基于现有目录：

- `cool-admin-vue`

前端能力边界：

- Vue 3
- TypeScript
- Element Plus
- Pinia
- Vite
- `@cool-vue/crud`
- `echarts` / `vue-echarts`
- Tailwind CSS

### 4.3 不允许超出的技术范围

首期规划和实现不新增：

- 新前端框架
- 新后端框架
- 新 ORM
- 新图表库
- 工作流引擎
- 外部问卷系统
- 独立 BI / 报表服务

### 6.2 页面粒度升级规则

原“前端页面文档”只明确了 `/assessments` 和 `/goals` 的通用 CRUD 页面。升级后按下面方式处理：

1. “已发起考核”继承 `assessment` 的列表、详情、创建、编辑能力。
2. “我的考核”复用 `assessment` 详情结构，但页面权限、默认筛选和可操作按钮不同。
3. “待我审批”复用 `assessment` 数据模型，但只保留审批相关动作。
4. “目标地图”继承 `goal` 的列表、详情、创建、编辑能力，并改成绩效域菜单入口。
5. 新增的“指标库、360 环评、PIP、薪资、晋升”采用与现有 CRUD 相同的页面组织方式。

### 6.3 页面布局统一规范

所有后台页面统一遵循下面结构：

1. 标题区：页面标题、面包屑、主要操作按钮
2. 筛选区：搜索项、状态、周期、角色、部门等筛选条件
3. 内容区：
   - 看板类：统计卡片 + 图表 + 趋势/分布
   - 列表类：表格 + 分页 + 行内操作
   - 详情类：基础信息 + 状态信息 + 记录时间线 + 动作区
4. 状态区：加载态、空态、错误态、无权限态

### 6.4 页面结构参考基线

首期后台页面结构不从零设计，统一参考 [18-页面结构与原型映射.md](/Users/shaotongli/Documents/xuedao/performance-management-system/docs/18-页面结构与原型映射.md)。

结论固定如下：

1. 参考来源是 `jixiao` 已实现的绩效页面结构，不是重新发明一套页面分层。
2. 复用的是模块拆分、列表组织、详情承载和动作区布局，不直接复制 `jixiao` 的独立工程代码。
3. `jixiao` 的单一 `/performance` 导航页只作为原型参考，首期仍按 Cool 动态菜单树进入各业务页。
4. `Assessments.vue` 在首期不作为单独菜单照搬，而是拆成“我的考核 / 已发起考核 / 待我审批”三个视角页。
5. `GoalMap.vue` 作为目标地图主结构基线，`ActionPlan.vue` 不独立成首期菜单，而是并入目标进度和 PIP 跟进能力。
6. `Feedback.vue` 和 `Feedback360.vue` 合并映射到一个 `/performance/feedback` 菜单页内部。

## 8. 数据与状态升级约束

### 8.1 核心状态

建议在首期统一下面几类状态：

- 评估单：`draft`、`submitted`、`approved`、`rejected`
- 目标：`draft`、`in-progress`、`completed`、`cancelled`
- PIP：`draft`、`active`、`completed`、`closed`
- 晋升：`draft`、`reviewing`、`approved`、`rejected`

### 8.2 敏感数据规则

下面数据默认按高风险处理：

1. 绩效分数明细
2. 审批意见
3. 薪资信息
4. 晋升评审结论

要求：

- 后端做权限判断
- 前端做显隐和空态处理
- 日志、导出、报错文案不泄露敏感字段

## 11. 升级后的结论

合并升级后的统一结论如下：

1. 可以在 `xuedao` 内单独建立 `performance-management-system/` 目录承接这个项目。
2. 可以将两份原始文档合并升级，而且应该这么做，否则后续实现会长期受“两套粒度”冲突影响。
3. 首期项目应以 `cool-admin-midway + cool-admin-vue` 为唯一实现边界。
4. 首期虽然纳入完整绩效域，但必须按阶段推进，先闭环、后增强、再做敏感模块。
