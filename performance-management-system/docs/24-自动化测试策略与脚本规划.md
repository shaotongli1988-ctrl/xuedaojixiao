# 自动化测试策略与脚本规划

## 文档职责

这份文档是“自动化测试策略和脚本规划”的唯一事实源。

它只负责定义：

1. 当前仓库已有测试脚本和能力边界
2. 首期允许使用的自动化测试方式
3. 首期不引入的测试框架范围
4. 绩效模块测试文件落点和命名规则
5. 各模块最小自动化验证要求

## 不负责内容

这份文档不负责定义：

- 业务测试矩阵明细
- UAT 验收步骤
- 联调环境账号和种子数据

这些内容分别以 `09`、`14` 文档为准。

## 首期总策略

首期自动化测试采用“保守收敛”策略：

1. 不新增第二套测试技术栈作为首期阻塞项
2. 优先复用当前仓库已有脚本
3. 对高风险纯逻辑补齐后端 `jest` 测试
4. 前端先以 `type-check` 和 `build` 作为自动化门禁
5. API 主流程继续以联调工具和关键手测为主
6. 菜单、路由、权限和高价值文档同步问题优先通过轻量仓库一致性守卫阻断，不等到联调再暴露

## 项目级红线

1. 任何行为变化都必须有自动化验证证据，不能只靠聊天确认、手工截图或口头说明通过。
2. 前端受影响时，`cool-admin-vue` 的 `npm run type-check` 和 `npm run build` 为必跑项。
3. 后端若涉及纯逻辑、状态机、等级计算、聚合统计、权限判断、默认值或兼容分支变化，必须补齐对应 `jest` 验证。
4. 若当前改动确实没有可稳定抽离的后端纯逻辑，允许不补 `jest`，但必须在任务卡中写清原因、替代验证方式和剩余风险。
5. 自动化验证结果必须可留痕，未留痕视为未执行。
6. 命中仓库一致性守卫范围的改动，必须保留守卫通过结果；守卫失败不能用口头说明替代。
7. 仓库级本地门禁和 CI 必须同时覆盖 `format / lint / typecheck|compile / build` 的统一基线；若全仓历史债导致无法直接全量执行，至少要落到“变更文件强阻断”的自动化脚本，不能退回人工判断。
8. `cool-admin-midway` 的本地 `npm run lint` 若依赖 `.shared-deps` 离线依赖树，必须由仓库脚本自动补齐兼容层，不能要求开发者手工修改 `node_modules` 后再继续验证。
9. 对这类本地兼容引导脚本，仓库回归测试必须至少包含一条“临时工作区真实执行”用例，验证脚本会在空白或半成品 `.shared-deps` 目录中产出关键兼容文件，而不只是检查 `package.json` 是否接线。
10. 若仓库提供离线依赖安装脚本，例如 `cool-admin-midway/scripts/install-local-deps.mjs`，安装脚本完成后也必须自动调用兼容引导脚本，确保“先安装依赖”和“先跑 lint”两条路径最终收敛到同一 `.shared-deps` 状态。
11. 若安装脚本支持“仅修复兼容层”的快速收口模式，例如 `--compat-only` 或 `npm run deps:local:repair`，该模式也必须纳入回归测试，确保在无网络或临时工作区中仍可直接验证本地自愈能力。
12. 若安装脚本支持本地 mock registry 或本地 tarball 元数据注入，例如 `--registry-dir`，仓库回归测试必须至少覆盖一条“临时工作区 + 本地 registry 完整安装链路”用例，验证默认安装路径在无公网条件下也能完成依赖落盘并收口兼容层。

## 仓库一致性守卫

为降低“代码能跑但菜单、权限、文档已经漂移”的提交风险，仓库额外提供一组变更感知的强阻断守卫，统一由 `scripts/run-repo-consistency-guards.mjs` 聚合执行。

### 当前守卫清单

1. `scripts/check-xuedao-ssot-manifest.mjs`
   - 校验 `contracts/ssot/xuedao-ssot-manifest.yaml` 中声明的关键文件、目录、脚本入口与报告根目录都真实存在
   - 当前至少覆盖仓库级 OpenAPI 主源、权限主源、`user` 鉴权语义、`base/user/dict` 运行时配置、环境变量 catalog、schema / migration ownership catalog、菜单 / 路由 / view topology catalog、`cool-uni` 移动端共享契约、`performance` 状态机主源、`performance` 业务字典、`dict` 业务字典聚合、`base/user/dict` 共享错误语义、`performance` 错误目录、repo consistency 聚合入口、`pre-push` gate 入口、交付记录模板、`reports/delivery` 报告目录以及 manifest 声明的生成消费者路径
2. `scripts/check-xuedao-ssot-conformance.mjs`
   - 按仓库级 SSOT 标准检查 `xuedao` 是否同时具备主源、运行时 guard 绑定、文档同步、strict-change 级 change/verification record 和最小报告留痕
   - 当前输出 `PASS / WARN / FAIL`，并按 `P0 / P1 / P2` 分类问题；其中 `P0 / P1` 为阻断项，`P2` 为提醒项
   - 默认会写出 `reports/delivery/xuedao-ssot-conformance.latest.{md,json}`，CI 可通过显式 `--report-md/--report-json` 改写为批次化文件名
3. `scripts/check-environment-config-ssot.mjs`
   - 校验 `contracts/ssot/environment-config.catalog.json` 与 curated runtime/config/automation roots 中实际使用的环境变量名是否一致
   - 当前先作为 conformance 报告链中的 report-only 治理守卫，不直接升级为 pre-push 强阻断
4. `scripts/check-database-schema-ssot.mjs`
   - 校验 `contracts/ssot/database-schema.catalog.json` 是否完整覆盖 `run-migrations.mjs`、`entities.ts`、schema bootstrap 入口和全部 migration 文件归属
   - 当前先作为 conformance 报告链中的 report-only 治理守卫，不直接升级为 pre-push 强阻断
5. `scripts/check-directory-naming-conflicts.mjs`
   - 检查 `cool-admin-vue/src/modules/performance/views/` 下与本次变更相关的同级目录命名冲突
   - 重点阻断 `resumePool` / `resumePool` 这类归一化后同名的目录并存
6. `scripts/check-menu-route-viewpath-drift.mjs`
   - 对齐 `menu.json`、`10-路由与菜单映射.md` 与真实 `viewPath` 文件
   - 只检查本次变更触达的 performance 页面菜单
7. `scripts/check-permission-key-alignment.mjs`
   - 对 `performance:*` 权限键执行 `consumer -> producer` 和 `producer -> consumer` 双向对账
   - 同时要求权限键出现在 [06-权限矩阵.md](/Users/shaotongli/Documents/xuedao/performance-management-system/docs/06-权限矩阵.md)
8. `scripts/check-doc-contract-writeback.mjs`
   - 命中 API、权限、路由、脱敏、自动化脚本变化时，强制检查 `04 / 06 / 10 / 12 / 24` 是否已回写

9. `scripts/sync-repo-openapi-ssot.mjs`
   - 以 `cool-admin-vue/build/cool/eps.json` 为输入，把 `base`、`demo`、`dict`、`plugin`、`recycle`、`space`、`task`、`user` 的现有 admin EPS 资源回写到 `contracts/openapi/xuedao.openapi.json`
   - 只负责仓库级主源收口，不负责把非 `performance` 前端动态 EPS 消费改造成静态 service wrapper
10. `scripts/sync-performance-openapi-ssot.mjs`
   - 以 `cool-admin-midway/src/modules/performance/domain/registry/contract-source.{json,mjs}` 登记的 producer 覆盖范围为入口，再结合 `performance` controller/service/types，把 `performance` 资源回写到 `contracts/openapi/xuedao.openapi.json`
   - 禁止回读 `cool-admin-vue/src/modules/performance/generated/*` 这类消费者生成物，避免生成类型反向污染 OpenAPI 主源

   - 当前会根据 registry source 中登记的 web / uni target 刷新对应 generated files，再由后续守卫和类型检查消费
12. `scripts/check-performance-contract-closure.mjs`
   - 校验 `performance` backend registry source 是否完整闭合 producer 覆盖、generated target、shared support adapter 与 uni wrapper 引用链
   - 对应仓库级 manifest 中的独立 machine source：`sourceOfTruth.performanceContractSource`
13. `scripts/check-performance-domain-model-ssot.mjs`
   - 扫描 `cool-uni/types/performance-*.ts` 与 admin 课程学习 wrapper
   - 阻断 wrapper 层重新手写 `Record / PageResult / Status` 领域模型
   - 要求业务实体模型统一回到 `generated/*` 或 `ApiResponse.data` 派生别名
   - 对高优先级 `performance` contract adapter，额外阻断局部 runtime enum 常量回流，必须统一复用 `cool-admin-vue/src/modules/performance/shared/contract-enums.ts`
14. `scripts/sync-eps-openapi-ssot.mjs`
   - 以 `contracts/openapi/xuedao.openapi.json` 和 `cool-admin-vue/build/cool/eps.d.ts` 为输入，生成 `cool-admin-vue/build/cool/eps.ssot.d.ts`
   - 当前负责把非 `performance` 动态 EPS 的方法签名、请求类型和响应类型回收到仓库级 OpenAPI 主源

### 触发策略

1. 守卫默认按“本次变更命中相关路径才执行”。
2. 目录命名守卫只检查与本次变更相关的页面目录层级，不对无关目录做全仓静态阻断。
3. 菜单 / 路由 / `viewPath` 守卫在命中 `menu.json`、`menu.ts` 或 performance 页面目录时执行。
4. 权限键守卫在命中 performance 前后端源码或 `menu.json` 时执行。
5. 高价值文档回写守卫在命中 API、权限、路由、脱敏、自动化脚本变化时执行。

## 首期允许使用的自动化方式

| 层级 | 首期是否纳入 | 方式 | 定位 |
| --- | --- | --- | --- |
| 后端单元测试 | 条件必须 | `cool-admin-midway` 的 `jest` | 涉及纯逻辑、状态、权限、聚合等行为变化时必须补齐 |
| 后端覆盖率 | 否，按需 | `npm run cov` | 非首期硬门槛 |
| 前端类型检查 | 是 | `cool-admin-vue` 的 `npm run type-check` | 必须通过 |
| 前端构建验证 | 是 | `cool-admin-vue` 的 `npm run build` | 必须通过 |
| 前端组件单测 | 否 | 暂不作为首期门槛 | 可二期评估 |
| E2E 自动化 | 否 | 暂不引入 Playwright/Cypress | 二期再评估 |

## 首期明确不引入的内容

以下内容不作为首期自动化测试基线：

1. `Playwright`
2. `Cypress`
3. `Vitest`
4. 自建浏览器端 E2E 测试平台
5. 为测试额外新建第二套前端或后端运行环境

原因：

1. 当前仓库没有现成配置
2. 与“严格沿用现有技术架构”原则不符
3. 首期更需要先保证模块主链落地和契约稳定

## 测试文件落点与命名规则

### 后端绩效模块

后端绩效测试文件统一落在：

- `cool-admin-midway/test/performance/`

推荐命名：

- `assessment.service.test.ts`
- `goal.service.test.ts`
- `dashboard.service.test.ts`
- `feedback.service.test.ts`
- `pip.service.test.ts`
- `promotion.service.test.ts`
- `salary.service.test.ts`

补充规则：

1. 优先测 service 层纯逻辑，不优先测自动化路由和复杂启动装配。
2. 若某逻辑依赖数据库查询，优先抽纯函数或小型 service 方法后再测。
3. 不为了测试强行改动整体架构，但允许把复杂计算逻辑从控制器中抽离。

### 前端绩效模块

前端首期没有单独测试文件硬要求。

首期前端自动化验证统一通过：

1. `cool-admin-vue npm run type-check`
2. `cool-admin-vue npm run build`

如果后续二期引入前端测试框架，再单独定义前端测试目录规范。
