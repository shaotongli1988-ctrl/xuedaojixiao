# 自动化测试策略与脚本规划

## 文档职责

这份文档是“自动化测试策略和脚本规划”的唯一事实源。

它只负责定义：

1. 当前仓库已有测试脚本和能力边界
2. 首期允许使用的自动化测试方式
3. 首期不引入的测试框架范围
4. 绩效模块测试文件落点和命名规则
5. 各模块最小自动化验证要求

## 不负责内容

这份文档不负责定义：

- 业务测试矩阵明细
- UAT 验收步骤
- 联调环境账号和种子数据

这些内容分别以 `09`、`14` 文档为准。

## 首期总策略

首期自动化测试采用“保守收敛”策略：

1. 不新增第二套测试技术栈作为首期阻塞项
2. 优先复用当前仓库已有脚本
3. 对高风险纯逻辑补齐后端 `jest` 测试
4. 前端先以 `type-check` 和 `build` 作为自动化门禁
5. API 主流程继续以联调工具和关键手测为主

## 项目级红线

1. 任何行为变化都必须有自动化验证证据，不能只靠聊天确认、手工截图或口头说明通过。
2. 前端受影响时，`cool-admin-vue` 的 `npm run type-check` 和 `npm run build` 为必跑项。
3. 后端若涉及纯逻辑、状态机、等级计算、聚合统计、权限判断、默认值或兼容分支变化，必须补齐对应 `jest` 验证。
4. 若当前改动确实没有可稳定抽离的后端纯逻辑，允许不补 `jest`，但必须在任务卡中写清原因、替代验证方式和剩余风险。
5. 自动化验证结果必须可留痕，未留痕视为未执行。

## 首期允许使用的自动化方式

| 层级 | 首期是否纳入 | 方式 | 定位 |
| --- | --- | --- | --- |
| 后端单元测试 | 条件必须 | `cool-admin-midway` 的 `jest` | 涉及纯逻辑、状态、权限、聚合等行为变化时必须补齐 |
| 后端覆盖率 | 否，按需 | `npm run cov` | 非首期硬门槛 |
| 前端类型检查 | 是 | `cool-admin-vue` 的 `npm run type-check` | 必须通过 |
| 前端构建验证 | 是 | `cool-admin-vue` 的 `npm run build` | 必须通过 |
| 前端组件单测 | 否 | 暂不作为首期门槛 | 可二期评估 |
| E2E 自动化 | 否 | 暂不引入 Playwright/Cypress | 二期再评估 |

## 首期明确不引入的内容

以下内容不作为首期自动化测试基线：

1. `Playwright`
2. `Cypress`
3. `Vitest`
4. 自建浏览器端 E2E 测试平台
5. 为测试额外新建第二套前端或后端运行环境

原因：

1. 当前仓库没有现成配置
2. 与“严格沿用现有技术架构”原则不符
3. 首期更需要先保证模块主链落地和契约稳定

## 测试文件落点与命名规则

### 后端绩效模块

后端绩效测试文件统一落在：

- `cool-admin-midway/test/performance/`

推荐命名：

- `assessment.service.test.ts`
- `goal.service.test.ts`
- `dashboard.service.test.ts`
- `feedback.service.test.ts`
- `pip.service.test.ts`
- `promotion.service.test.ts`
- `salary.service.test.ts`

补充规则：

1. 优先测 service 层纯逻辑，不优先测自动化路由和复杂启动装配。
2. 若某逻辑依赖数据库查询，优先抽纯函数或小型 service 方法后再测。
3. 不为了测试强行改动整体架构，但允许把复杂计算逻辑从控制器中抽离。

### 前端绩效模块

前端首期没有单独测试文件硬要求。

首期前端自动化验证统一通过：

1. `cool-admin-vue npm run type-check`
2. `cool-admin-vue npm run build`

如果后续二期引入前端测试框架，再单独定义前端测试目录规范。

## 各模块最小自动化验证要求

### 模块 0：模块骨架与基础配置

必须通过：

1. 前端 `npm run type-check`
2. 前端 `npm run build`

### 模块 1：评估单

必须补齐至少 1 组后端 `jest` 测试，覆盖：

1. `totalScore = sum(score * weight) / sum(weight)`
2. 等级边界 `S/A/B/C`
3. `draft -> submitted`
4. `submitted -> approved / rejected`
5. 非法状态流转拒绝

同时必须通过：

1. 前端 `npm run type-check`
2. 前端 `npm run build`

### 模块 2：目标地图

必须补齐至少 1 组后端 `jest` 测试，覆盖：

1. `progressRate` 计算
2. `targetValue > 0`
3. `startDate <= endDate`
4. `completed` 后不可继续更新

同时必须通过：

1. 前端 `npm run type-check`
2. 前端 `npm run build`

### 模块 3：驾驶舱基础聚合

必须补齐至少 1 组后端 `jest` 测试，覆盖：

1. `averageScore`
2. `pendingApprovalCount`
3. `goalCompletionRate`
4. `departmentDistribution`
5. `gradeDistribution`

同时必须通过：

1. 前端 `npm run type-check`
2. 前端 `npm run build`

### 模块 4：指标库

若改动涉及下面语义，必须补齐至少 1 组后端 `jest`，优先覆盖：

1. 重复编码校验
2. 启用/禁用状态过滤

若当前阶段仅落标准 CRUD 壳层，且无独立纯逻辑，允许暂不补 `jest`，但必须在任务卡说明原因。

同时必须通过：

1. 前端 `npm run type-check`

1. 单任务同评价人只能提交一次

同时必须通过：

1. 前端 `npm run type-check`
2. 前端 `npm run build`

1. `draft -> active`

同时必须通过：

1. 前端 `npm run type-check`

### 模块 7：晋升管理

必须补齐至少 1 组后端 `jest` 测试，覆盖：

1. `draft -> reviewing`
2. `reviewing -> approved / rejected`
3. 非法状态拒绝
4. 独立创建时必须有原因

同时必须通过：

1. 前端 `npm run type-check`
2. 前端 `npm run build`

### 模块 8：薪资管理

必须补齐至少 1 组后端 `jest` 测试，覆盖：

1. `draft -> confirmed`
2. `confirmed -> archived`
3. `confirmed` 后只能通过 `changeAdd` 调整

同时必须通过：

1. 前端 `npm run type-check`
2. 前端 `npm run build`

1. 后端执行了哪些 `jest` 文件
2. 前端 `type-check` 结果

4. 哪些模块未补自动化测试及原因
