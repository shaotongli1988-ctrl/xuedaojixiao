## 首期总策略

首期自动化测试采用“保守收敛”策略：

1. 不新增第二套测试技术栈作为首期阻塞项
2. 优先复用当前仓库已有脚本
3. 对高风险纯逻辑补齐后端 `jest` 测试
4. 前端先以 `type-check` 和 `build` 作为自动化门禁
5. API 主流程继续以联调工具和关键手测为主
6. 菜单、路由、权限和高价值文档同步问题优先通过轻量仓库一致性守卫阻断，不等到联调再暴露

## 项目级红线

1. 任何行为变化都必须有自动化验证证据，不能只靠聊天确认、手工截图或口头说明通过。
2. 前端受影响时，`cool-admin-vue` 的 `npm run type-check` 和 `npm run build` 为必跑项。
3. 后端若涉及纯逻辑、状态机、等级计算、聚合统计、权限判断、默认值或兼容分支变化，必须补齐对应 `jest` 验证。
4. 若当前改动确实没有可稳定抽离的后端纯逻辑，允许不补 `jest`，但必须在任务卡中写清原因、替代验证方式和剩余风险。
5. 自动化验证结果必须可留痕，未留痕视为未执行。
6. 命中仓库一致性守卫范围的改动，必须保留守卫通过结果；守卫失败不能用口头说明替代。

## 仓库一致性守卫

为降低“代码能跑但菜单、权限、文档已经漂移”的提交风险，仓库额外提供一组变更感知的强阻断守卫，统一由 `scripts/run-repo-consistency-guards.mjs` 聚合执行。

### 当前守卫清单

1. `scripts/check-directory-naming-conflicts.mjs`
   - 检查 `cool-admin-vue/src/modules/performance/views/` 下与本次变更相关的同级目录命名冲突
   - 重点阻断 `resumePool` / `resumePool` 这类归一化后同名的目录并存
2. `scripts/check-menu-route-viewpath-drift.mjs`
   - 对齐 `menu.json`、`10-路由与菜单映射.md` 与真实 `viewPath` 文件
   - 只检查本次变更触达的 performance 页面菜单
3. `scripts/check-permission-key-alignment.mjs`
   - 对 `performance:*` 权限键执行 `consumer -> producer` 和 `producer -> consumer` 双向对账
   - 同时要求权限键出现在 [06-权限矩阵.md](/Users/shaotongli/Documents/xuedao/performance-management-system/docs/06-权限矩阵.md)
4. `scripts/check-doc-contract-writeback.mjs`
   - 命中 API、权限、路由、脱敏、自动化脚本变化时，强制检查 `04 / 06 / 10 / 12 / 24` 是否已回写

### 触发策略

1. 守卫默认按“本次变更命中相关路径才执行”。
2. 目录命名守卫只检查与本次变更相关的页面目录层级，不对无关目录做全仓静态阻断。
3. 菜单 / 路由 / `viewPath` 守卫在命中 `menu.json`、`menu.ts` 或 performance 页面目录时执行。
4. 权限键守卫在命中 performance 前后端源码或 `menu.json` 时执行。
5. 高价值文档回写守卫在命中 API、权限、路由、脱敏、自动化脚本变化时执行。

### 执行位置

1. `pre-push` 本地门禁默认执行这组守卫。
2. 需要手动排查时，可直接运行：

```bash
node ./scripts/run-repo-consistency-guards.mjs
```

## 首期允许使用的自动化方式

| 层级 | 首期是否纳入 | 方式 | 定位 |
| --- | --- | --- | --- |
| 后端单元测试 | 条件必须 | `cool-admin-midway` 的 `jest` | 涉及纯逻辑、状态、权限、聚合等行为变化时必须补齐 |
| 后端覆盖率 | 否，按需 | `npm run cov` | 非首期硬门槛 |
| 前端类型检查 | 是 | `cool-admin-vue` 的 `npm run type-check` | 必须通过 |
| 前端构建验证 | 是 | `cool-admin-vue` 的 `npm run build` | 必须通过 |
| 前端组件单测 | 否 | 暂不作为首期门槛 | 可二期评估 |
| E2E 自动化 | 否 | 暂不引入 Playwright/Cypress | 二期再评估 |

## 首期明确不引入的内容

以下内容不作为首期自动化测试基线：

1. `Playwright`
2. `Cypress`
3. `Vitest`
4. 自建浏览器端 E2E 测试平台
5. 为测试额外新建第二套前端或后端运行环境

原因：

1. 当前仓库没有现成配置
2. 与“严格沿用现有技术架构”原则不符
3. 首期更需要先保证模块主链落地和契约稳定
