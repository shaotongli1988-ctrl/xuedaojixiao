# 回滚说明

## 文档职责

这份文档是“回滚策略”的唯一事实源。

它只负责定义：

1. 代码回滚方式
2. 数据回滚方式
3. 权限和菜单回滚方式
4. 配置回滚方式
5. 回滚风险提示

## 不负责内容

这份文档不负责定义：

- 正向实施步骤
- 页面功能需求
- 测试用例清单

## 总体回滚原则

1. 首期绩效模块必须采用可回退的增量建设方式。
2. 代码、菜单、权限、数据四类回滚路径都要独立可执行。
3. 不允许在同一发布中执行不可逆删表、删列、清空数据式回滚。
4. 涉及薪资和审批数据时，回滚前必须先做数据备份或导出快照。

## 无迁移时的代码回滚方式

适用场景：

1. 只改前端页面
2. 只改后端业务逻辑
3. 未执行数据库结构变更

回滚方式：

1. 回退到上一稳定代码版本。
2. 同步恢复上一版菜单和权限数据。
3. 重新发布前后端服务。
4. 按关键主路径做最小回归：
   - 登录
   - 菜单加载
   - 旧功能可访问

## 有表结构调整时的回滚前提

首期数据库变更要求默认采用“只增不删”策略：

1. 新增表允许保留，不要求发布当日物理删除。
2. 新增字段允许保留，不要求发布当日回退 DDL。
3. 如果功能回滚，以停入口、停菜单、停权限为优先，不以删库删表为优先。

只有在以下条件全部满足时，才允许做结构级回退：

1. 已确认没有生产数据依赖新表或新字段
2. 已完成可恢复备份
3. 已验证回退脚本
4. 已确认不会影响旧版本启动

## 菜单和权限变更回退方式

1. 回滚菜单时优先删除或禁用新增的绩效菜单节点。
2. 同步撤销绩效模块按钮权限授权关系。
3. 如果代码已回退但菜单未回退，必须先隐藏入口，避免用户点击空页面。
4. 菜单回退后重新验证动态菜单装配结果。

### 权限授权回退方式

1. 回滚权限时，不只回退菜单节点，还必须同步撤销 `base_sys_role_menu` 中的绩效授权关系。
2. 如果角色聚合字段 `base_sys_role.menuIdList` 已写入绩效菜单或按钮权限，必须一并恢复到上一稳定版本的授权结果。
3. 若本次问题只涉及授权错误，优先做“撤权 + 隐藏入口 + 保留数据”，不要把权限回退误做成数据删除。
4. 回退后至少复核 `hr_admin`、`manager_rd`、`employee_platform` 三类账号的 `permmenu` 结果，确认菜单显隐和按钮权限同步生效。

## 数据回滚与修复要求

### 评估、目标、环评、PIP、晋升

1. 如果是逻辑错误，优先做数据修复，不直接删除业务记录。
2. 审批类数据回退必须保留审计痕迹。
3. 环评记录如果已提交，不允许静默覆盖为未提交。

### 薪资

1. 薪资数据回滚前必须导出受影响记录快照。
2. 已确认薪资如需修正，优先新增调整记录，不直接覆盖历史金额。
3. 任何涉及金额的修复都必须记录操作者、原因和时间。

## 配置回滚方式

1. 如果仅配置、菜单或权限导致问题，优先回滚配置而不是回滚全部代码。
2. 如果驾驶舱统计异常，可先关闭绩效菜单入口，再排查聚合逻辑。
3. 如果敏感字段脱敏失效，优先下线相关页面入口和导出能力。
4. 当前本地验证基线相关配置主要是 [cool-admin-midway/src/config/config.local.ts](/Users/shaotongli/Documents/xuedao/cool-admin-midway/src/config/config.local.ts) 与 [cool-admin-vue/src/config/proxy.ts](/Users/shaotongli/Documents/xuedao/cool-admin-vue/src/config/proxy.ts)；如果只是端口、代理或本地 schema 口径异常，优先恢复这两处配置，不要误回滚业务数据。
5. 若需要从当前 latest 入口回退到上一稳定入口，必须同时更新交付说明中的有效联调入口和版本记录，避免发布说明与实际入口漂移。

## 不可逆操作风险声明

以下操作视为高风险，不允许无备份直接执行：

1. 删除绩效主表或清空业务数据
2. 删除薪资调整记录
3. 覆盖已确认薪资金额
4. 删除已提交的环评反馈
5. 删除已审批评估单

## 回滚后最低验证

1. 旧版本可正常启动。
2. 动态菜单不出现失效入口。
3. 非绩效模块不受影响。
4. 敏感数据不因回滚过程暴露。
5. 关键旧路径最小可用。
