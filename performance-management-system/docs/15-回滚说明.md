## 二期主题 5：自动审批流回退到手工审批主链冻结基线（2026-04-17）

说明：

1. 本节定义二期主题 5 自动审批流在阶段 0 进入开发前已冻结的“回退到现有手工审批主链”口径。
2. 该回退口径只适用于 `assessment / promotion` 两类对象，不扩展到 `PIP / goal / feedback / salary / suggestion / dashboard`。

### 当前冻结结论

1. 当前唯一结论：`自动审批流回退到手工审批主链口径已冻结，可进入开发`
2. 自动审批流不是第二套对外业务状态体系；回退的目标是恢复现有手工审批入口，而不是删除审批实例审计。

### 回退触发条件

1. 自动审批实例进入 `pending_resolution` 或 `manual_pending`，且 `HR` 判定当前实例需要切回现有手工审批主链。
2. 自动审批配置被关闭，但已有未终态实例仍需继续处理。
3. 自动审批链路出现权限、解析或通知异常，需要先回到现有手工审批主链止血。

### 回退目标

1. 自动审批实例结束并记为 `terminated`。
2. 源业务对象继续停留在其既有人工审批主链可处理状态：
   - `assessment` 保持 `submitted`
   - `promotion` 保持 `reviewing`
3. 后续处理入口切回既有手工审批接口：
   - `/admin/performance/assessment/approve`
   - `/admin/performance/assessment/reject`
   - `/admin/performance/promotion/review`

### 回退对象

1. `performance_approval_config.enabled`
2. `performance_approval_instance.status / fallbackReason / fallbackOperatorId / finishTime`
3. `performance_approval_action_log`
4. 源业务对象当前可继续处理的手工审批入口与既有权限链路

### 回退步骤

1. `HR` 先按对象类型关闭自动审批配置，防止新实例继续创建。
2. `HR` 对目标实例执行 `approval-flow/fallback`，并写入回退原因。
3. 系统将自动审批实例记为 `terminated`，保留节点快照和全部审计日志，不删除历史记录。
4. 系统不得自动把源业务对象改成 `approved / rejected`；只恢复到现有手工审批入口继续处理。
5. 回退完成后，前端和调用方不再对该对象实例调用 `approval-flow/approve`、`approval-flow/reject`、`approval-flow/transfer` 等自动审批动作接口。

### 回退后的最小验证

1. `approval-flow/info` 返回该实例 `status = terminated`，且带出 `fallbackReason`。
2. 源业务对象重新出现在既有手工审批待办入口：
   - 评估单进入既有“待我审批”
   - 晋升单继续按既有 `reviewing` 链路展示
3. 既有手工审批接口可继续成功处理该对象。
4. 审批实例历史节点和审计留痕仍可查询，不因回退被物理删除。

## 总体回滚原则

1. 首期绩效模块必须采用可回退的增量建设方式。
2. 代码、菜单、权限、数据四类回滚路径都要独立可执行。
3. 不允许在同一发布中执行不可逆删表、删列、清空数据式回滚。
4. 涉及薪资和审批数据时，回滚前必须先做数据备份或导出快照。

## 无迁移时的代码回滚方式

适用场景：

1. 只改前端页面
2. 只改后端业务逻辑
3. 未执行数据库结构变更

回滚方式：

1. 回退到上一稳定代码版本。
2. 同步恢复上一版菜单和权限数据。
3. 重新发布前后端服务。
4. 按关键主路径做最小回归：
   - 登录
   - 菜单加载
   - 旧功能可访问

## 有表结构调整时的回滚前提

首期数据库变更要求默认采用“只增不删”策略：

1. 新增表允许保留，不要求发布当日物理删除。
2. 新增字段允许保留，不要求发布当日回退 DDL。
3. 如果功能回滚，以停入口、停菜单、停权限为优先，不以删库删表为优先。

只有在以下条件全部满足时，才允许做结构级回退：

1. 已确认没有生产数据依赖新表或新字段
2. 已完成可恢复备份
3. 已验证回退脚本
4. 已确认不会影响旧版本启动

## 菜单和权限变更回退方式

1. 回滚菜单时优先删除或禁用新增的绩效菜单节点。
2. 同步撤销绩效模块按钮权限授权关系。
3. 如果代码已回退但菜单未回退，必须先隐藏入口，避免用户点击空页面。
4. 菜单回退后重新验证动态菜单装配结果。

### 权限授权回退方式

1. 回滚权限时，不只回退菜单节点，还必须同步撤销 `base_sys_role_menu` 中的绩效授权关系。
2. 如果角色聚合字段 `base_sys_role.menuIdList` 已写入绩效菜单或按钮权限，必须一并恢复到上一稳定版本的授权结果。
3. 若本次问题只涉及授权错误，优先做“撤权 + 隐藏入口 + 保留数据”，不要把权限回退误做成数据删除。
4. 回退后至少复核 `hr_admin`、`manager_rd`、`employee_platform` 三类账号的 `permmenu` 结果，确认菜单显隐和按钮权限同步生效。

## 不可逆操作风险声明

以下操作视为高风险，不允许无备份直接执行：

1. 删除绩效主表或清空业务数据
2. 删除薪资调整记录
3. 覆盖已确认薪资金额
