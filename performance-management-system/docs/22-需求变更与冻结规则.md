# 需求变更与冻结规则

## 文档职责

这份文档是“需求冻结和变更治理”的唯一事实源。

它只负责定义：

1. 需求冻结点
2. 变更类型
3. 变更处理路径
4. 变更后的文档回写要求
5. 版本记录要求

## 不负责内容

这份文档不负责定义：

- 原始需求全文
- 模块实现顺序
- 缺陷关闭标准

这些内容分别以 `01`、`19`、`23` 文档为准。

## 冻结规则

### 首次冻结

满足以下条件后，视为进入“首期需求冻结”：

1. 首期范围明确
2. 技术边界明确
3. 状态、权限、菜单、字段和导出边界明确
4. 首个模块可直接开始开发

### 冻结后的默认规则

1. 不在当前周期内临时引入新角色模型。
2. 不在当前周期内临时引入新技术框架。
3. 不把二期能力挤进首期开发中。
4. 不在未回写文档前直接改代码实现。
5. 不以聊天确认替代 [25-项目最高准则.md](/Users/shaotongli/Documents/xuedao/performance-management-system/docs/25-项目最高准则.md) 和唯一事实源回写。

## 变更分类

| 类型 | 定义 | 默认处理方式 |
| --- | --- | --- |
| `clarification` | 原文档已有方向，但表述不清，需要明确化 | 直接回写对应唯一事实源，并记版本 |
| `scope-change` | 首期新增或删减功能范围 | 先评估影响，再决定是否进入当前周期 |
| `contract-change` | 接口、字段、状态、权限、导出语义变化 | 必须回写 `03`、`04`、`05`、`06`、`07`、`10`、`12` 受影响文档 |
| `process-change` | 模块顺序、阶段门禁、验收、自动化门禁或交付流程变化 | 必须回写 `08`、`17`、`19`、`20`、`21`、`23`、`24`、`25` |
| `defect-fix` | 修正偏离既定事实源的实现错误 | 不视为需求新增，但要更新任务卡和问题台账 |

## 变更处理路径

1. 先判断是澄清、范围变更、契约变更、流程变更还是缺陷修复。
2. 判断影响范围：
   - 文档
   - 代码
   - 菜单权限
   - 联调数据
   - 测试用例
3. 若影响首期边界、角色、权限、状态、API 契约，先暂停编码。
4. 先更新唯一事实源，再继续代码和测试。
5. 更新 [13-版本与变更记录.md](/Users/shaotongli/Documents/xuedao/performance-management-system/docs/13-版本与变更记录.md)。

## 必须暂停编码的变更场景

1. 新增首期功能模块
2. 改变审批角色
3. 改变状态机
4. 改变权限键或菜单结构
5. 改变核心接口字段、响应结构或错误语义
6. 改变敏感字段可见范围
7. 改变自动化测试门槛、豁免条件或任务关闭规则

## 版本记录要求

1. 每次实质性变更都要回写 [13-版本与变更记录.md](/Users/shaotongli/Documents/xuedao/performance-management-system/docs/13-版本与变更记录.md)。
2. 版本记录必须写明：
   - 变更日期
   - 变更内容
   - 影响文档
   - 备注
3. 聊天确认不替代版本记录。

## 禁止项

1. 不允许只改代码不改文档。
2. 不允许只改某一份引用文档，不改唯一事实源主文档。
3. 不允许把“缺陷修复”包装成“需求变更”绕过冻结。
4. 不允许未评估影响就把二期内容纳入首期。
