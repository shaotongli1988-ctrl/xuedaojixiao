# 实施计划

## 文档职责

这份文档是“实施阶段安排”的唯一事实源。

它只负责定义：

1. 阶段顺序
2. 每阶段目标
3. 每阶段依赖
4. 每阶段交付物
5. 每阶段验收条件

## 不负责内容

这份文档不负责定义：

- 具体表结构
- 具体接口字段
- 测试案例明细

## 首期阶段安排

### 第 0 阶段：基础约束

- 阶段目标：
  - 冻结首期业务边界、状态机、权限、菜单、字段字典和脱敏规则
  - 固定 `jixiao -> xuedao` 页面结构映射
  - 明确后端与前端目录落点
- 前置依赖：
  - `01`、`02`、`03`、`04`、`05`、`06`、`07`、`10`、`12`、`18` 文档完成首版冻结
- 主要交付物：
  - 绩效项目文档全集
  - 模块目录规划
  - 菜单、权限、路由、状态和枚举基线
- 验收条件：
  - 不再存在 `P0` 级需求歧义
  - 每个模块都有明确页面入口和技术落点
  - 没有超出 `cool-admin-vue + cool-admin-midway` 的实现方案
- 文件落点计划：
  - 后端：`cool-admin-midway/src/modules/performance/`
  - 前端：`cool-admin-vue/src/modules/performance/`

### 第 1 阶段：核心闭环

- 范围：
  - 我的考核
  - 已发起考核
  - 待我审批
  - 目标地图
  - 驾驶舱基础页与首批聚合指标
- 依赖：
  - 第 0 阶段完成
  - 评估单、目标、驾驶舱聚合接口设计冻结
- 后端交付：
  - `entity/assessment.ts`
  - `entity/assessment-score.ts`
  - `entity/goal.ts`
  - `entity/goal-progress.ts`
  - `service/assessment.ts`
  - `service/goal.ts`
  - `service/dashboard.ts`
  - `controller/admin/assessment.ts`
  - `controller/admin/goal.ts`
  - `controller/admin/dashboard.ts`
- 前端交付：
  - `views/dashboard/index.vue`
  - `views/my-assessment/index.vue`
  - `views/initiated/index.vue`
  - `views/pending/index.vue`
  - `views/goals/index.vue`
  - `components/assessment-form.vue`
  - `components/assessment-detail.vue`
  - `components/approval-drawer.vue`
  - `components/goal-progress-drawer.vue`
- 结构要求：
  - 评估模块按三张列表拆页
  - 详情优先抽屉化，不新增独立第二套路由
  - 目标页面按 `GoalMap` 结构承载列表、进度和统计区
  - 驾驶舱从本阶段开始固定为 6 指标返回结构
  - 与评估、目标直接相关的指标在本阶段先落真实聚合
  - `五环节进度` 中依赖指标库和薪资模块的统计口径，在对应模块上线后补齐最终真实数据
- 验收条件：
  - 员工可完成自评提交
  - 部门经理可在待我审批完成通过/驳回
  - HR 可查看全局数据但不能审批
  - 驾驶舱页面、筛选和聚合接口结构可用
  - 驾驶舱已完成与本阶段模块对应指标的真实聚合校验
  - 目标可新增、编辑、更新进度
- 最小验证：
  - 后端相关接口自测
  - 前端 `npm run type-check`
  - 前端 `npm run build`
  - 核心流程手测：创建评估 -> 员工提交 -> 经理审批 -> 目标进度更新 -> 驾驶舱聚合刷新

### 第 2 阶段：模型增强

- 范围：
  - 指标库
  - 360 环评
- 依赖：
  - 第 1 阶段评估单和目标主流程稳定
- 后端交付：
  - `entity/indicator.ts`
  - `entity/feedback-task.ts`
  - `entity/feedback-record.ts`
  - `service/indicator.ts`
  - `service/feedback.ts`
  - `controller/admin/indicator.ts`
  - `controller/admin/feedback.ts`
- 前端交付：
  - `views/indicator-library/index.vue`
  - `views/feedback/index.vue`
  - `components/feedback-task-form.vue`
  - `components/feedback-submit-drawer.vue`
  - `components/feedback-summary-drawer.vue`
- 结构要求：
  - 指标库按标准 CRUD 页面实现
  - 360 环评只保留一个菜单入口，内部覆盖任务列表、填写和汇总
- 验收条件：
  - HR/经理可创建环评任务并显式选择评价人
  - 评价人同任务只能提交一次
  - 员工本人只能看汇总，不能看他人单条反馈
  - 指标库可被评估单和目标地图稳定引用
- 最小验证：
  - 环评任务创建、提交、汇总、权限裁剪
  - 指标库 CRUD 与引用校验
  - 前端 `type-check` / `build`

### 第 3 阶段：状态类扩展

- 范围：
  - PIP
  - 晋升管理
- 依赖：
  - 第 1 阶段评估详情组件已稳定，可承接来源 `assessmentId`
- 后端交付：
  - `entity/pip.ts`
  - `entity/pip-record.ts`
  - `entity/promotion.ts`
  - `entity/promotion-record.ts`
  - `service/pip.ts`
  - `service/promotion.ts`
  - `controller/admin/pip.ts`
  - `controller/admin/promotion.ts`
- 前端交付：
  - `views/pip/index.vue`
  - `views/promotion/index.vue`
  - `components/pip-track-drawer.vue`
  - `components/promotion-review-drawer.vue`
- 结构要求：
  - 两者都按手工创建优先实现
  - 支持从评估详情带入 `assessmentId`
  - 支持独立创建，但必须填写原因
  - 不实现自动建议引擎
- 验收条件：
  - PIP 可创建、跟进、完成/关闭
  - 晋升可创建、提交评审、通过/驳回
  - 评估详情到 PIP/晋升的带参跳转有效
- 最小验证：
  - 来源评估单和独立创建两条路径
  - 合法/非法状态流转校验
  - 前端 `type-check` / `build`

### 第 4 阶段：薪资真实功能

- 范围：
  - 薪资列表
  - 薪资详情
  - 薪资调整记录
- 依赖：
  - 权限、脱敏、导出边界已冻结
- 后端交付：
  - `entity/salary.ts`
  - `entity/salary-change.ts`
  - `service/salary.ts`
  - `controller/admin/salary.ts`
- 前端交付：
  - `views/salary/index.vue`
  - `components/salary-change-drawer.vue`
- 结构要求：
  - 只有 `HR` 有入口和金额维护权限
  - 已确认薪资禁止直接改金额
  - 变更只能通过调整记录写入
  - 不做复杂薪资引擎
  - 不开放导出
- 验收条件：
  - `HR` 可录入 `baseSalary`、`performanceBonus`、`adjustAmount`、`finalAmount`
  - 经理和员工无后台薪资入口
  - 已确认记录只能通过调整记录变更
  - 列表、详情、导出和日志都不泄露无权限金额字段
- 最小验证：
  - 权限回归
  - 脱敏回归
  - 调整记录与只读校验
  - 前端 `type-check` / `build`

### 第 5 阶段：联调与交付

- 范围：
  - API 联调
  - 菜单和权限导入
  - 文档回写
  - 交付物整理
- 主要交付物：
  - 后端模块代码
  - 前端模块代码
  - 菜单与权限初始化数据
  - 测试记录
  - 交付说明与回滚说明
- 验收条件：
  - 文档、代码、菜单、权限、接口、测试结果一致
  - 驾驶舱 6 个正式指标全部切换到最终真实统计口径
  - 首期不开放能力保持关闭状态
  - 无 `P0` / `P1` 遗留阻塞项
- 最小验证：
  - 前端 `npm run type-check`
  - 前端 `npm run build`
  - 后端按变更范围执行 `npm run test` 或最小接口自测
  - 菜单、权限、导出、脱敏、状态流转回归

## 阶段依赖总规则

1. 未完成前一阶段验收，不进入下一阶段编码。
2. 每阶段优先交付“可闭环模块”，不并行摊大饼式铺开全部页面。
3. 每阶段结束都必须回写 `09-测试清单.md`、`14-交付说明.md`、`15-回滚说明.md` 中受影响部分。
