# 需求澄清清单

## 文档定位

这份文档用于在正式进入第一阶段代码实现前，集中收敛当前仍可能导致需求漂移的高风险问题。

它不是长期唯一事实源，而是“需求确认入口”。每一项确认完成后，必须回写到对应的唯一事实源文档中，不能只停留在这份清单里。

## 使用规则

1. 这里列出的事项，优先处理 `P0` 和 `P1` 风险项。
2. 每一项都要明确：
   - 当前歧义是什么
   - 推荐确认方向是什么
   - 影响哪些文档
   - 如果不确认，会出现什么漂移
3. 确认后，必须同步回写到对应文档，再把本项标记为“已确认”。

## 优先级说明

- `P0`：不确认就不能进入编码
- `P1`：不确认会导致高概率返工，建议在第一阶段前确认
- `P2`：可以后续补，但需要记录责任和时点

## 一、首批必须确认项

### 1. 评估单完整流程

- 优先级：`P0`
- 当前歧义：
  - 评估单到底由谁发起
  - 谁负责填写
  - 谁负责审批
  - 是否固定为单级审批
- 当前推荐方向：
  - 首期固定为：
    - HR 或部门经理发起评估单
    - 员工填写自评
    - 部门经理审批
    - HR 可查看与归档
  - 首期不做多级审批和可配置审批流
- 待确认问题：
  1. 员工能否自己发起评估单
  2. 是否允许“经理代员工填写”
  3. 审批是否只有一个节点
  4. HR 在首期里是审批人还是归档查看人
- 影响文档：
  - `04-API设计.md`
  - `05-状态机与流转规则.md`
  - `06-权限矩阵.md`
  - `09-测试清单.md`
- 不确认的后果：
  - 三张列表“我的考核 / 已发起考核 / 待我审批”会定义冲突
  - 审批接口和按钮权限容易返工
- 确认结论：
  - 首期评估单流程固定为：
    1. HR 或部门经理发起评估单
    2. 员工填写自评并提交
    3. 部门经理审批通过或驳回
    4. HR 负责查看、归档和全局管理，不参与首期审批
- 状态：已确认

### 2. 绩效评分公式与等级边界

- 优先级：`P0`
- 当前歧义：
  - `totalScore` 的计算公式未固定
  - 指标权重如何参与计算未固定
  - `grade` 的分段规则未固定
- 当前推荐方向：
  - 首期统一为加权总分：
    - `totalScore = sum(score * weight) / sum(weight)`
  - 等级先固定四档或五档，不做组织自定义规则
- 待确认问题：
  1. 首期等级是 `A/B/C/D` 还是 `S/A/B/C/D`
  2. 每档分数区间是多少
  3. 目标完成率是否单独作为评分项参与总分
  4. 是否允许手动覆盖等级
- 影响文档：
  - `03-数据库设计.md`
  - `04-API设计.md`
  - `07-字段字典与枚举.md`
  - `11-指标与统计口径.md`
- 不确认的后果：
  - 驾驶舱均分、等级分布、详情页总分会全部漂移
- 已确认部分：
  1. 首期绩效等级固定为 `S / A / B / C`
  2. 分数区间固定为：
     - `S`: `>= 90`
     - `A`: `>= 80 且 < 90`
     - `B`: `>= 70 且 < 80`
     - `C`: `< 70`
- 确认结论：
  1. `totalScore = sum(score * weight) / sum(weight)`
  2. 目标完成率作为一个普通评分项参与加权公式，不单独额外计算一次
  3. 首期不允许手工覆盖等级，等级全部由 `totalScore` 自动计算得到
- 状态：已确认

### 3. 三张评估列表的查询口径

- 优先级：`P0`
- 当前歧义：
  - 同一条评估单是否可同时出现在多个列表
  - 每张列表默认看哪些状态、哪些角色视角
- 当前推荐方向：
  - `我的考核`：只看当前登录人的被考核记录
  - `已发起考核`：只看当前登录人发起或负责维护的记录
  - `待我审批`：只看当前登录人拥有审批权限且状态为 `submitted` 的记录
- 待确认问题：
  1. 经理兼任发起人和审批人时，是否允许在两个列表都出现
  2. HR 是否能看到全部三个列表
  3. 默认排序按更新时间还是待办时间
- 影响文档：
  - `04-API设计.md`
  - `06-权限矩阵.md`
  - `10-路由与菜单映射.md`
  - `09-测试清单.md`
- 不确认的后果：
  - 前端页面功能相互覆盖，后端接口 mode 参数容易反复改
- 确认结论：
  1. `我的考核`
     - 只看当前登录员工作为 `employeeId` 的记录
     - 默认按 `updateTime desc` 排序
  2. `已发起考核`
     - 只看当前登录人作为发起人或负责人创建的记录
     - 默认按 `createTime desc` 排序
  3. `待我审批`
     - 只看当前登录部门经理可审批，且 `status = submitted` 的记录
     - 默认按 `submitTime asc` 排序，最早提交的优先处理
  4. 同一条评估单允许同时出现在两个列表里，但前提是视角不同
  5. `HR` 可以查看全部三张列表，但首期在“待我审批”里没有审批按钮，只做查看
- 状态：已确认

### 4. 360 环评规则

- 优先级：`P1`
- 当前歧义：
  - 评价人从哪里来
  - 是否匿名
  - 汇总是否平均分
  - 截止后是否可补交
- 当前推荐方向：
  - 首期采用显式选择评价人
  - 不做匿名
  - 汇总先按简单平均分
  - 截止后默认不可补交
- 待确认问题：
  1. 评价关系类型有哪些
  2. 是否允许一个人提交多次
  3. 汇总页员工本人是否可见
- 影响文档：
  - `03-数据库设计.md`
  - `04-API设计.md`
  - `05-状态机与流转规则.md`
  - `07-字段字典与枚举.md`
  - `12-数据权限与脱敏规则.md`
- 不确认的后果：
  - 环评任务表和反馈表字段会返工
- 确认结论：
  1. 评价人由 `HR` 或 `部门经理` 在发起任务时显式选择
  2. 首期不匿名，评价记录可追溯到具体评价人
  3. 每个评价人对同一任务只能提交一次
  4. 汇总分按简单平均分计算
  5. 截止后默认不可补交
  6. 员工本人不能看单条他人反馈内容，只能看汇总结果
  7. `HR` 和对应 `部门经理` 可以看汇总，也可以看单条反馈内容
- 状态：已确认

### 5. PIP 与晋升的触发条件

- 优先级：`P1`
- 当前歧义：
  - 是人工创建，还是由绩效结果自动建议
  - 来源评估单是否必填
- 当前推荐方向：
  - 首期统一手工创建
  - 但如果从评估详情发起，则默认带入来源评估单
- 待确认问题：
  1. 是否允许脱离评估单独立创建
  2. PIP 是否仅限低绩效等级触发
  3. 晋升是否仅限高绩效等级触发
- 影响文档：
  - `03-数据库设计.md`
  - `04-API设计.md`
  - `05-状态机与流转规则.md`
  - `08-实施计划.md`
- 不确认的后果：
  - 详情跳转入口、来源字段、查询口径不稳定
- 确认结论：
  1. 首期 `PIP` 和 `晋升` 都允许手工创建
  2. 从评估详情页发起时，默认自动带入 `assessmentId`
  3. 也允许脱离评估单独立创建，但首期必须手动填写原因
  4. `PIP` 首期不强制绑定低绩效等级才能创建
  5. `晋升` 首期不强制绑定高绩效等级才能创建
  6. 首期不做“系统自动建议创建 PIP / 晋升”的规则引擎
  7. 后续如果要做自动建议，只作为二期增强，不影响首期表结构和接口
- 状态：已确认

### 6. 薪资模块首期深度

- 优先级：`P0`
- 当前歧义：
  - 首期薪资是“真实金额维护”，还是“只展示结果”，还是“只保留菜单和占位”
- 当前推荐方向：
  - 如果要纳入首期交付，建议最小闭环为：
    - HR 可维护绩效关联薪资结果
    - 经理和员工在后台不可见
    - 支持调整记录，但不做复杂薪资计算引擎
- 待确认问题：
  1. 是否录入真实金额
  2. 是否允许编辑已确认薪资
  3. 是否需要导出
- 影响文档：
  - `03-数据库设计.md`
  - `04-API设计.md`
  - `06-权限矩阵.md`
  - `12-数据权限与脱敏规则.md`
  - `09-测试清单.md`
- 不确认的后果：
  - 敏感数据范围和页面深度反复改动
- 确认结论：
  1. 首期薪资模块纳入真实功能，不是占位页
  2. `HR` 可以录入和维护真实金额
  3. 首期允许维护：
     - `baseSalary`
     - `performanceBonus`
     - `adjustAmount`
     - `finalAmount`
     - 调整记录
  4. `部门经理` 和 `员工` 在后台都没有薪资管理入口
  5. 首期不开放薪资导出
  6. 已确认薪资不允许直接修改金额，如需变更只能新增调整记录
  7. 首期不做复杂薪资计算引擎，只做结果录入和调整记录管理
- 状态：已确认

## 二、建议在第一阶段前确认项

### 7. 部门经理数据范围

- 优先级：`P1`
- 当前歧义：
  - 所辖部门是否包含下级部门
  - 是否支持跨部门兼任管理
- 当前推荐方向：
  - 首期先按“直接管理部门 + 下级部门”处理
  - 兼任关系如无现成主数据支持，则首期不做
- 影响文档：
  - `06-权限矩阵.md`
  - `12-数据权限与脱敏规则.md`
- 确认结论：
  1. 首期部门经理默认可查看自己直接管理部门及这些部门的所有下级部门
  2. 数据范围判断只基于组织架构里的部门树
  3. 首期不支持跨部门兼任经理的额外授权模型
  4. 如需兼任或临时代理，作为二期增强，不影响首期接口和表结构
  5. 经理对可见范围、可审批范围、可跟进范围统一按部门树口径处理
- 状态：已确认

### 8. 驾驶舱首期指标范围

- 优先级：`P1`
- 当前歧义：
  - 招聘达成率、内训通关率、会议效能是否必须首期真实上线
- 当前推荐方向：
  - 首期不纳入真实实现，只保留扩展位
- 影响文档：
  - `11-指标与统计口径.md`
  - `08-实施计划.md`
- 当前补充说明：
  - 现有技术架构和首期范围都只覆盖绩效域自身
  - 如果把跨招聘、培训、会议指标硬塞进首期，会直接造成文档与实现脱节
- 确认结论：
  1. 首期驾驶舱只展示绩效域内可直接产出的 6 个指标：
     - 全员绩效均分
     - 待审批数量
     - 目标完成率
     - 五环节进度
     - 部门绩效分布
     - 绩效等级分布
  2. 招聘达成率、内训通关率、会议效能看板不进入首期真实交付
  3. 上述跨模块指标首期前端不展示
  4. 相关能力统一延后到二期实现
- 状态：已确认

### 9. 菜单与权限初始化方式

- 优先级：`P1`
- 当前歧义：
  - 是否直接按现有 `base/menu.json` 导入方式处理
  - 是否需要单独脚本
- 当前推荐方向：
  - 首期沿用现有菜单导入方式，不新造权限初始化机制
- 影响文档：
  - `10-路由与菜单映射.md`
  - `14-交付说明.md`
- 确认结论：
  1. 首期沿用现有 Cool 基座的菜单导入方式，不新造权限初始化机制
  2. 绩效模块菜单最终按 `base/menu.json` 同结构数据导入
  3. 按钮权限继续通过菜单树里的 `type = 2` 节点维护
  4. 前端继续依赖现有动态菜单装配逻辑，不新增本地硬编码菜单方案
  5. 首期不单独做“绩效权限初始化脚本平台”，只在现有导入链路里补绩效菜单和权限数据
- 状态：已确认

### 10. 导出范围

- 优先级：`P1`
- 当前歧义：
  - 哪些列表允许导出
  - 导出给哪些角色
  - 导出是否需要脱敏
- 当前推荐方向：
  - 首期不开放薪资导出
  - 绩效导出仅限 HR，且按权限裁剪字段
- 影响文档：
  - `04-API设计.md`
  - `12-数据权限与脱敏规则.md`
  - `09-测试清单.md`
- 确认结论：
  1. 首期不开放 `薪资导出`
  2. `绩效评估导出` 仅 `HR` 可用
  3. `目标地图导出` 仅 `HR` 和 `部门经理` 可用
  4. `360 环评`、`PIP`、`晋升` 首期不开放导出
  5. 所有允许导出的内容都必须按角色裁剪字段
  6. 导出内容里不包含：
     - 薪资金额
     - 单条他人环评反馈全文
     - 非本权限范围的审批意见
  7. 导出能力放到二期再增强，不做通用导出平台
- 状态：已确认

## 四、建议确认方式

建议后续按下面顺序逐项确认，不要一次性并行确认全部问题：

1. 先确认评估单流程
2. 再确认评分与等级
3. 再确认三张评估列表口径
4. 再确认薪资模块首期深度
5. 最后确认环评、PIP、晋升和导出规则

这样可以避免后项依赖前项而重复改文档。
