# 模块开发任务拆分

## 文档职责

这份文档是“逐模块开发任务卡边界”的唯一事实源。

它只负责定义：

1. 推荐开发顺序
2. 每个模块的代码落点
3. 每个模块的前后端任务边界
4. 每个模块的最小完成定义
5. 每个模块进入下一模块前必须具备的验证证据

## 不负责内容

这份文档不负责定义：

- 字段名和表结构细节
- 接口字段全文
- 权限规则全文
- 页面视觉样式

这些内容分别以 `03`、`04`、`06`、`18` 文档为准。

## 总体约束

1. 首期全部模块都纳入同一套文档和实现规划，但编码时必须按模块逐步落地。
2. 技术实现只能使用 `cool-admin-midway + cool-admin-vue` 现有框架能力，不新增第二套前后端工程。
3. 菜单、按钮权限和动态路由必须复用现有 Cool 导入链路，不新增本地硬编码菜单机制。
4. 每完成一个模块，至少要补齐一次最小验证，再进入下一个模块。
5. 若前一模块仍存在 `P0 / P1` 阻塞项，不进入后一模块编码。
6. 涉及行为变化的模块任务，必须同步产出自动化验证证据，最低门槛以 [24-自动化测试策略与脚本规划.md](/Users/shaotongli/Documents/xuedao/performance-management-system/docs/24-自动化测试策略与脚本规划.md) 为准。

## 模块 0：模块骨架与基础配置

### 目标

1. 建立 `performance` 后端模块目录。
2. 建立 `performance` 前端模块目录。
3. 准备菜单导入数据、权限键命名和页面空壳。
4. 固定公共组件、服务命名和模块入口。

### 代码落点

- 后端：`cool-admin-midway/src/modules/performance/`
- 前端：`cool-admin-vue/src/modules/performance/`
- 菜单数据：复用 `cool-admin-midway/src/modules/base/menu.json` 同结构

### 最小交付

- 后端模块目录和基础文件可被项目识别
- 前端 `modules/performance/` 目录可被动态路由装配
- 菜单、`router`、`viewPath`、`perms` 与 [10-路由与菜单映射.md](/Users/shaotongli/Documents/xuedao/performance-management-system/docs/10-路由与菜单映射.md) 一致

### 进入下一模块前必须完成

- 菜单树结构无漂移
- 权限键命名无漂移
- 页面空壳可正常访问或按权限隐藏

## 模块 1：评估单

### 目标

1. 打通 `HR/经理发起 -> 员工自评提交 -> 部门经理审批`。
2. 落地三张列表：我的考核、已发起考核、待我审批。
3. 落地评估详情、评分明细和审批动作。

### 后端任务

- 实体：`assessment`、`assessment-score`
- 服务：列表查询、详情、自评提交、审批通过/驳回、总分与等级计算
- 控制器：`page / info / add / update / delete / submit / approve / reject / export`

### 前端任务

- 页面：`my-assessment`、`initiated`、`pending`
- 组件：`assessment-form`、`assessment-detail`、`approval-drawer`、`score-item-table`
- 交互：三视图共用详情和表单组件，不复制三套逻辑

### 最小完成定义

- 员工能提交本人草稿评估单
- 部门经理能在待我审批中通过或驳回
- `HR` 能查看全局，但没有审批按钮
- `totalScore` 与 `grade` 完全由服务端生成

### 最小验证

- 正常：新增评估、员工提交、经理审批通过
- 异常：非法状态提交、非法状态审批、无权限审批
- 边界：`90 / 80 / 70 / 69.99` 等级边界

## 模块 2：目标地图

### 目标

1. 落地目标列表、详情和进度更新记录。
2. 支持 `HR/经理` 管理目标，员工更新本人进度。

### 后端任务

- 实体：`goal`、`goal-progress`
- 服务：分页、详情、新增、修改、删除、进度更新、导出
- 规则：`progressRate` 服务端计算，不接受前端覆盖

### 前端任务

- 页面：`goals/index.vue`
- 组件：`goal-progress-drawer.vue`
- 结构：以 `GoalMap` 为主参考，承载筛选、列表、统计和进度抽屉

### 最小完成定义

- `HR/经理` 可新增、编辑、删除目标
- 员工可更新本人目标进度
- 目标导出只对 `HR/经理` 开放

### 最小验证

- 正常：新增目标、更新进度、完成目标
- 异常：`completed` 后更新被拒绝、非本人更新被拒绝
- 边界：`startDate <= endDate`、`targetValue > 0`

## 模块 3：驾驶舱基础聚合

### 目标

1. 先打通驾驶舱页面、筛选器和汇总接口。
2. 先实现与评估单、目标直接相关的真实指标。
3. 固定 6 指标响应结构，为后续模块补齐最终真实口径预留一致返回。

### 本模块先落地的真实指标

- `averageScore`
- `pendingApprovalCount`
- `goalCompletionRate`
- `departmentDistribution`
- `gradeDistribution`

### 暂按最终结构返回的指标

- `stageProgress`

说明：

- `stageProgress` 的最终真实统计依赖指标库和薪资模块。
- 在模块 4 和模块 8 完成前，本字段必须保持最终对象结构，但统计值允许按当前已落地模块返回阶段性结果。

### 最小验证

- 页面切换周期、部门筛选可正常请求
- 评估审批或目标进度更新后，驾驶舱数据刷新正确
- 返回结构与 [04-API设计.md](/Users/shaotongli/Documents/xuedao/performance-management-system/docs/04-API设计.md) 完全一致

1. 落地指标标准 CRUD。
2. 让评估单评分项有稳定配置来源。
3. 为驾驶舱 `stageProgress.indicatorConfigured` 提供真实数据基础。

### 后端任务

### 前端任务

- 页面：`indicator-library/index.vue`

### 最小完成定义

## 模块 5：360 环评

### 目标

1. 落地任务、填写、汇总三段式页面。
2. 打通显式选人、单人单次提交、简单平均分汇总。

### 后端任务

- 实体：`feedback-task`、`feedback-record`
- 服务：任务创建、详情、提交、汇总

### 前端任务

- 页面：`feedback/index.vue`
- 组件：`feedback-task-form`、`feedback-submit-drawer`、`feedback-summary-drawer`

### 最小完成定义

- `HR/经理` 可创建环评任务并显式选择评价人
- 评价人同任务只能提交一次
- 员工本人只能看汇总，不能看他人单条反馈

### 最小验证

- 正常：任务创建、反馈提交、汇总查看
- 异常：重复提交、截止后补交、无权限查看单条记录

## 模块 6：PIP

### 后端任务

### 前端任务

## 模块 8：薪资管理

### 目标

1. 落地真实金额维护和调整记录。
2. 保证金额只对 `HR` 可见。
3. 保证 `confirmed` 后只能通过调整记录变更。

### 后端任务

- 实体：`salary`、`salary-change`
- 服务：分页、详情、新增、修改、确认、归档、调整记录新增

### 前端任务

- 页面：`salary/index.vue`
- 组件：`salary-change-drawer.vue`

### 最小完成定义

- `HR` 可维护 `baseSalary / performanceBonus / adjustAmount / finalAmount`
- 部门经理和员工无后台薪资入口
- `confirmed` 状态不能直接改金额

### 最小验证

- 正常：新增薪资、确认、归档、调整记录新增
- 异常：非 `HR` 被拒绝、`confirmed` 直接修改被拒绝
- 敏感：接口、页面、日志不泄露金额字段

## 二期主题 11：采购与供应商管理

### 开发前提

1. [2026-04-18-二期主题11-采购与供应商管理-阶段0最终验收.md](/Users/shaotongli/Documents/xuedao/performance-management-system/tasks/2026-04-18-二期主题11-采购与供应商管理-阶段0最终验收.md) 已明确主题11阶段0冻结完成。
2. 当前实现必须继续保持“一个主题三窗口”推进，不得口头拆成“采购订单管理 / 供应商管理”两个独立主题。
3. 当前实现只能落在已冻结的三组资源内：
   - `purchaseOrder`
   - `supplier`
   - `purchaseReport`

### 目标

1. 落地采购增强主链：采购单、询价、审批、收货、报表与供应商。
2. 固定 `HR / 部门经理 / 员工` 三类角色在采购增强域的真实权限与数据边界。
3. 为后续是否扩到付款、对账与库存总账联动保留清晰非目标边界。

### 推荐任务拆分

1. 实现 A - 后端：
   - `purchaseOrder / supplier / purchaseReport` 的最小接口闭环
   - `submitInquiry / submitApproval / approve / reject / receive / close`
   - 采购订单部门树范围校验
   - 供应商敏感字段角色裁剪
   - 删除与状态约束
2. 实现 B - 前端：
   - 采购管理 / 询价管理 / 采购审批 / 订单管理 / 收货管理 / 采购报表
   - 供应商列表 / 详情 / 新增 / 编辑
   - 经理采购单可维护、供应商只读
   - 员工无页面入口
3. 实现 C - 联调与收口：
   - 菜单、按钮权限和角色绑定
   - 真实账号联调与主题11增强回归
   - 任务卡、测试记录和最终验收收口

### 代码落点

- 后端：`cool-admin-midway/src/modules/performance/`
- 前端：`cool-admin-vue/src/modules/performance/`
- 任务卡：`performance-management-system/tasks/`

### 最小完成定义

1. `HR` 可管理采购增强与供应商全量主链，并可查看采购报表。
2. 部门经理仅可在本人部门树范围内管理采购增强主链，对供应商只保留只读查询。
3. 员工对采购增强和供应商均无页面、无接口、无数据返回。
4. 采购订单删除只允许 `draft`；`reject` 固定回 `draft`；`close` 必须带原因；供应商删除只允许 `inactive` 且无关联有效采购订单。
5. 文档、代码、权限、状态、脱敏与测试记录一致，且无未关闭 `P0 / P1`。

### 最小验证

1. 后端最少补齐采购增强 / 供应商相关 `jest`，覆盖正常、异常、边界、权限和状态约束。
2. 前端必须执行 `corepack pnpm run type-check` 与 `corepack pnpm run build`。
3. 联调阶段至少验证：
   - `hr_admin`
   - `manager_rd`
   - `employee_platform`
4. 真实联调至少覆盖：
   - 采购订单与供应商分页 / 详情
   - 询价 / 审批 / 收货 / 报表动作
   - 经理供应商只读
   - 员工拒绝
   - 采购订单 `draft` 删除限制
   - 供应商 `inactive + 无关联有效采购订单` 删除限制

## 每个模块通用完成门槛

1. 受影响的唯一事实源文档已回写。
2. 自动化验证至少满足 [24-自动化测试策略与脚本规划.md](/Users/shaotongli/Documents/xuedao/performance-management-system/docs/24-自动化测试策略与脚本规划.md) 的首期基线。
3. 前端最小校验已执行，至少覆盖 `type-check` 或构建路径。
4. 后端最小接口或服务校验已执行。
5. 权限、状态和异常路径至少各验证一次。
6. 未完成项如果不是 `P0 / P1`，必须明确 owner 和补齐时点。
7. 若当前模块发生行为变化但缺少自动化验证证据，不得视为完成。
