# 新增候选主题与阶段 0 冻结入口

## 文档职责

这份文档用于解决一个固定问题：

当前二期主题 `1-6` 已完成当前阶段收口后，如果还要继续推进，下一步该如何合法开启新的候选主题，而不是口头进入“主题 7”。

它只负责：

1. 定义新增候选主题的进入前提
2. 定义阶段 0 冻结最小材料清单
3. 定义推荐的并行窗口拆分方式
4. 定义“可进入开发前评审 / 不可进入冻结”的唯一输出格式

它不负责：

1. 定义新主题的业务范围正文
2. 定义新主题的 API、表结构、状态机、权限和脱敏细节
3. 直接授权进入设计、开发或联调
4. 替代 [30-二期范围冻结草案.md](/Users/shaotongli/Documents/xuedao/performance-management-system/docs/30-二期范围冻结草案.md) 成为候选范围主文档

## 当前适用前提

只有在以下条件同时成立时，才使用本入口：

1. [2026-04-18-二期主题1-6-总收口.md](/Users/shaotongli/Documents/xuedao/performance-management-system/tasks/2026-04-18-二期主题1-6-总收口.md) 仍是当前有效总结论
2. 当前仓库中不存在新的已冻结候选主题
3. 拟推进事项不是“继续实现主题 1-6 的已冻结内容”，而是“新增一个候选主题”

补充说明：

1. 当前仓库没有默认存在的“主题 7”。
2. 若没有新的候选主题名称和业务目标，本入口只能停留在提名准备，不能推进到冻结完成。

## 新增候选主题的准入判断

新增候选主题只有命中以下任一来源，才允许进入阶段 0 冻结：

1. 首期或二期现有文档中已经明确写成“当前不纳入、后续再做”的能力
2. 现有已关闭主题外新增的独立业务能力，且不与主题 `1-6` 的已冻结结论重复
3. 当前项目级治理项，需要单独成主题而不能继续归入现有主题 `1-6`

以下情况一律不得提名：

1. 仅有聊天口头想法，没有仓库内来源依据
2. 只是对主题 `1-6` 已冻结范围的实现补充，却想包装成新主题
3. 会直接推翻首期或二期已关闭的状态机、权限边界或 current latest 结论，但没有先完成变更评估

## 阶段 0 冻结最小输入

新增候选主题进入阶段 0 之前，至少要补齐下面 8 项输入：

1. 候选主题名称
2. 业务目标
3. 明确非目标
4. 首期 / 二期现有事实源中的来源依据
5. 影响范围：
   - 文档
   - 前端
   - 后端
   - 权限
   - 测试
   - 交付 / 回滚
6. 最小验收口径
7. 是否触碰主题 `1-6` 已关闭结论
8. 负责人和当前阶段

若以上任一项缺失，正式结论只能是：

- `新增候选主题暂不成立，唯一阻塞项为 ...`

## 阶段 0 冻结最小产物

一个新增候选主题至少要形成下面 5 类仓库产物，才允许宣称“已完成阶段 0 冻结”：

1. 在 [30-二期范围冻结草案.md](/Users/shaotongli/Documents/xuedao/performance-management-system/docs/30-二期范围冻结草案.md) 新增候选项或明确记录“不纳入”
2. 新建主题主卡：
   - 命名建议：`tasks/YYYY-MM-DD-二期主题X-<主题名>-阶段0冻结.md`
3. 按复杂度补分窗口卡：
   - 范围与非目标
   - API / 权限 / 数据边界
   - 测试矩阵与任务拆分
   - 最终验收
4. 回写 [13-版本与变更记录.md](/Users/shaotongli/Documents/xuedao/performance-management-system/docs/13-版本与变更记录.md)
5. 必要时回写：

## 推荐并行窗口

新增候选主题默认按下面 4 个窗口并行冻结，除非主题足够简单可以合并：

### 窗口 A：范围与非目标

负责：

1. 明确为什么要做
2. 明确为什么不做其他相邻能力
3. 明确是否会影响已关闭主题

允许改：

1. `30`
2. `16`
3. 主题主卡

### 窗口 B：契约与边界

负责：

1. API 方向
2. 权限边界
3. 数据可见范围
4. 状态 / 动作主链是否需要单独冻结

允许改：

1. `04`
2. `05`
3. `06`
4. `12`
5. 主题窗口卡

### 窗口 C：测试矩阵与拆分

负责：

1. 正常 / 异常 / 边界 / 权限 / 状态测试矩阵
2. 后续实现阶段拆分顺序
3. 串并行边界

允许改：

1. `09`
2. `24`
3. 主题窗口卡

### 窗口 D：最终总验收

负责：

1. 汇总 A / B / C 输出
2. 交叉核验是否还有 `P0 / P1`
3. 给出唯一正式结论

允许改：

1. 主题阶段 0 最终验收卡
2. `13`
