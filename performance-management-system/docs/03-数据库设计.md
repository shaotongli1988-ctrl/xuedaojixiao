# 数据库设计

## 文档职责

这份文档是“数据库结构”的唯一事实源。

它只负责定义：

1. 表名
2. 字段名
3. 字段类型
4. 默认值和是否可空
5. 主键、外键、唯一约束
6. 索引和查询关键约束

## 不负责内容

这份文档不负责定义：

- 接口路径
- 页面布局
- 按钮权限
- 测试执行方式

- 业务名称：绩效管理系统首期
- 所属模块：`performance`

- 关联需求：[01-绩效管理系统-合并升级版需求与页面文档.md](/Users/shaotongli/Documents/xuedao/performance-management-system/docs/01-绩效管理系统-合并升级版需求与页面文档.md)

## 设计目标

为绩效域核心业务提供稳定的数据结构，覆盖评估、目标、指标、360 环评、PIP、晋升和薪资。所有表均遵循现有基座的实体基类模式，公共字段统一继承 `id`、`createTime`、`updateTime`、`tenantId`。

## 总体命名规则

1. 表名前缀统一使用 `performance_`。
2. 所有业务主表都保留 `status` 字段。
3. 用户、部门等主数据只存 `id`，不重复冗余名称作为主来源。
4. 审批意见、备注、反馈内容使用 `text` 或长 `varchar`。

## 数据读写规则

- HR 管理员：
  - 可写：指标、评估、目标、环评任务、PIP、晋升、薪资
  - 可读：全部绩效域数据
- 部门经理：
  - 可写：本人负责范围内的评估反馈、审批意见、PIP 跟进、晋升评审
  - 可读：所属团队范围数据
- 员工：
  - 可写：自评、目标进度、环评反馈中本人负责部分
  - 可读：本人数据
- 系统维护字段：
  - `id`
  - `createTime`
  - `updateTime`
  - `tenantId`
  - 审批相关时间字段

补充规则：

- 薪资金额字段只允许 `HR` 写入。
- 已确认薪资记录不允许直接覆盖金额字段。
- 调整动作必须写入 `performance_salary_change`。

## 数据迁移或兼容说明

- 首期为新增模块，默认没有历史表兼容压力。
- 需要初始化的内容：
  - 指标库基础字典
  - 菜单和权限
- 暂不要求历史数据迁移，但必须预留导入脚本入口。

## 风险与注意事项

- 薪资表涉及敏感数据，查询必须和权限文档同步。
- `performance_assessment_score` 采用快照字段，避免指标配置变更后影响历史评估。
- 驾驶舱不单独建统计结果表，首期以实时聚合为主。
- `totalScore` 和 `grade` 由服务端统一计算和落库，首期不允许前端手工传入覆盖最终等级。
- 首期薪资不做计算引擎，不根据评估自动反推金额，只做结果录入和调整记录管理。
- 环评首期不匿名，`feedbackUserId` 为必存字段。
- 同一任务下同一评价人只允许一条正式反馈记录。
- `PIP` 和 `晋升` 首期不强制绑定特定绩效等级才允许创建。
- `PIP` 和 `晋升` 首期不做自动建议引擎。
