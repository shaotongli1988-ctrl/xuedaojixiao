# 数据库设计

## 文档职责

这份文档是“数据库结构”的唯一事实源。

它只负责定义：

1. 表名
2. 字段名
3. 字段类型
4. 默认值和是否可空
5. 主键、外键、唯一约束
6. 索引和查询关键约束

## 不负责内容

这份文档不负责定义：

- 接口路径
- 页面布局
- 按钮权限
- 测试执行方式

- 关联需求：[01-绩效管理系统-合并升级版需求与页面文档.md](/Users/shaotongli/Documents/xuedao/performance-management-system/docs/01-绩效管理系统-合并升级版需求与页面文档.md)

## 设计目标

为绩效域核心业务提供稳定的数据结构，覆盖评估、目标、指标、360 环评、PIP、晋升和薪资。所有表均遵循现有基座的实体基类模式，公共字段统一继承 `id`、`createTime`、`updateTime`、`tenantId`。

## 总体命名规则

1. 表名前缀统一使用 `performance_`。
2. 所有业务主表都保留 `status` 字段。
3. 用户、部门等主数据只存 `id`，不重复冗余名称作为主来源。
4. 审批意见、备注、反馈内容使用 `text` 或长 `varchar`。

## 基座表补充设计

### 基座用户表扩展：`base_sys_user`

- 用途：持久化登录用户当前选中的绩效 persona 偏好，供 `Web` 与 `cool-uni` 共用。
- 说明：该字段只保存“当前视角偏好”，不是权限、范围、工作台页面或角色能力的事实源；`availablePersonas / roleKind / scope / workbenchPages / surfaceAccess` 仍由后端按当前权限和部门范围实时推导。

| 字段名 | 类型 | 必填 | 默认值 | 索引 | 说明 |
| --- | --- | --- | --- | --- | --- |
| activePerformancePersonaKey | varchar(64) | 否 | 空 | 否 | 当前绩效视角偏好键，空表示回退默认 persona |

补充规则：

1. 当前阶段不新增 persona 关系表，也不把 persona 落成独立字典表。
2. 该字段允许存空；存空时，后端返回当前账号按权限推导出的 `defaultPersonaKey`。
3. 即使库里已存值，后端每次返回上下文前仍必须校验该值是否仍属于当前账号的 `availablePersonas`；失效值只能回退默认视角，不能继续生效。
4. 该字段只允许由专用接口 `/admin/base/comm/performanceAccessContext/activePersona` 写入；`personUpdate` 不承担该字段写入职责。

## 二期主题 5：自动审批流存储冻结基线（2026-04-17）

本节记录自动审批流进入开发前已冻结的最小存储模型。自动审批流只服务 `performance` 模块内的 `assessment / promotion` 两类对象，不抽离为跨模块共享工作流引擎。

### 当前冻结结论

- 当前唯一结论：`自动审批流存储模型已冻结，可进入开发`

### 存储总原则

1. 自动审批流不改写源业务对象既有主表字段结构；`assessment.status` 与 `promotion.status` 仍保持当前状态集合。
2. 自动审批实例、节点和审计留痕使用独立表保存，源业务对象只保存既有提交 / 评审状态，不新增第二套对外业务状态字段。
3. 每个 `objectType` 只允许 1 份启用中的配置；每个 `objectType + objectId` 同时只允许 1 个未终态自动审批实例。
4. 通知首期只保留业务语义和接口位，不存短信、钉钉等外部渠道专有配置。
