## 适配总原则

1. 复用的是“页面结构和交互分层”，不是直接复制 `jixiao` 的技术实现代码。
2. 当前页面仍落在 `cool-admin-vue/src/modules/performance/` 技术目录下，不新建第二套前端工程；该目录是历史技术挂载，不等于全部页面都属于绩效域。
3. `jixiao` 的 `/performance` 首页导航卡片只作为原型参考，不作为首期必须交付页面。
4. 当前运行态继续使用 Cool 动态菜单树承载页面；后续若改成“导航栏 + 侧栏”联动，一级分组必须先服从业务域归属冻结。
5. 首期详情优先采用“列表页 + 抽屉/弹窗详情”，跨模块跳转场景才考虑补隐藏详情页。
6. 同一业务对象的三种视角页面可以拆成独立菜单，但必须复用同一套详情和表单组件。

## 首期页面结构结论

### 1. 评估模块

首期不保留 `jixiao` 的单一“绩效考核”总列表页作为菜单，而是拆成三个业务视角页面：

1. 我的考核
2. 已发起考核
3. 待我审批

三页复用以下组件：

- `assessment-form.vue`
- `assessment-detail.vue`
- `approval-drawer.vue`
- `score-item-table.vue`

### 2. 目标模块

目标模块以 [GoalMap.vue](/Users/shaotongli/Documents/jixiao/src/views/performance/GoalMap.vue) 为主参考，而不是以 [Goals.vue](/Users/shaotongli/Documents/jixiao/src/views/performance/Goals.vue) 为主。原因如下：

1. `GoalMap.vue` 已覆盖筛选、指标联动、进度展示和统计区，更接近首期目标闭环。
2. `Goals.vue` 更适合做简化列表页，可吸收其轻量列表表达，但不能替代首期主页面结构。

### 3. 360 环评模块

首期只保留一个菜单入口 `/performance/feedback`，但页面内部按三个区域组织：

1. 任务列表区
2. 发起/填写弹窗区
3. 汇总结果区

对应参考来源：

- 任务列表区参考 [Feedback.vue](/Users/shaotongli/Documents/jixiao/src/views/performance/Feedback.vue)
- 填写和汇总区参考 [Feedback360.vue](/Users/shaotongli/Documents/jixiao/src/views/performance/Feedback360.vue)

### 4. PIP 与行动计划

首期不单独复制 [ActionPlan.vue](/Users/shaotongli/Documents/jixiao/src/views/performance/ActionPlan.vue) 为独立菜单。行动计划拆解如下：

1. 与绩效改进直接相关的任务跟进，纳入 `PIP` 的跟进记录能力。
2. 与目标执行直接相关的任务推进，纳入“目标进度更新记录”。
3. 如果后续需要独立任务中心，作为二期增强，再评估是否独立菜单。

## 二期 `cool-uni` 首批移动端冻结映射（2026-04-17）

本节只负责冻结二期 `cool-uni` 首批移动端范围，用于进入设计评审；不授权直接写移动端页面代码，也不替代 `04 / 05 / 06 / 10 / 12` 的业务主口径。

### 冻结原则

1. `cool-uni` 是首期已闭环能力的渠道下沉，不是新业务模块。
2. 移动端不等于 PC 全量镜像，只允许进入已冻结的首批页面。
3. 登录态、权限和业务接口默认复用现有 `cool-admin-midway` 能力，不新增第二套账号体系。
4. 当移动端与 PC 展示层不一致时，以 PC 侧业务契约、状态机、权限和脱敏文档为主口径。

### 首批目标角色

1. `员工`

补充说明：

1. `员工` 是首批主角色。
2. `部门经理` 只开放待审批和本部门范围内的轻量查看 / 操作。

### 必须新增接口的判定标准

仅在同时满足以下条件之一时，才允许提出移动端新增接口需求，并且必须先回写 `04-API设计.md`、`06-权限矩阵.md`、`09-测试清单.md`、`12-数据权限与脱敏规则.md`：

1. 现有接口无法在不放宽权限或数据范围的前提下返回移动端所需数据。
2. 现有接口无法覆盖移动端必须存在的原子动作，且前端拼装会破坏状态机或权限校验。
3. 现有接口返回包含 PC 专属敏感字段，无法通过现有角色裁剪安全复用。
4. 现有接口只能支撑后台管理视角，无法满足员工“仅本人”或经理“仅部门树范围”的移动端主路径。

### 登录态复用原则

1. 沿用现有验证码登录链路，不新增手机号快捷登录、短信登录或移动端独立账号体系。
2. 沿用现有 token、权限菜单和后端鉴权中间件，不新增移动端专用鉴权协议。
3. 页面可见性以 `permmenu` 和既有权限键为准，移动端只做显示裁剪，不新增口头权限。

### 权限和数据范围

1. 员工：
   - 仅可见本人评估、本人目标、本人相关环评任务和汇总
   - 不可见待我审批、驾驶舱、薪资、PIP、晋升、指标库
2. 部门经理：
   - 可见本部门及下级部门范围内的待我审批、目标、环评
   - 不可在移动端获得 `HR` 全局管理能力
3. 若移动端与 PC 端口径不一致，以 [04-API设计.md](/Users/shaotongli/Documents/xuedao/performance-management-system/docs/04-API设计.md)、[05-状态机与流转规则.md](/Users/shaotongli/Documents/xuedao/performance-management-system/docs/05-状态机与流转规则.md)、[06-权限矩阵.md](/Users/shaotongli/Documents/xuedao/performance-management-system/docs/06-权限矩阵.md)、[10-路由与菜单映射.md](/Users/shaotongli/Documents/xuedao/performance-management-system/docs/10-路由与菜单映射.md)、[12-数据权限与脱敏规则.md](/Users/shaotongli/Documents/xuedao/performance-management-system/docs/12-数据权限与脱敏规则.md) 为唯一主口径。

### 二期阶段 1 信息架构与路由设计结论

本节只固定 `cool-uni` 首批信息架构和页面拆分设计，不代表已进入实现。

#### 推荐路由策略

1. 采用“单入口工作台 + 业务页面栈路由”。
2. 保留 `pages/user/login` 作为登录入口。
3. 保留 `pages/index/home` 作为工作台首页。
4. 首批业务页统一进入 `pages/performance/*` 路由组，详情和动作页作为隐藏二级路由。
5. 不引入“工作台 / 我的”双 Tab，也不引入四业务并列 Tab。

#### 固定路由树

```text

/pages/index/home

/pages/performance/assessment/list
/pages/performance/assessment/detail
/pages/performance/assessment/edit

/pages/performance/approval/list

/pages/performance/goal/list
/pages/performance/goal/detail
/pages/performance/goal/edit
/pages/performance/goal/progress

/pages/performance/feedback/list
/pages/performance/feedback/detail
/pages/performance/feedback/submit
/pages/performance/feedback/summary
```

#### 页面关系规则

1. 登录成功后固定 `reLaunch` 到工作台。
2. 工作台是首批唯一业务首页，其余页面都从工作台进入。
3. `assessment/detail` 由“我的考核”和“待我审批”共享，不重复设计两套详情页。
4. `goal/edit`、`goal/progress`、`feedback/submit`、`feedback/summary` 都属于隐藏动作页，不作为工作台一级入口。
5. 页面直达缺少来源上下文时，统一回退到工作台。

#### 页面状态规则

1. 工作台按权限裁剪入口卡片，不展示无权限入口。
2. 业务列表页必须同时具备空状态、错误态和无权限态。
3. 详情页记录不存在时，展示“记录不存在或已失效”，不能停留空白页。
4. 提交 / 编辑 / 进度更新动作失败时，保留当前表单上下文并允许重试。
