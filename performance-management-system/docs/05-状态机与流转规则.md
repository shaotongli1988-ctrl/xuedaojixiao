# 状态机与流转规则

## 文档职责

这份文档是“业务状态流转”的唯一事实源。

它只负责定义：

1. 每个业务对象的状态集合
2. 合法流转路径
3. 非法流转处理方式
4. 页面按钮和后端动作的对应关系

## 不负责内容

这份文档不负责定义：

- 表结构字段类型
- 接口字段格式
- 菜单可见性

## 总体规则

1. 状态流转以服务端为唯一事实源，前端只负责显隐和禁用。
2. 同一动作只能由一个明确状态触发，不能跨多个状态模糊执行。
3. 非法流转必须返回明确错误，不允许静默成功。

## 1. 评估单状态机

### 流程角色基线

- 发起人：`HR` 或 `部门经理`
- 填写人：`员工`
- 审批人：`部门经理`
- 归档和全局查看：`HR`

### 状态集合

| 状态 | 含义 |
| --- | --- |
| `draft` | 草稿，可编辑 |
| `submitted` | 已提交，等待审批 |
| `approved` | 已审批通过 |
| `rejected` | 已驳回，允许回到编辑态 |

### 合法流转

| 当前状态 | 动作 | 下一状态 | 执行人 |
| --- | --- | --- | --- |
| `draft` | 新建 / 保存 | `draft` | HR / 部门经理 / 员工自评场景 |
| `draft` | 提交 | `submitted` | 员工 |
| `submitted` | 审批通过 | `approved` | 部门经理 |
| `submitted` | 审批驳回 | `rejected` | 部门经理 |
| `rejected` | 重新编辑保存 | `draft` | 员工 |

### 非法流转

- `approved` 不允许再提交或再编辑。
- `submitted` 不允许直接删除。
- `rejected` 不允许直接审批通过，必须先回到 `draft` 再重新提交。
- `HR` 首期不允许执行审批通过或审批驳回。
- 员工不允许新建评估单。

### 页面按钮映射

| 页面 | 状态 | 可见按钮 |
| --- | --- | --- |
| 我的考核 | `draft` | 保存、提交 |
| 我的考核 | `submitted` | 查看 |
| 已发起考核 | `draft` | 编辑、删除、查看 |
| 已发起考核 | `submitted` | 查看 |
| 待我审批 | `submitted` | 通过、驳回、查看 |

补充说明：

1. 首期评估单的唯一提交入口固定在“我的考核”，由员工执行。
2. `已发起考核` 页面不提供发起人代提交按钮。
3. 审批权固定只属于部门经理。

## 2. 目标状态机

### 状态集合

| 状态 | 含义 |
| --- | --- |
| `draft` | 草稿 |
| `in-progress` | 进行中 |
| `completed` | 已完成 |
| `cancelled` | 已取消 |

### 合法流转

| 当前状态 | 动作 | 下一状态 |
| --- | --- | --- |
| `draft` | 首次更新进度 | `in-progress` |
| `in-progress` | 更新进度 | `in-progress` |
| `in-progress` | 达成目标值 | `completed` |
| `draft` | 取消 | `cancelled` |
| `in-progress` | 取消 | `cancelled` |

### 非法流转

- `completed` 不允许再更新进度。
- `cancelled` 不允许再恢复成进行中，首期需要重新建目标。
- 首期没有独立“发布目标 / 完成目标 / 取消目标”动作接口时，服务端通过 `progressUpdate` 自动承接 `draft -> in-progress` 与 `in-progress -> completed`。

## 3. 360 环评任务状态机

### 角色与可见范围基线

- 发起人：`HR` 或 `部门经理`
- 评价人：发起任务时显式选择
- 被评价员工：只能查看汇总结果
- `HR` 和对应 `部门经理`：可查看汇总和单条反馈

### 状态集合

| 状态 | 含义 |
| --- | --- |
| `draft` | 未发起 |
| `running` | 已发起，等待评价 |
| `closed` | 已关闭 |

### 环评记录状态集合

| 状态 | 含义 |
| --- | --- |
| `draft` | 未提交 |
| `submitted` | 已提交 |

### 合法流转

| 当前状态 | 动作 | 下一状态 |
| --- | --- | --- |
| `draft` | 发起任务 | `running` |
| `running` | 提交反馈 | `running` |
| `running` | 全部完成或人工关闭 | `closed` |

### 非法流转

- `closed` 后不允许再新增反馈。
- `draft` 状态不允许查看汇总结果。
- 超过截止时间默认不允许补交。
- 同一评价人不允许对同一任务重复提交。

## 4. PIP 状态机

### 来源规则基线

- 支持从评估详情发起，自动带入 `assessmentId`
- 支持脱离评估单独立创建
- 首期不做自动建议触发

### 状态集合

| 状态 | 含义 |
| --- | --- |
| `draft` | 草稿 |
| `active` | 进行中 |
| `completed` | 完成 |
| `closed` | 关闭 |

### 合法流转

| 当前状态 | 动作 | 下一状态 |
| --- | --- | --- |
| `draft` | 启动 PIP | `active` |
| `active` | 提交跟进 | `active` |
| `active` | 标记完成 | `completed` |
| `active` | 关闭 | `closed` |

### 非法流转

- `completed` 和 `closed` 不允许继续写跟进记录。
- `draft` 不允许直接完成。

## 5. 晋升状态机

### 来源规则基线

- 支持从评估详情发起，自动带入 `assessmentId`
- 支持脱离评估单独立创建
- 首期不做自动建议触发

### 状态集合

| 状态 | 含义 |
| --- | --- |
| `draft` | 草稿 |
| `reviewing` | 评审中 |
| `approved` | 通过 |
| `rejected` | 驳回 |

### 合法流转

| 当前状态 | 动作 | 下一状态 |
| --- | --- | --- |
| `draft` | 提交评审 | `reviewing` |
| `reviewing` | 评审通过 | `approved` |
| `reviewing` | 评审驳回 | `rejected` |

### 非法流转

- `approved` 不允许再变更为 `rejected`。
- `rejected` 首期不支持直接重开，需复制新单据。

### 合法流转

| 当前状态 | 动作 | 下一状态 |
| --- | --- | --- |
| `draft` | 确认薪资 | `confirmed` |

## 7. 二期主题 4 自动建议状态机（冻结）

### 适用边界

1. 本节只用于二期主题 4 的状态与人工确认链路冻结，不代表当前仓库已实现自动建议引擎。
2. 首批自动建议对象固定只覆盖：
   - `pip`
   - `promotion`
3. 自动建议只允许基于现有评估结果域生成，不扩展到薪资、自动审批流、跨模块驾驶舱或外部数据域。
4. 自动建议不能改写首期已关闭的手工主链：
   - `PIP` 仍以手工创建和既有状态流转为准
   - `晋升` 仍以手工创建、手工提交评审和手工评审为准
5. 同一 `assessmentId + suggestionType` 在 `pending / accepted` 状态下最多只允许存在 1 条建议记录。

### 建议触发基线

1. 只有评估单进入 `approved` 后，才允许产出自动建议。
2. `draft / submitted / rejected` 的评估单不允许产出自动建议。
3. 首批阈值固定复用当前已冻结的绩效等级口径，不新增独立评分体系。

| 评估结果 | 建议类型 | 处理结果 |
| --- | --- | --- |
| `grade = C`，即 `totalScore < 70` | `pip` | 生成 `pending` 建议 |
| `grade = S / A`，即 `totalScore >= 80` | `promotion` | 生成 `pending` 建议 |
| `grade = B` | 无 | 不生成建议 |

### 建议状态集合

| 状态 | 含义 |
| --- | --- |
| `pending` | 系统已产生建议，等待人工处理 |
| `accepted` | 人工已确认建议成立，可继续按建议手工创建正式单据 |
| `ignored` | 人工已查看但本轮不采用建议，建议关闭 |
| `rejected` | 人工已判定建议不成立，建议关闭 |
| `revoked` | 待处理或已采用建议被认定为错误建议并撤销，建议关闭 |

补充说明：

1. 本轮不引入 `expired` 独立业务状态。
2. 超期未处理只作为 `pending` 的时间提醒或筛选维度，不进入状态机。
3. “人工新建正式单据覆盖建议”不是独立状态；覆盖后建议仍保持 `accepted`，并记录已关联的正式单据。

### 合法流转

| 当前状态 | 动作 | 下一状态 | 执行人 |
| --- | --- | --- | --- |
| `pending` | 采用建议 | `accepted` | `HR` / 有对应范围和创建权限的部门经理 |
| `pending` | 忽略建议 | `ignored` | `HR` / 有对应范围和创建权限的部门经理 |
| `pending` | 驳回建议 | `rejected` | `HR` / 有对应范围和创建权限的部门经理 |
| `pending` | 撤销建议 | `revoked` | `HR` |
| `accepted` | 发现误报后撤销 | `revoked` | `HR` |

### 人工确认链路基线

1. 建议生成后先进入 `pending`，只进入“待人工处理”链路。
2. `accepted` 不等于自动创建正式单据。
3. `accepted` 后系统只允许提供“去创建”动作，跳转到既有 `PIP` 或 `晋升` 手工创建入口，并预填：
   - `assessmentId`
   - `employeeId`
   - `suggestionType`
   - `suggestionId`
4. 若处理人直接通过现有手工创建入口新建正式单据来覆盖建议，则建议状态仍记为 `accepted`，并记录关联的正式单据标识。
5. 正式单据是否创建成功，仍以既有 `PIP / 晋升` 模块自己的创建结果为准。
6. 首批不允许“采用建议后自动创建 `PIP / 晋升` 单据”。

### 可见与处理角色

1. `HR`
   - 可查看全部建议
   - 可采用、忽略、驳回、撤销全部建议
2. 部门经理
   - 仅可查看自己直接管理部门及其下级部门范围内的建议
   - 仅可采用、忽略、驳回自己范围内且自己具备对应创建权限的建议
   - 不可执行撤销
3. 员工
   - 不可查看建议
   - 不可采用、忽略、驳回、撤销建议

### 错误建议撤销与追责口径

1. `revoked` 只用于“错误建议”关闭，不用于代替 `ignored` 或 `rejected`。
2. 执行撤销时必须填写撤销原因，原因至少归入以下类别之一：
   - 阈值命中错误
   - 评估数据已更正
   - 数据范围判断错误
   - 重复建议
3. 撤销时必须保留最小审计字段：
   - `assessmentId`
   - `suggestionType`
   - `triggerGrade`
   - `triggerScore`
   - `ruleVersion`
   - `operatorId`
   - `operateTime`
   - `revokeReason`
4. 若建议已进入 `accepted` 且已据此创建正式 `PIP / 晋升` 单据，撤销建议不自动回滚正式单据。
5. 已创建的正式单据，必须回到其所属模块按既有状态机处理：
   - `PIP` 按 `draft / active / completed / closed`
   - `晋升` 按 `draft / reviewing / approved / rejected`

### 非法流转

- `accepted` 不允许直接改为 `rejected` 或 `ignored`。
- `ignored` 不允许再改回 `pending` 或 `accepted`。
- `rejected` 不允许再改回 `pending` 或 `accepted`。
- `revoked` 不允许再恢复。
- 部门经理不允许撤销建议。
- 员工不允许查看或处理建议。
- `accepted` 不允许自动跳过人工创建环节直接生成正式 `PIP / 晋升` 单据。

## 8. 二期主题 5 自动审批流状态机与节点规则（冻结）

### 适用边界

1. 本节只用于二期主题 5 窗口 B 的状态机、审批节点和人工介入规则冻结，不代表自动审批流主题已完成全部阶段 0 冻结。
2. 自动审批流是“审批实例层”能力，不改写源业务对象首期已关闭的业务状态机正文。
3. 首批只允许固定串行单链路，不支持并行会签、条件分支、动态加签、减签或任意插入节点。
4. 同一时刻只允许存在 1 个当前激活节点和 1 个当前实际审批人。
5. `pending_resolution / in_review / manual_pending` 只属于审批实例状态，不直接作为源业务对象对外展示的新业务状态。

### 节点模型

1. 审批节点固定为“预定义顺序节点列表”：
   - 节点顺序在实例创建时固定
   - 后续审批过程中不允许普通审批人改节点顺序
2. 每个节点只能选择 1 种审批人解析方式，不允许混用多种解析规则后再二次择优。
3. 每个节点在激活时必须解析出且只解析出 1 个有效审批人；解析结果为 0 人或多于 1 人时，实例不得继续自动流转。
4. 允许的审批人解析方式白名单固定为：
   - `specified_user`：指定用户
   - `applicant_direct_manager`：申请人直属经理
   - `employee_department_manager`：被申请对象所在部门负责人
   - `department_tree_role`：部门树范围内指定角色
   - `hr_manual_assign`：由 `HR` 人工指定
5. 节点审批人解析时机固定为：
   - 实例创建后激活首节点前
   - 前一节点通过、准备激活下一节点前
   - 当前审批人失效、离职、停用或组织关系变化后重新激活前

### 审批实例状态集合

| 状态 | 含义 |
| --- | --- |
| `pending_resolution` | 节点审批人未能唯一解析，等待 `HR` 人工指定 |
| `in_review` | 当前存在且仅存在 1 个待审批节点 |
| `manual_pending` | 因超时、审批人失效或组织变化需要 `HR` 人工介入 |
| `approved` | 全部串行节点已审批通过 |
| `rejected` | 任一节点已驳回 |
| `withdrawn` | 发起人在允许时机内主动撤回 |
| `terminated` | `HR` 强制终止自动审批实例 |

### 节点状态集合

| 状态 | 含义 |
| --- | --- |
| `pending` | 当前待审批 |
| `approved` | 当前节点已通过 |
| `rejected` | 当前节点已驳回 |
| `timed_out` | 当前节点超时未处理 |
| `transferred` | 当前节点已转办给新审批人 |
| `cancelled` | 因撤回或强制终止而取消 |

### 合法流转

| 当前状态 | 动作 | 下一状态 | 执行人 |
| --- | --- | --- | --- |
| 创建实例 | 首节点审批人唯一解析成功 | `in_review` | 系统 |
| 创建实例 | 首节点审批人解析失败 | `pending_resolution` | 系统 |
| `pending_resolution` | `HR` 指定审批人并恢复 | `in_review` | `HR` |
| `in_review` | 当前节点审批通过且仍有后续节点 | `in_review` | 当前审批人 |
| `in_review` | 当前节点审批通过且已到末节点 | `approved` | 当前审批人 |
| `in_review` | 当前节点审批驳回 | `rejected` | 当前审批人 |
| `in_review` | 当前节点转办 | `in_review` | 当前审批人 / `HR` |
| `in_review` | 当前节点超时 | `manual_pending` | 系统 |
| `in_review` | 当前审批人失效或组织关系变化导致当前节点失效 | `manual_pending` | 系统 |
| `manual_pending` | `HR` 重新指定审批人并恢复 | `in_review` | `HR` |
| `pending_resolution` | 发起人撤回 | `withdrawn` | 发起人 |
| `in_review` | 发起人撤回（仅首节点未被处理） | `withdrawn` | 发起人 |
| `pending_resolution` | `HR` 强制终止 | `terminated` | `HR` |
| `manual_pending` | `HR` 强制终止 | `terminated` | `HR` |
| `in_review` | `HR` 强制终止 | `terminated` | `HR` |

### 超时、转办、驳回与撤回口径

1. 超时：
   - 超时后不允许自动通过，也不允许自动驳回。
   - 当前节点记为 `timed_out`，审批实例进入 `manual_pending`。
   - 进入 `manual_pending` 后，只允许 `HR` 恢复、强制终止或回退到既有手工审批主链。
2. 转办：
   - 允许转办，但只允许作用于当前激活节点。
   - 当前审批人可向“仍满足当前节点解析边界”的候选人转办。
   - `HR` 可在异常场景下强制转办到边界外用户，但必须留审计原因。
   - 转办不改变审批实例状态，节点保持待处理，只更新当前审批人与转办审计。
3. 驳回：
   - 任一节点驳回即终止自动审批实例，实例状态进入 `rejected`。
   - 源业务对象回到既有“驳回等价状态”：
     - 已定义 `rejected` 的对象，回到 `rejected`
     - 未定义 `rejected` 的对象，回到其既有可编辑草稿等价状态
   - 驳回后不允许从自动审批实例内直接重开，需按源业务对象既有主链重新提交。
4. 撤回：
   - 发起人仅可在 `pending_resolution` 或“首节点仍为 `pending` 且尚无人处理”的 `in_review` 状态撤回。
   - 任一节点一旦发生 `approved / rejected / transferred / timed_out`，发起人不再拥有撤回权。
   - 撤回后审批实例进入 `withdrawn`，全部未处理节点记为 `cancelled`。

### 与现有手工审批主链的兼容规则

1. 自动审批实例是源业务对象“提交后到最终结论前”的内部审批层，不新增第二套对外业务状态集合。
2. 源业务对象在自动审批实例处于 `pending_resolution / in_review / manual_pending` 时，继续停留在自身既有的 `submitted / reviewing` 等价状态。
3. 自动审批实例进入 `approved` 时，源业务对象进入自身既有的最终通过状态。
4. 自动审批实例进入 `rejected` 时，源业务对象进入自身既有的驳回或回编状态。
5. 只有 `HR` 允许把 `pending_resolution / manual_pending` 的自动审批实例回退到现有手工审批主链。
6. 回退到既有手工审批主链时：
   - 自动审批实例结束并记为 `terminated`
   - 源业务对象不自动通过、不自动驳回
   - 后续审批责任切回源业务对象已有的手工审批入口和审计链路

### 人工介入边界

1. 以下动作只允许 `HR` 操作：
   - 审批人解析失败后的人工指定
   - 超时或审批人失效后的恢复
   - 强制转办到解析边界外用户
   - 强制终止自动审批实例
   - 回退到现有手工审批主链
2. 普通审批人和发起人都不允许在实例创建后修改节点顺序、节点数量和节点解析规则。
3. 部门经理或其他业务审批人只能处理自己当前被激活的节点，不能越级处理后续节点。
4. 员工或发起人不能以“撤回”代替“驳回 / 终止”，也不能在节点已被处理后重新打开实例。

### 非法流转

- `approved / rejected / withdrawn / terminated` 不允许再恢复为 `in_review`。
- `manual_pending` 不允许由普通审批人直接恢复。
- `pending_resolution` 不允许由普通审批人直接指定审批人。
- 非 `HR` 不允许把自动审批实例回退到既有手工审批主链。
- 非当前审批人不允许审批、驳回或普通转办当前节点。
- 不允许并行激活两个节点。
- 不允许因为超时而自动审批通过。

## 非法流转处理方式

1. 后端返回业务失败，`message` 明确指出原因。
2. 前端显示错误提示并保持当前页面状态不变。
3. 不允许因为无效动作而“假成功”刷新列表。
