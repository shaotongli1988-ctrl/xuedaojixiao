# 权限矩阵

## 文档职责

这份文档是“权限规则”的唯一事实源。

它只负责定义：

1. 角色
2. 菜单权限
3. 页面按钮权限
4. 接口权限
5. 数据范围权限

## 不负责内容

这份文档不负责定义：

- 页面字段布局
- 接口字段格式
- 脱敏算法细节

脱敏细节以 `12-数据权限与脱敏规则.md` 为准。

## 权限命名规则

沿用当前 Cool 项目既有风格，统一使用：

`performance:<resource>:<action>`

示例：

- `performance:assessment:page`
- `performance:assessment:approve`
- `performance:goal:progressUpdate`
- `performance:salary:info`

## 接口权限

### HR 管理员

- 拥有绩效模块全部查看类接口权限。
- 拥有薪资和指标配置接口权限。
- 拥有评估单新增、编辑、删除和归档管理权限。
- 首期不拥有评估审批动作权限。
- 首期独占薪资管理入口和金额维护权限。

### 部门经理

- 可访问：
  - 团队评估分页、详情、审批
  - 团队目标分页、更新进度
  - PIP 启动、跟进、完成、关闭
  - 晋升提交、评审
  - 环评相关接口
- 不可访问：
  - 指标库配置
  - 全量薪资分页
  - 薪资详情和薪资调整接口

### 员工

- 可访问：
  - 我的考核列表和详情
  - 本人自评提交
  - 本人目标查看和进度更新
  - 指派给自己的环评反馈提交
- 不可访问：
  - 他人评估单
  - 审批接口
  - 薪资管理后台
- 不可创建评估单
- 不可访问任何薪资管理接口

## 二期主题 6：跨模块驾驶舱权限正式冻结（2026-04-18）

本节负责冻结主题 6 的正式查看权限和拒绝语义；它不开放任何明细、动作按钮或源域后台跳转。

### 正式权限键

| 能力 | 权限键 | 说明 |
| --- | --- | --- |
| 跨模块驾驶舱汇总查看 | `performance:dashboard:crossSummary` | 仅用于主题 6 的只读聚合接口与页面 |

### 角色权限矩阵

| 能力 | `HR` | 部门经理 | 员工 | 说明 |
| --- | --- | --- | --- | --- |
| 查看跨模块驾驶舱入口 | 是 | 是 | 否 | 员工无入口 |
| 请求 `/admin/performance/dashboard/crossSummary` | 是 | 是 | 否 | 经理仅限本人部门树范围 |
| 查看全局汇总 | 是 | 否 | 否 | 经理不得越权看全局 |
| 查看部门树范围汇总 | 是 | 是 | 否 | 经理仅在来源支持部门树映射时可见真实值 |
| 查看招聘 / 培训 / 会议原始明细 | 否 | 否 | 否 | 全角色禁止下钻 |

### 权限正文

1. `员工` 不可见跨模块驾驶舱入口、页面和接口。
2. `HR` 拥有主题 6 三类指标族的全量聚合查看权限，但仍只可见聚合值、状态提示和更新时间。
3. `部门经理` 仅可在本人部门树范围内查看聚合结果，不得请求或缓存跨部门结果。
4. 若来源域当前不支持部门树映射、组织归属或部门裁剪，`部门经理` 不得回退查看全局值；对应卡片必须显示 `unavailable`。
5. 不允许因为已有 `performance:dashboard:summary` 权限，就自动继承主题 6 的跨模块驾驶舱查看权限。
6. 任何角色都不得通过主题 6 接口下钻查看招聘、培训、会议域原始明细。

## 二期主题 3：`cool-uni` 登录与权限设计（2026-04-17）

本节只负责固定二期主题 3 阶段 1 的移动端登录态复用、页面显隐、动作白名单和拒绝策略；不代表 `cool-uni` 页面已实现，也不新增移动端专用权限键。

### 登录态复用方案

1. 移动端登录必须复用现有后台验证码登录链路：
   - `/admin/base/open/captcha`
   - `/admin/base/open/login`
   - `/admin/base/open/refreshToken`
   - `/admin/base/comm/permmenu`
   - `/admin/base/comm/logout`
2. 不新增移动端专用账号体系、手机号快捷登录、短信登录或独立 token 协议。
3. 移动端页面可见性和动作权限继续以现有 `permmenu` 和权限键为准，不新增 `mobile:*` 或 `uni:*` 权限键。
4. 移动端工作台入口卡片必须按“两层裁剪”生成：
   - 第一层：只保留主题 3 已冻结的首批页面白名单
   - 第二层：再与当前登录用户的 `permmenu` 权限交集裁剪
5. 即使当前账号在 PC 端还拥有其他权限，如“已发起考核”“指标库”“PIP”“晋升”“薪资”，移动端首批也不得显示这些页面入口。

### 首批明确不开放的移动端动作

1. 评估单发起、删除、导出
2. 目标新增、删除、导出
3. 环评任务发起、导出
4. 指标库、驾驶舱、PIP、晋升、薪资相关全部动作
5. 任何后台配置动作

### 无权限与越权拒绝策略

1. 页面级：
   - 若未登录或 token 失效，进入登录页，不得展示业务页旧缓存。
   - 若页面不在移动端首批白名单，或当前账号缺少对应权限键，则工作台不显示入口。
   - 若用户通过深链或本地路由强行进入无权限页面，显示“无权限访问该页面”空态，不放大权限、不切换为其他业务页。
2. 动作级：
   - 若动作不在首批动作白名单，即使 PC 端有对应权限，移动端首批也不得展示按钮。
   - 若按钮已展示但后端返回 `403` 或等效拒绝结果，前端只提示“无权限执行该操作”，不做乐观更新。
3. 数据级：
   - 员工越权查询一律按“本人口径”重新裁剪，不因前端携带 `employeeId`、`departmentId` 放宽。
   - 经理越权查询一律按“部门树口径”重新裁剪，不因前端携带跨部门 `employeeId` 放宽。
4. 列表为空与无权限要分开：
   - 合法范围内无数据：显示空数据态
   - 页面或动作无权限：显示无权限态
5. 不允许通过“前端隐藏、后端放行”的方式交付移动端权限；后端仍是最终鉴权裁判。

## 落地规则

1. 前端用 `v-permission` 和 `_permission` 控制按钮。
2. 后端必须再次校验角色和数据范围。
3. 菜单可见不等于数据可见，列表查询必须附带服务端数据范围条件。
4. 部门经理的数据范围统一按组织架构部门树计算，不单独支持跨部门兼任授权。

1. 自动建议权限不复用现有 `performance:pip:*` 或 `performance:promotion:*` 权限键，统一使用 `performance:suggestion:*`。
2. `HR` 拥有建议查看和全部处理权限，但这不等于放宽 `PIP / 晋升`、薪资或其他模块的既有权限边界。
3. 部门经理只允许处理自己部门树范围内的建议；超范围查询和动作必须被后端拒绝。
4. 部门经理执行 `accept` 前，除具备 `performance:suggestion:accept` 外，还必须具备对应正式单据的创建权限：
   - `pip` 建议要求 `performance:pip:add`
   - `promotion` 建议要求 `performance:promotion:add`
5. 员工既没有建议入口，也没有建议接口和按钮权限；前端隐藏不构成放行依据，后端必须继续拒绝。
6. 若后续实现采用独立页面或菜单入口，统一以 `performance:suggestion:page` 作为入口权限键，不另起新键名。

## 二期主题 5：自动审批流权限冻结基线（2026-04-17）

本节记录自动审批流进入开发前已冻结的权限键和角色边界。自动审批流不新增独立菜单页权限，统一通过对象详情、待办入口和动作权限进入。

### 权限键定义

| 能力 | 权限键 |
| --- | --- |
| 配置详情 | `performance:approvalFlow:configInfo` |
| 配置保存 | `performance:approvalFlow:configSave` |
| 实例详情 | `performance:approvalFlow:info` |

### 角色权限矩阵

| 权限键 | HR 管理员 | 部门经理 | 员工 | 说明 |
| --- | --- | --- | --- | --- |
| `performance:approvalFlow:configInfo` | 是 | 否 | 否 | 仅 `HR` 可查看配置 |
| `performance:approvalFlow:configSave` | 是 | 否 | 否 | 仅 `HR` 可保存配置 |
| `performance:approvalFlow:info` | 是 | 是 | 是 | `HR` 全量；经理仅部门树范围 / 本人参与；员工仅本人发起实例 |
| `performance:approvalFlow:approve` | 是 | 是 | 否 | 还必须命中“当前节点当前审批人” |
| `performance:approvalFlow:reject` | 是 | 是 | 否 | 还必须命中“当前节点当前审批人” |
| `performance:approvalFlow:transfer` | 是 | 是 | 否 | 普通转办仅当前审批人；`HR` 可强制转办 |
| `performance:approvalFlow:withdraw` | 是 | 是 | 是 | 还必须命中“当前实例发起人” |
| `performance:approvalFlow:remind` | 是 | 是 | 是 | 还必须命中“当前实例发起人”或 `HR` |
| `performance:approvalFlow:resolve` | 是 | 否 | 否 | 仅 `HR` 处理 `pending_resolution / manual_pending` |
| `performance:approvalFlow:fallback` | 是 | 否 | 否 | 仅 `HR` 可回退到手工审批主链 |
| `performance:approvalFlow:terminate` | 是 | 否 | 否 | 仅 `HR` 可强制终止 |

### 权限规则

1. 自动审批流配置权限不继承现有 `assessment / promotion` 的 `page / info` 查看权限，必须使用独立配置权限键。
2. `approve / reject / transfer` 同时要求：
   - 调用方拥有对应动作权限键
   - 调用方是当前节点当前审批人
   - 实例状态为 `in_review`
3. `withdraw / remind` 同时要求：
   - 调用方拥有对应动作权限键
   - 调用方是当前实例发起人，或调用方是 `HR`
4. `resolve / fallback / terminate` 只允许 `HR`；部门经理和员工即使能看到实例，也不能执行这三类人工介入动作。
5. 员工没有配置、审批、驳回、转办权限，但可以在本人发起且规则允许的实例上执行 `withdraw / remind`。
6. 自动审批流权限键不替代源业务对象的 `submit`、`approve`、`reject`、`review` 权限；源业务对象的提交仍按既有权限校验，自动审批流只接管“提交后的审批层”。
7. 若自动审批流实例已回退到手工审批主链，后续动作重新回到 `performance:assessment:approve`、`performance:assessment:reject`、`performance:promotion:review` 的既有权限链路。
