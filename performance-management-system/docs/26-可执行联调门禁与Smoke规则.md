# 可执行联调门禁与 Smoke 规则

## 文档职责

这份文档是“阶段联调门禁”和“端到端 smoke 放行标准”的唯一事实源。

它只负责定义：

1. 阶段联调必须经过的执行顺序
2. Smoke 检查的最小覆盖范围
3. 什么情况下可以把任务从 `blocked` 推进到下一状态

## 不负责内容

这份文档不负责定义：

- 业务需求全文
- API 字段全文
- 菜单、权限、状态和数据模型全文

这些内容分别以 `01`、`04`、`05`、`06`、`10`、`12` 文档为准。

## 基本原则

1. 阶段联调必须以“可执行结果”而不是“聊天结论”作为放行证据。
2. 单测、类型检查、构建通过，不等于联调通过。
3. 登录成功，不等于模块联调通过。
4. 只有端到端 smoke 通过，才能证明“运行事实源”已成立。
5. 没有 smoke 通过记录，任务卡不得从 `blocked` 改为 `done` 或“进入下一阶段”。

## 阶段联调强制顺序

进入阶段联调后，必须按下面顺序执行，不允许跳步：

1. 后端启动成功
   - `cool-admin-midway npm run dev` 可稳定启动
2. 联调种子数据准备成功
   - 目标环境已执行阶段联调 seed 或等效初始化
3. 登录成功
   - 至少一名联调账号可通过验证码和登录接口拿到 token
4. 权限菜单成功
   - `permmenu` 返回与角色相符的菜单和权限集合
5. 模块主接口成功
   - 当前模块主链至少一条真实接口链路通过
6. 数据范围成功
   - 至少验证一条“可见”和一条“不可见”数据范围
7. 前端门禁成功
   - `type-check`、`build` 通过
8. 任务卡回写完成
   - 任务卡已回写执行命令、结果、阻塞和复验状态

只要其中任一步未通过，阶段联调仍视为 `blocked`。

## Smoke 最小覆盖范围

首期绩效模块的 smoke 至少覆盖：

1. 验证码获取
2. 真实登录
3. `permmenu`
4. 模块 1：评估单列表主路径
5. 模块 2：目标列表主路径
6. 至少 3 类角色
   - `HR`
   - `部门经理`
   - `普通员工`
7. 至少 1 条越权不可见校验

当前绩效项目以如下脚本为首期阶段联调 smoke 基线：

- [cool-admin-midway/scripts/smoke-stage2-performance.mjs](/Users/shaotongli/Documents/xuedao/cool-admin-midway/scripts/smoke-stage2-performance.mjs)

6. `/admin/performance/goal/page`

若后续模块扩展联调范围，可扩展脚本，但不得削弱本节定义的最小覆盖面。

## 放行标准

只有同时满足下面条件，才允许把当前阶段联调判定为通过：

1. smoke 脚本返回成功退出码
2. smoke 输出中不存在 `FAIL`
3. 任务卡已记录执行命令、执行日期和通过结论
4. 若曾存在 `P0 / P1`，对应问题已复验关闭

## 阻塞处理规则

出现以下任一情况，直接判定为阶段联调阻塞：

1. 登录失败
2. 受保护接口报错
3. `ctx.admin`、权限缓存、部门范围等共享上下文异常
4. 菜单权限与角色不一致
5. 主链接口成功但数据范围错误
6. smoke 未运行
7. smoke 已运行但未回写任务卡

## 证据回写要求

每次阶段联调执行后，至少回写：

1. 执行命令
2. 执行日期
3. 通过或失败结论
4. 失败范围
5. 是否阻塞下一阶段
