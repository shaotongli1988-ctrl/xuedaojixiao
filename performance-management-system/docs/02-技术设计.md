# 技术设计

## 文档职责

这份文档是“技术实现方案”的唯一事实源。

它只负责定义：

1. 前后端模块边界
2. 代码目录落点
3. 组件复用策略
4. 聚合统计实现方式
5. 技术约束和明确不做什么

## 不负责内容

这份文档不负责定义：

- 表结构细节
- 接口字段细节
- 权限矩阵细节
- 测试用例细节

这些内容分别以 `03`、`04`、`06`、`09` 文档为准。

## 设计目标

在不超出 `xuedao` 当前技术架构的前提下，为绩效管理系统首期提供可直接落地的前后端模块设计。首期实现必须完全基于现有的 `cool-admin-midway` 和 `cool-admin-vue` 组织方式，不额外引入新的前端框架、后端框架、审批引擎或独立报表服务。

## 方案概览

整体方案采用“单一绩效模块 + 多业务资源子实体”的实现方式：

1. 后端新增一个 `performance` 模块。
2. 前端新增一个 `performance` 模块。
3. 评估、目标、指标、环评、PIP、晋升、薪资均作为该模块下的子资源。
4. 标准 CRUD 能力优先走 Cool 基座的标准控制器和服务模式。
5. 审批、提交、跟进、汇总等动作接口在标准 CRUD 基础上单独增加。

## 技术边界

### 现有后端技术基线

- 目录：`/Users/shaotongli/Documents/xuedao/cool-admin-midway`
- 框架：Midway 3
- ORM：TypeORM
- 数据库驱动：`mysql2`
- 基座：`@cool-midway/core`
- 实现模式：
  - `src/modules/<module>/config.ts`
  - `src/modules/<module>/entity/*.ts`
  - `src/modules/<module>/service/*.ts`
  - `src/modules/<module>/controller/admin/*.ts`

### 现有前端技术基线

- 目录：`/Users/shaotongli/Documents/xuedao/cool-admin-vue`
- 框架：Vue 3 + TypeScript
- UI：Element Plus
- 状态管理：Pinia
- 构建：Vite
- 图表：`echarts` + `vue-echarts`
- 基座：
  - `ModuleConfig`
  - `service.*.*`
  - `v-permission`
  - `@cool-vue/crud`

### 明确不做

1. 不新增独立审批流引擎。
2. 不新增独立报表或 BI 服务。
3. 不新增新的前端状态管理库。
4. 不新增新的 ORM 或查询框架。
5. 不把绩效模块拆成多个独立仓库或多个运行时服务。

## 页面结构

- 页面职责：管理类页面统一入口
- 适用页面：
  - 已发起考核
  - 待我审批
  - 目标地图
  - 指标库
  - 360 环评
  - PIP
  - 晋升
  - 薪资
- 主要组成：
  - 筛选表单
  - 数据表格
  - 行内动作
  - 弹窗或抽屉

### 页面结构二：详情 / 动作页

- 页面职责：展示单据详情并承载审批、提交、跟进动作

  - 我的考核详情
  - 评估详情
  - PIP 详情
  - 晋升详情
  - 薪资详情
- 主要组成：
  - 基础信息
  - 状态信息
  - 评分明细或记录列表
  - 动作区

### 页面结构三：驾驶舱看板页

- 页面职责：展示绩效总览和待办
- 主要组成：
  - KPI 卡片

  - 部门分布图
  - 待审批提醒

### 标准 CRUD 子资源

下面这些资源优先复用 Cool 的标准 CRUD 风格：

- `assessment`
- `goal`
- `indicator`
- `pip`
- `promotion`
- `salary`

接口风格基线：

- `POST /page`
- `GET /info`
- `POST /add`
- `POST /update`
- `POST /delete`

### 动作型接口

下面这些动作在标准 CRUD 之外增加自定义接口：

- 评估单提交：`/submit`
- 评估单审批通过：`/approve`
- 评估单审批驳回：`/reject`
- 目标进度更新：`/progress/update`
- 环评反馈提交：`/feedback/submit`
- PIP 跟进提交：`/track`
- 晋升审批：`/review`
- 薪资调整记录新增：`/change/add`

## 数据流与调用关系

### 标准列表页调用关系

1. 前端页面通过 `service.performance.<resource>.page()` 发起请求。
2. 请求走 `cool-admin-vue/src/cool/service/request.ts`，统一附带 token 和语言。
3. 后端控制器进入 `controller/admin/<resource>.ts`。
4. 控制器调用对应 `service/<resource>.ts`。
5. Service 使用 TypeORM Repository 读写实体。
6. 响应统一返回 `{ code, message, data }`，前端只在 `code === 1000` 时认为成功。

### 审批动作调用关系

1. 页面详情或待办列表点击审批动作。
2. 前端调用动作接口，如 `service.performance.assessment.approve()`。
3. 后端先校验权限和状态是否合法。
4. 合法时更新主表状态并写入审批意见或操作记录。
5. 返回最新状态，前端刷新详情和列表。

### 驾驶舱调用关系

1. 驾驶舱页面只调用 `dashboard` 聚合接口。
2. `dashboard` service 内部聚合 `assessment`、`goal` 等表。
3. 前端不直接在页面里拼接多个资源请求再二次统计。

## 复用关系

### 后端复用

1. 所有实体继承现有基座 [base.ts](/Users/shaotongli/Documents/xuedao/cool-admin-midway/src/modules/base/entity/base.ts) 的公共字段模式。
2. 标准 CRUD 优先复用 `@CoolController` 和 `BaseService`。
3. 模块配置沿用现有 [demo/config.ts](/Users/shaotongli/Documents/xuedao/cool-admin-midway/src/modules/demo/config.ts) 形式。

### 前端复用

1. 模块配置沿用 [config.ts](/Users/shaotongli/Documents/xuedao/cool-admin-vue/src/modules/base/config.ts) 的 `ModuleConfig` 结构。
2. 请求能力沿用 [request.ts](/Users/shaotongli/Documents/xuedao/cool-admin-vue/src/cool/service/request.ts) 和 [base.ts](/Users/shaotongli/Documents/xuedao/cool-admin-vue/src/cool/service/base.ts)。

## 关键设计决策

### 决策 1：只建一个 `performance` 模块

- 为什么这样做：
  - 与当前 Cool 项目按模块管理的方式一致。
  - 可以把权限、菜单、聚合统计、共享组件放在同一模块下管理。
- 不这样做的原因：
  - 如果一开始拆成多个独立模块，菜单、权限、统计和复用关系会过早碎片化。

### 决策 2：标准 CRUD + 动作接口混合

- 为什么这样做：
  - 与当前基座能力一致，改动面最小。
  - 列表、详情、增删改查保持统一风格。
  - 只有提交、审批、跟进这类动作才单独扩展接口。
- 不这样做的原因：
  - 全部自定义接口会增加前后端复杂度。

### 决策 3：驾驶舱只走聚合接口

- 为什么这样做：
  - 避免前端拼多个接口再算统计，口径容易漂。
  - 统计逻辑集中在后端更容易复用和修正。
- 不这样做的原因：
  - 页面内自行计算会导致多个页面出现不同口径。

### 决策 4：首期不接独立审批引擎

- 为什么这样做：
  - 当前项目没有现成审批引擎基线。
  - 首期业务状态和节点数可通过状态机 + 动作接口覆盖。
- 不这样做的原因：
  - 新增审批引擎会直接超出当前技术框架。

## 风险与回退方案

- 风险：过早抽取过多公共组件，导致模块边界不清。
  - 回退方式：先保留模块内组件，跨页面稳定后再抽公共层。
- 风险：评估、待办、我的考核使用三套接口模型。
  - 回退方式：统一以 `assessment` 为核心数据源，只在查询条件和动作权限上区分。
- 风险：驾驶舱统计依赖跨模块数据，但首期未开发其他业务模块。
  - 回退方式：首期只上线绩效域可直接计算的指标，其他指标保留扩展位。
