# 二期范围正式冻结版

## 文档职责

这份文档用于冻结“二期总范围”，只做开发前准备，不授权直接进入设计实现。

它只负责：

1. 收口当前允许进入二期阶段 0 评审的正式范围
2. 说明为什么这些能力放到二期
3. 说明每项候选的业务价值、依赖和缺失事实源
4. 给出正式优先级、准入条件和禁止开工条件

它不负责：

1. 定义二期正式需求全文
2. 定义二期 API、表结构、状态机或权限细节
3. 把二期候选提前并入首期阶段 4 验收或阶段 5 交付
4. 把“建议优先级”直接写成排期承诺或开发开始指令

## 正式冻结结论

- 唯一结论：`二期范围已正式冻结，可进入分项设计冻结`
- 结论成立前提：
  - 首期以 [2026-04-17-阶段4-业务验收记录.md](/Users/shaotongli/Documents/xuedao/performance-management-system/tasks/2026-04-17-阶段4-业务验收记录.md) 和 [2026-04-17-阶段5-上线前交付收口.md](/Users/shaotongli/Documents/xuedao/performance-management-system/tasks/2026-04-17-阶段5-上线前交付收口.md) 为准，当前已闭环
  - 二期仍只允许做“冻结、澄清、验收口径、任务拆分”
  - 当前技术边界固定为 `cool-admin-vue + cool-admin-midway + cool-uni`
- 当前不允许的动作：
  - 直接进入二期实现
  - 以聊天结论替代冻结包
  - 把未补齐事实源的候选项推进到设计或开发

## 二期总目标

1. 冻结二期唯一候选范围，确保不再混入首期已关闭事项。
2. 明确二期优先级、进入条件、不得开工条件和首开项。
3. 把每个候选项推进到“可启动开发前评审”的正式冻结包，而不是停留在口头草案。

## 二期非目标

1. 不修改首期已关闭的功能、权限、状态机和交付结论。
2. 不开发前后端业务代码、不调整共享实现、不新增脚手架或基础设施。
3. 不把任何候选项写成“已决定开发”或“已完成设计”。
4. 不在当前阶段补二期 API、表结构、状态机、权限正文。

## 冻结原则

1. 当前二期候选只允许来自首期已明确“不纳入范围”的事项
2. 当前技术边界仍固定为 `cool-admin-vue + cool-admin-midway + cool-uni`
3. 任一候选在补齐事实源前，不得直接进入设计或开发
4. 当前文档给出的是“正式冻结优先级”，不等同于排期承诺

## 进入条件

1. 首期阶段 4 / 阶段 5 已闭环，且其结论不再回退。
2. 二期候选仅来自首期明确“不纳入范围”的事项。
3. 当前冻结结论、阻塞项、证据和下一步已回写仓库文档 / 任务卡。

## 不得开工条件

1. 任一候选项缺少最小事实源清单时，不得进入设计或开发。
2. 任一候选项若会改首期共享实现、首期状态机或首期已关闭结论，必须先单独冻结再评审。
3. 任一候选项若只能依赖聊天说明、截图或口头共识，不得推进。
4. 未建任务卡、未回写版本记录、未完成阶段 0 守卫留痕时，不得视为冻结完成。
