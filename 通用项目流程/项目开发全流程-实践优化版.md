# 多项目 CRM 项目开发全流程
## 实践优化版

## 1. 先说结论

你之前那套表已经有骨架了，但按 `multicrm` 的真实实践看，至少还有 9 个地方必须改成“固定流程”，否则项目还是会反复踩坑：

| 问题 | 以前容易怎么做 | 现在必须怎么做 |
| --- | --- | --- |
| 需求没澄清就开写 | 先写一版再边做边问 | 先做需求澄清、范围冻结、验收矩阵 |
| 测试放到最后 | 功能写完再补测 | 先写测试矩阵，再补失败用例/骨架，再开始实现 |
| 后端分层口径过旧 | `entity/mapper/service/controller` | 固定 `api/dto/domain/repo`，按五层职责开发 |
| `project_id` 放太晚 | 功能先通，后面再补过滤 | 表、接口、会话、权限、前端上下文第一天就带上 |
| 编排先写代码后补蓝图 | 先改 `server.go` 再回头补文档 | 先改 `source/13`/`source/13-flows`，再接代码 |
| 前端只盯页面 | 先写页面，再补 API/权限/路由 | 固定 `ACT -> OR -> API -> 权限 -> 路由 -> 页面` |
| UI 只做最后美化 | 页面先堆出来，最后统一调样式 | 先做 Shell、导航、内容区 recipe、token SoT |
| 主要靠 Postman/人工验证 | 本地点通就算完成 | 每一层都有最小自动化验证和门禁 |
| 发布证据不完整 | 只说“测过了” | 必须留下验证、回滚、风险、未执行项、诊断入口 |

一句话总结：

- 以前更像“写一个 CRM 功能”。
- 现在应该是“按唯一事实源、测试先行、编排驱动、门禁收口去交付一个多项目平台能力”。

## 2. 现在这套流程的四个总原则

### 2.1 先把需求说清楚，再开始设计

1. 先问清业务目标、范围、非目标、角色、状态、异常、权限。
2. 先把“这次不做什么”写下来。
3. 先把验收标准写成能测的句子，再进入设计和开发。

### 2.2 先把测试想清楚，再开始实现

1. 不是最后补测试，而是开发前先列测试矩阵。
2. 正常、异常、边界、权限、状态、契约六类证据必须提前规划。
3. 高风险动作先写失败用例或测试骨架，再写实现。

### 2.3 先按层归位，再写代码

1. 先判断改动属于页面层、项目层、编排层、模块层还是公共基座层。
2. 模块只管单一职责，编排只管跨模块调度。
3. 页面只管展示和触发，不能偷偷承担业务规则。

### 2.4 先固定壳层和合同，再铺页面和样式

1. 前端先做 Shell、导航、页面 scaffold、contracts、API 层。
2. UI 先做信息架构、内容区 recipe、token SoT。
3. Stitch 是设计 token 和布局 recipe 的桥，不是跳过前端架构直接出业务页的捷径。

## 3. 小白也能照着走的完整流程

下面这版不是“建议顺序”，而是“少一步就容易出事”的顺序。

### 阶段 T：需求澄清、范围冻结、测试先行

| 步骤 | 动作 | 输入 | 输出 | 必须做什么 | 主要防坑 |
| --- | --- | --- | --- | --- | --- |
| T01 | 收需求原话和业务背景 | 聊天记录、口头需求、反馈记录 | 原始需求清单 | 先写“要解决什么问题、谁来用、什么时候用、为什么做” | 防止一上来就按自己理解乱做 |
| T02 | 做需求澄清 | 原始需求清单 | 澄清结果 | 把角色、状态、入口、异常、权限、边界、非目标逐项问清或从现有资料补齐 | 防止做到一半才发现需求没定义 |
| T03 | 冻结范围和非目标 | 澄清结果 | `source/00-范围与边界.md` | 明确本次做什么、不做什么、影响哪些模块 | 防止边开发边改范围 |
| T04 | 写验收矩阵 | 业务流程、澄清结果 | `source/06-验收标准.md` | 正常、异常、边界、权限、状态、契约六类验收必须成文 | 防止“做完了但验不了” |
| T05 | 先列测试矩阵 | 验收矩阵、风险面 | 测试任务清单 | 先列模块测、编排测、前端测、守卫、发布前回归分别跑什么 | 防止测试漏项和最后补测失控 |
| T06 | 先补失败用例或测试骨架 | 验收矩阵、测试任务清单 | failing test / test skeleton | 高风险改动先写失败测试、最少也要先写测试骨架和待补说明 | 防止代码写完才发现根本不好测 |
| T07 | 冻结验证计划和卡点 | 测试任务清单、任务卡 | 验证计划 + 阶段卡点 | 写清每一阶段必须跑什么、什么没跑不能继续 | 防止“以为差不多可以提测” |
| T08 | 先跑 readiness 守卫 | 范围、验收、计划 | readiness 结果 | 范围冻结、验收冻结不过，不准进入实现 | 防止带着模糊需求开工 |

| 步骤 | 动作 | 输入 | 输出 | 必须做什么 | 主要防坑 |
| --- | --- | --- | --- | --- | --- |
| A01 | 冻结状态机 | 澄清结果、业务规则 | `source/01` | 明确状态集合、合法流转、非法流转 | 防止后端和前端各自理解状态 |
| A02 | 冻结权限和错误码 | 角色规则、异常规则 | `source/02`、`source/03` | 明确角色、权限键、拒绝语义、错误码语义 | 防止权限和错误处理漂移 |
| A03 | 冻结 OpenAPI 和事件契约 | 页面动作、模块入口 | `source/04`、`source/05` | 接口、字段、错误语义、事件 topic 先成文 | 防止前后端合同各写各的 |
| A04 | 冻结五层职责 | 架构方案 | `source/09` | 明确页面层/项目层/编排层/模块层/基座层职责 | 防止逻辑塞错层 |
| A05 | 冻结模块注册和表归属 | 模块方案、数据方案 | `source/10`、`source/12` | 未登记模块不开发，表归属先写清 | 防止模块职责交叉 |

| A07 | 冻结前端规范和 token SoT | 前端方案、设计方案 | `source/frontend/01`、`source/frontend/04*`、`source/frontend/05*` | 壳层、路由、页面 recipe、token 统一成文 | 防止 UI 和前端后补返工 |

### 阶段 X：按任务类型拆执行分支

这一段必须放在主流程里。  
不是所有任务都走一模一样的长链路，但所有任务都必须先过“分支判定”。

| 步骤 | 动作 | 输入 | 输出 | 必须做什么 | 主要防坑 |
| --- | --- | --- | --- | --- | --- |
| X01 | 先判任务类型 | 任务卡、范围、影响面 | 任务类型结论 | 先判断这次是后端模块、前端页面、全栈合同、编排、迁移发布、热修复、治理文档哪一类 | 防止所有任务都按同一条流水线执行 |
| X02 | 纯后端模块分支 | 只改模块/API/规则/仓储 | 后端执行分支 | 主要走 `B -> C -> E -> I -> J`，涉及跨模块再补 `D` | 防止后端任务漏掉项目隔离和合同验证 |
| X03 | 纯前端页面分支 | 只改页面/路由/组件/样式 | 前端执行分支 | 主要走 `F -> G -> H -> I -> J` | 防止前端任务跳过合同同步和权限状态 |
| X04 | 全栈 API / 合同变更分支 | OpenAPI、字段、错误语义变化 | 全栈执行分支 | 必走后端 + 前端双侧收口：`B -> C -> D/E -> F/G -> I -> J` | 防止只改生产者不改消费者 |
| X05 | 编排 / 主链 / sidecar 变更分支 | flow、event、补偿、蓝图变化 | 编排执行分支 | 强制走 `D -> I -> J`，且必须蓝图先行 | 防止编排代码先改、事实源后补 |
| X06 | 迁移 / 发布敏感分支 | migration、回填、seed、配置切换 | 发布敏感分支 | 在正常实现外，必须补 rollout、rollback、恢复路径、发布验证 | 防止能上线不能回滚 |
| X07 | 热修复 / 线上故障分支 | 紧急 bug、线上止血、快速回滚 | 热修复分支 | 可以缩小范围，但不能跳过 SoT、验证、回滚说明 | 防止紧急修复留下脏账 |
| X08 | 文档 / 治理 / 守卫分支 | 只改 source、模板、守卫、流程 | 治理执行分支 | 明确不改业务行为，重点跑治理门禁 | 防止把治理改动当普通文字修改 |
| X09 | 给分支绑定验证包 | 任务类型结论、验收矩阵 | 分支验证清单 | 每种任务类型都要写清“最小必须验证包” | 防止不知道该跑哪些检查 |

#### 按任务类型绑定的最小必跑测试与门禁

这一段也必须列在主流程里。  
如果只是写“补测试”，Codex 和人都会卡在一句话上：

- 到底跑哪个命令？
- 哪些是最小必跑？
- 哪些属于扩大回归？

下面这张表就是给执行人直接照着跑的。

| 任务类型 | 最小必跑测试与门禁 | 常见补充项 | 说明 |
| --- | --- | --- | --- |
| 纯后端模块 | `go test -count=1 ./internal/modules/<module>/...`、`bash tests/common/module_import_boundary_guard.sh`、`bash tests/common/module_cycle_guard.sh`、`bash tests/common/module_structure_guard.sh` | `bash tests/common/module_legacy_style_guard.sh`、`bash tests/common/source10_nm_scaffold_guard.sh`、`bash tests/common/order_cross_project_exception_guard.sh` | 模块改动先跑模块测试，再跑结构守卫 |
| 纯前端页面 | `corepack pnpm --dir frontend sync:contracts`、`make frontend-governance-check`、`bash tests/common/frontend_action_route_api_alignment_guard.sh`、`make frontend-typecheck`、`make frontend-test` | `make frontend-build`、`make frontend-responsive-check`、`make frontend-bundle-budget-check`、`make frontend-check-fast` | 页面、路由、权限、API 对齐至少要过一遍 |
| 全栈 API / 合同变更 | 后端模块测试 + 前端合同同步；再跑 `bash tests/common/contract_field_drift.sh`、`bash tests/common/contract_path_drift.sh`、`bash tests/common/source_contract_smoke.sh` | `make frontend-check-fast`、`make mainline-check` | 这类任务最容易只改生产者不改消费者，合同漂移必须单独测 |
| 编排 / 主链 / sidecar | `bash tests/common/flow_blueprint_guard.sh`、`bash tests/common/flow_topic_consistency_guard.sh`、`bash tests/common/mainline_contract_anchor_guard.sh`、`make mainline-check` | `bash tests/common/orchestration_module_boundary_guard.sh`、`bash tests/common/orchestration_project_boundary_guard.sh`、`bash tests/common/orchestration_platform_boundary_guard.sh` | 编排任务必须同时验证蓝图、topic、anchor 和主链 |
| 项目隔离 / 路由 / 权限 | `bash tests/common/source07_project_routing_guard.sh`、`bash tests/common/project_sot_consistency_guard.sh`、`bash tests/common/orchestration_project_boundary_guard.sh` | `bash tests/common/mainline_project_strategy_guard.sh`、`bash tests/common/page_orchestration_alignment_guard.sh` | 这类任务高风险，`project_id`、路由、权限不能只靠人工看 |
| 迁移 / 发布敏感 | `make readiness-check TASK_CARD=<path>`、`make delivery-check TASK_CARD=<path> FULL=1`、`make governance-check` | `make pre-commit`、`make source07-release-log RECORD=...` | 重点不只是代码能跑，而是能放、能回、能审 |
| 热修复 / 线上故障 | 定向测试命令 + `make stage-check TASK_CARD=<path>` + `make pre-commit` | `make delivery-check TASK_CARD=<path>`、受影响专项守卫 | 热修复可以缩范围，不能缩证据 |
| 文档 / 治理 / 守卫 | `make governance-check`、`bash tests/common/governance_semantic_guard.sh`、`bash tests/common/acceptance_id_uniqueness_guard.sh`、`bash tests/common/sot_candidate_guard.sh` | `bash tests/common/source13_platform_baseline_guard.sh`、`bash tests/common/source23_product_board_guard.sh` | 治理改动也要测试，不是改文档就等于无风险 |

再记一条最简单的人话规则：

1. 先跑最小必跑项。
2. 只要改动跨层、跨合同、跨项目，就补常见补充项。
3. 要发布时，再统一跑 `I` 阶段的总门禁。

### 阶段 B：后端分层设计

| 步骤 | 动作 | 输入 | 输出 | 必须做什么 | 主要防坑 |
| --- | --- | --- | --- | --- | --- |
| B01 | 先判定本次改动属于哪一层 | 任务卡、SoT | 变更分层清单 | 先归层再开发，不能一上来直接写代码 | 防止大杂烩式实现 |
| B02 | 定项目层职责 | `project_id`、路由策略 | 项目层设计说明 | 明确项目路由、项目差异、只读策略入口 | 防止项目逻辑到处散落 |
| B03 | 定编排边界 | 主链步骤、事件、补偿 | 编排边界清单 | 明确主链、sidecar、blocking/non-blocking | 防止编排越界或支链阻断主链 |
| B04 | 定模块边界 | 模块清单、表归属 | 模块职责说明 | 明确单一职责、公开契约、私有实现边界 | 防止模块互相穿透 |
| B05 | 定公共基座边界 | 响应、审计、鉴权、幂等 | 基座能力清单 | 基座只提供能力，不承载业务判断 | 防止基座变“第二业务层” |
| B06 | 先列后端最小改动路径 | 变更分层清单 | 实现顺序表 | 先模块、后编排、再项目层收口 | 防止顺序错导致返工 |

### 阶段 C：后端脚手架、模块编写、测试先行落地

| 步骤 | 动作 | 输入 | 输出 | 必须做什么 | 主要防坑 |
| --- | --- | --- | --- | --- | --- |
| C00 | 先补模块测试骨架 | 验收矩阵、模块边界 | 模块测试骨架 | 至少先把正常/异常/边界/权限用例位子占住 | 防止模块写完后没地方补测 |
| C01 | 创建模块脚手架 | `source/10`、模块名 | `api/dto/domain/repo` 目录 | 固定用当前项目脚手架，不走旧 `entity/mapper/service/controller` | 防止旧风格回流 |
| C02 | 在 `dto` 冻结请求/响应模型 | `source/04`、`source/03` | DTO 与校验入口 | 先按合同写模型和校验 | 防止页面模型反推后端模型 |
| C03 | 在 `domain` 写业务规则 | `source/01`、`source/06` | domain 规则实现 | 状态机、默认值、约束都放 domain | 防止业务规则散在 API 或 repo |
| C04 | 在 `repo` 写唯一 DB 出口 | 表结构、数据字典 | repo 实现 | 所有查询写入都只从 repo 走 | 防止编排/API 直查库 |
| C05 | 在 `api` 暴露接口 | `source/04` | 模块 API 入口 | API 层只负责接参、调用、返回统一包 | 防止 API 层偷写业务 |
| C06 | 先补 `project_id` 和权限校验 | `source/07`、`source/02` | 项目隔离和鉴权逻辑 | 表、查询、请求、会话、权限同时带 `project_id` | 防止功能通了再补隔离 |
| C07 | 及时补 `small` 子模块和深规则 | 模块实现、扩展动作 | `small/*`、深规则 | 大模块下细动作必须有小模块和规则白名单 | 防止脚手架过度泛化 |
| C08 | 跑模块定向测试 | 模块代码、测试骨架 | 模块测试证据 | 每做完一个模块就跑最小自动化验证 | 防止最后一把梭测不动 |
| C09 | 跑结构边界守卫 | 模块代码 | 守卫通过证据 | 跑结构、循环依赖、旧风格回流守卫 | 防止代码结构写歪后期难救 |

### 阶段 D：编排设计与落地

| 步骤 | 动作 | 输入 | 输出 | 必须做什么 | 主要防坑 |
| --- | --- | --- | --- | --- | --- |
| D00 | 先判断是主链还是支链 | 页面动作、业务流 | 编排归类 | 先判定 blocking / non-blocking / sidecar | 防止非阻断支链把主链卡死 |
| D01 | 先改编排蓝图 | `source/13`、`source/13-flows` | 蓝图变更 | 先改蓝图再改代码 | 防止代码先跑、文档追不上 |
| D02 | 绑定 `ACT-*` 到 `OR-*` | 页面动作、蓝图 | `ACT -> OR` 映射 | 每个动作明确归属编排组和步骤 | 防止动作编号失联 |
| D03 | 在编排层接模块契约 | 模块 API/Event | 编排执行入口 | 编排只消费模块公开契约和事件 | 防止编排直接调私有 repo/domain |
| D04 | 明确补偿、重试、回放 | 蓝图、异常路径 | 编排补偿设计 | 失败怎么办、是否重试、如何回放必须先写 | 防止联调后才补异常链 |
| D05 | 跑编排专项守卫 | 蓝图、代码 | 编排验证证据 | 跑 blueprint、topic consistency、anchor 守卫 | 防止蓝图和代码漂移 |
| D06 | 留编排 strict-change 记录 | 验收、验证结果 | 编排记录 | 写清 SoT、风险、回滚、未执行项 | 防止高风险变更无证据 |

### 阶段 E：项目隔离、权限、路由策略

| 步骤 | 动作 | 输入 | 输出 | 必须做什么 | 主要防坑 |
| --- | --- | --- | --- | --- | --- |
| E00 | 先设计项目隔离链路 | `source/07`、会话策略 | 项目隔离方案 | 先画出表、API、session、route、前端上下文的 `project_id` 链路 | 防止后补 where 条件 |
| E01 | DB 层带 `project_id` | 表结构、repo 查询 | 数据隔离实现 | 业务表和查询必须有 `project_id` 过滤 | 防止串项目数据 |
| E02 | API 层二次校验项目归属 | 请求参数、登录态 | API 项目归属校验 | 后端不能只信前端传值 | 防止越权访问 |
| E03 | 动作和权限键对齐 | `ACT-*`、RBAC | 动作权限矩阵 | 可见、禁用、拒绝态必须统一 | 防止前后端权限不一致 |
| E04 | 项目层保持只读 | 路由和策略实现 | 干净项目层 | 项目层不承载业务流程 | 防止项目层变业务层 |
| E05 | 跑项目路由和隔离守卫 | route、project SoT | 守卫通过证据 | 跑项目路由、项目 SoT 一致性守卫 | 防止发布后才发现串项目 |

### 阶段 F：前端编排、壳层、脚手架准备

| 步骤 | 动作 | 输入 | 输出 | 必须做什么 | 主要防坑 |
| --- | --- | --- | --- | --- | --- |
| F00 | 先列前端准备清单 | 页面清单、前端规范 | 前端准备清单 | 先准备 `routeConfig`、`actionPolicy`、`permissionScopePolicy`、`errorPolicy` 等 | 防止以为“起个 Vite 就算脚手架” |
| F01 | 冻结前端目录六件套 | `source/frontend/01` | 模块目录骨架 | 固定 `api/pages/components/router/stores/utils` | 防止目录越写越散 |
| F02 | 先做 App Shell | 页面域、导航分组 | `AppShell` | 先做主壳层、导航、top bar、内容区 | 防止页面铺开后大返工 |
| F03 | 冻结导航和页面容器 | `source/11`、route specs | 导航结构、页面壳容器 | 一级导航、二级导航、页面容器先成型 | 防止信息架构后补 |
| F04 | 先做页面 scaffold | 页面类型、UI recipe | `AppPageShell`、`ActionCreatePageScaffold` 等 | 列表页、表单页、详情页、工作台先有共享骨架 | 防止每页各写各的 |
| F05 | 冻结 token/theme SoT | Stitch 提取结果、设计规则 | token 产物、`theme.css` | 样式变量先统一，不准页面硬编码颜色/尺寸 | 防止样式系统失控 |

### 阶段 G：前端动作对齐、合同同步、页面实现

| 步骤 | 动作 | 输入 | 输出 | 必须做什么 | 主要防坑 |
| --- | --- | --- | --- | --- | --- |
| G00 | 先补消费者合同测试骨架 | OpenAPI、页面动作 | 合同同步清单 | 先确认哪些页面、API、类型会受影响 | 防止前端被动追后端改动 |
| G01 | 先确认页面动作 `ACT-*` | `source/11` | 动作清单 | 页面先有动作编号，再写实现 | 防止页面先写后补动作 |
| G02 | 动作对齐编排组 `OR-*` | `source/13` | `ACT -> OR` 关系 | 页面知道自己触发哪个编排步骤 | 防止页面只知道 API 不知道业务流 |
| G03 | 同步消费者合同 | OpenAPI、components | consumer contracts | 先 `sync:contracts`，再动前端 API | 防止手写类型漂移 |
| G04 | 写模块 API 层 | contracts、接口路径 | `modules/<module>/api/*` | 页面禁止直接 `fetch` | 防止请求分散到页面里 |
| G05 | 对齐路由、动作、权限键 | routeConfig、actionPolicy、RBAC | 三向对账结果 | 路由显隐、动作显隐、权限校验必须一致 | 防止按钮能看不能点、能点不能过 |
| G06 | 页面层只做展示和触发 | 页面骨架、模块 API | 页面实现 | 页面不写业务规则、错误语义、权限裁决 | 防止页面越来越厚 |
| G07 | 补全页面状态 | 页面实现、验收矩阵 | 加载态/空态/错误态/禁用态/拒绝态 | 页面不只做 happy path | 防止联调时才发现状态不全 |
| G08 | 跑前端对齐守卫 | 页面、路由、API | 守卫通过证据 | 跑动作-路由-API 对齐守卫 | 防止多端长期漂移 |

### 阶段 H：UI 设计与 Stitch 落地

| 步骤 | 动作 | 输入 | 输出 | 必须做什么 | 主要防坑 |

| H00 | 先定壳层框架 | 页面域、使用频次 | 框架草图 | 先定一级导航、二级导航、头部、内容区 | 防止 UI 从局部页面反推整体结构 |
| H01 | 先定导航信息架构 | 页面清单、角色差异 | 导航结构图 | 导航按业务域，不是按代码目录 | 防止导航混乱 |
| H02 | 先定内容区 recipe | 页面类型 | `overview/list/form/detail/workspace` recipe | 统一内容区布局和页面层级 | 防止每页自己发明布局 |
| H03 | 用 Stitch 提取 token 和布局规则 | 设计稿、视觉规范 | token SoT、layout recipe 命名 | Stitch 负责把设计变量工程化 | 防止设计稿只停留在图里 |
| H04 | 同步组件 token 覆盖 | token SoT、组件库 | 组件 token 覆盖清单 | 高颜色、高频尺寸、间距、圆角都走 token | 防止页面回流硬编码 |

| H06 | 最后做页面精修 | 真实页面、验收反馈 | 视觉精修结果 | 精修间距、响应式、空态、文案、细节态 | 防止把视觉精修误当整体架构设计 |

### 阶段 I：联调、回归、UAT、发布前证据

| 步骤 | 动作 | 输入 | 输出 | 必须做什么 | 主要防坑 |
| --- | --- | --- | --- | --- | --- |
| I00 | 每次有意义改动都跑最小验证 | 本次改动 | 局部验证证据 | 模块改模块测，前端改前端快检，编排改编排守卫 | 防止问题堆到最后一起炸 |
| I01 | 跑模块定向测试 | 后端模块代码 | 后端局部证据 | 正常、异常、边界、权限至少覆盖到本次改动 | 防止只测 happy path |
| I02 | 跑编排主链验证 | 编排代码、蓝图 | 主链回归证据 | 主链、sidecar、topic、anchor 分开验证 | 防止主链被旁路改坏 |
| I03 | 跑前端快速门禁 | 页面/API/路由代码 | 快速验证证据 | typecheck、关键单测、基础 build 快检 | 防止前端小错积累 |
| I04 | 跑前端完整门禁 | 浏览器环境、前端实现 | 完整前端证据 | E2E、build、bundle、治理门禁 | 防止发布前只做局部验证 |
| I05 | 跑治理总门禁 | source、代码、守卫脚本 | 治理验证证据 | 契约、结构、项目路由、source 对齐都要检查 | 防止“代码能跑但流程坏了” |
| I06 | 做 UAT 交接包 | 验收矩阵、环境信息 | UAT 手册 | 写环境、入口、账号、步骤、预期结果、反馈路径 | 防止提测后没人会验 |
| I07 | 收口发布证据包 | 所有验证结果 | 发布证据包 | 收敛 what ran / what not run / remaining risk / rollback | 防止只留零散聊天记录 |
| I08 | 跑 stage-check | 任务卡、验证证据 | readiness/delivery 结论 | 没有足够证据，不准进入下一阶段 | 防止拍脑袋放行 |

### 阶段 J：strict-change、发布、回滚、沉淀

| 步骤 | 动作 | 输入 | 输出 | 必须做什么 | 主要防坑 |
| --- | --- | --- | --- | --- | --- |
| J00 | 先写变更摘要 | 范围、验证结果 | 变更摘要 | 写清做了什么、没做什么、影响范围 | 防止交接含糊 |
| J01 | 高风险变更写 strict-change | 任务范围、门禁结果 | `docs/records/*strict-change*.md` | SoT、验收、风险、回滚、未执行项都要写 | 防止高风险改动不可追溯 |
| J02 | 涉及 `source/07` 必回写发布台账 | 路由变更、策略变更 | release log | 路由、版本、hash、record 形成台账 | 防止项目路由改动失控 |
| J03 | 提交前跑 pre-commit | 代码、文档、脚本 | 可提交状态 | 不靠人记，靠 git 钩子阻断低级错误 | 防止漏跑关键检查 |
| J04 | 发布前核对 acceptance -> verification | 验收矩阵、测试结果 | 放行结论 | 每条验收都要有对应证据 | 防止“测试过了但不知验收过没过” |
| J05 | 明确 rollout、rollback、诊断入口 | 版本、模块、监控 | 发布/回滚说明 | 上线步骤、回滚动作、日志入口、告警入口都要写 | 防止出故障不会退 |
| J06 | 补性能、容量、可观测性说明 | 发布证据、监控信息 | 非功能说明 | 没做压测也要明确写未做和风险 | 防止上线后性能/诊断失控 |
| J07 | 把新坑回写进流程 | 本次反馈、review 结论 | 流程更新记录 | 重复出现的坑必须反哺流程、模板、守卫 | 防止同类问题一再复发 |

## 4. 后端这条线，应该怎么傻瓜式执行

### 4.1 后端总顺序

1. 先完成 `T` 阶段：需求澄清、验收矩阵、测试计划。
2. 再完成 `A` 阶段：状态、权限、错误码、OpenAPI、模块登记。
3. 再做 `B` 阶段：先判层，再定模块/编排/项目边界。
4. 再做 `C` 阶段：测试骨架 -> 脚手架 -> DTO -> domain -> repo -> api -> `project_id` -> 模块测试。
5. 如果涉及跨模块，再做 `D` 阶段：蓝图 -> 编排代码 -> 编排守卫。
6. 最后做 `E/I/J`：项目隔离、总验证、strict-change、发布说明。

### 4.2 小模块固定写法

固定顺序只有这一条：

1. `dto`
2. `domain`
3. `repo`
4. `api`
5. `tests`

不要再按旧习惯写成：

- `entity`
- `mapper`
- `service`
- `controller`

也不要先把接口跑通，再回头补：

- 状态机
- 权限拒绝
- `project_id`
- `smallModule`
- 测试

这些都必须跟实现同步出现。

### 4.3 大模块和编排怎么分

大模块负责：

- 业务域能力边界
- 小模块收口
- 本域数据和规则稳定

编排负责：

- 跨模块顺序
- 事件发布/消费
- 补偿/重试/回放
- 主链和 sidecar 关系

记住一句人话：

- 模块像“一个部门把自己事情做好”。
- 编排像“跨部门把顺序串起来”。
- 编排不是“更大的模块”。

## 5. 前端这条线，应该怎么傻瓜式执行

### 5.1 前端总顺序

1. 先准备政策和壳层，不先写业务页。
2. 先做 `AppShell`、导航、页面 scaffold。
3. 先同步 contracts，再写 API 层。
4. 再写页面，只负责展示、触发、状态。
5. 最后补权限态、错误态、空态、精修和 E2E。

### 5.2 页面开发固定顺序

1. 页面动作 `ACT-*`
2. 编排组 `OR-*`
3. OpenAPI / contracts
4. 模块 API 层
5. 权限键 / route
6. 页面骨架
7. 页面状态
8. 页面精修

如果跳过前四步直接写页面，后面一般一定返工。

### 5.3 开始铺页面前，前端至少要先准备好

- `routeConfig`
- `actionPolicy`
- `permissionScopePolicy`
- `errorPolicy`
- `listQueryProtocol`
- `formSubmitPolicy`
- `telemetryPolicy`
- `uiAccessibilityPolicy`
- `AppShell`
- `AppPageShell`
- `ActionCreatePageScaffold`
- `theme.css`
- contracts 同步命令

这才叫前端脚手架。  
不是说建个 `frontend/` 目录就算开始了。

## 6. UI 和 Stitch 这条线，应该怎么傻瓜式执行

### 6.1 UI 正确顺序

1. 定壳层框架
2. 定导航结构
3. 定内容区 recipe
4. 定页面骨架
5. 定 token
6. 定主题覆盖
7. 最后做页面精修

### 6.2 Stitch 的正确位置

Stitch 最适合做的是：

- 从设计稿提 token
- 提炼布局 recipe
- 同步到 `theme.css`
- 给前端提供可复用的样式和布局事实源

Stitch 不适合做的是：

- 跳过壳层和路由，直接决定页面业务结构
- 直接生成最终业务逻辑页面
- 替代前端合同、权限、状态设计

一句话：

- Stitch 是“设计到代码”的桥。
- 不是“设计替前端做架构”的工具。

## 7. 把真实踩坑固定进流程后的对照表

| 真实踩坑 | 现在固定到哪一步 | 不做会卡在哪 |
| --- | --- | --- |
| 需求没问清就开做 | `T01-T04` | readiness 不过 |
| 没有测试先行 | `T05-T07`、`C00`、`G00` | 开发阶段证据不完整，提测卡住 |
| 任务类型没先判 | `X01-X09` | 走错分支，漏验证或做过头 |
| 后端写成旧四层 | `C01`、`C09` | 模块结构守卫不过 |
| 功能通了才补 `project_id` | `E00-E03`、`C06` | 项目隔离守卫不过，属于高风险 blocker |
| 编排直接直查库/调私有实现 | `D03`、`D05` | 编排 review 和 guard 卡住 |
| 页面直接写 `fetch` | `G03-G04` | 合同漂移，前端对齐守卫不过 |
| 路由/权限/动作不一致 | `G05`、`G08` | 联调和权限验收卡住 |
| 先堆页面，后补壳层和导航 | `F02-F04`、`H00-H02` | 大量返工 |
| 设计稿只有图，没有 token SoT | `H03-H04`、`H07` | 样式难复用，token 校验不过 |
| 只做 happy path | `T04`、`C08`、`G07`、`I01-I04` | 验收和 UAT 卡住 |
| 提交/发布只说“测过了” | `I07`、`J01`、`J04-J06` | handoff/release 结论不成立 |
| 同类问题反复出现 | `J07` | 流程不会越来越稳 |

## 8. 真正落地时，最简单的执行顺序

如果你是小白，只想照表抄，最简单就记这一条：

1. 先做 `T`：需求澄清、验收、测试计划。
2. 再做 `A`：唯一事实源。
3. 再做 `X`：先判断本次是哪类任务。
4. 后端类任务走 `B-C-D/E`。
5. 前端类任务走 `F-G-H`。
6. 最后统一做 `I-J`：联调、发布、回滚、沉淀。

不要反过来。

最容易出事的反顺序是：

1. 先写页面
2. 再补接口
3. 再补权限
4. 再补测试
5. 最后补文档

这基本就是返工套餐。
