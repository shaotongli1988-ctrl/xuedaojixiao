# 核心模块修改前检查

这份清单给团队成员和 AI 在动手前快速过一遍。

它解决的不是“怎么写代码”，而是“这次能不能直接改、应该先看什么、要不要先停一下”。

项目进入长期维护阶段后，真正容易出问题的，通常不是普通页面，而是核心模块、公共结构、主流程和高风险链路。

所以这份文档要短、要实用、要能直接照着做。

---

## 一、哪些算核心模块

下面这些，一般都应该按核心模块看待。

### 1. 登录、鉴权、权限

例如：

- 登录流程
- token / session 处理
- 角色权限
- 菜单权限
- 接口权限
- 数据权限

### 2. 主业务流程

例如：

- 下单
- 支付
- 审批
- 结算
- 发货
- 同步
- 提交流程

只要一个流程一改就会影响多个页面、多个角色、多个端，就应该按核心模块处理。

### 3. 主数据和公共数据结构

例如：

- 用户
- 订单
- 商品
- 课程
- 组织
- 配置
- 状态枚举
- 公共返回结构

### 4. 公共能力和公共结构

例如：

- 公共组件
- 公共 hooks / composables
- 公共 service
- 公共工具函数
- 请求封装
- 错误处理
- 全局拦截器
- 路由守卫
- 公共表单或表格基类

### 5. 会影响多个端的部分

例如：

- 后端接口结构
- 多端共用字段
- 登录态
- 文件上传
- 消息通知
- 统一配置中心

如果你不确定它算不算核心模块，可以先按“可能是”处理，而不是先按普通改动处理。

---

## 二、修改前必须先确认什么

### 1. 先确认这次改动影响谁

至少先回答下面几个问题：

- 影响哪个端
- 影响哪个模块
- 会不会影响其他角色
- 会不会影响其他页面或接口
- 会不会影响历史数据、状态流转、权限判断

如果这些问题还答不上来，就不要急着改。

### 2. 先确认当前项目有没有已经定制过

尤其是基于 base 发展出来的项目，先确认：

- 当前实现是不是已经偏离 base
- 这种偏离是业务需要，还是临时兼容
- 这次修改应该参考当前项目，还是参考 base

只要这里判断错了，后面代码很可能越改越偏。

### 3. 先确认有没有现成文档

改核心模块前，至少先查：

- 项目文档入口
- 项目执行流程
- 当前模块相关文档
- 开发前检查清单

如果这些文档已经存在，不要跳过。

### 4. 先确认是不是公共结构

有些改动看起来只是改一个地方，其实改的是公共入口。

比如：

- 一个公共表单组件
- 一个公共请求方法
- 一个统一权限判断
- 一个全局状态字段

这种情况不能只看当前调用点，要先看引用范围。

### 5. 先确认风险够不够高

如果涉及下面任意一种，默认按高风险处理：

- 核心流程
- 权限
- 数据结构
- 状态流转
- 多端联动
- 公共组件或公共方法
- 外部依赖
- 上线后难回退的改动

## 三、哪些情况下不要直接改代码

下面这些情况，建议先停一下，不要直接下手。

### 1. 你还没看清主流程

如果只知道“改哪个文件”，但不知道整个流程怎么走，不要直接改。

### 2. 当前项目和 base 明显不一样，但原因不清楚

这种时候最危险。

你看到不一致，不代表当前项目写错了，也可能是业务定制。

### 3. 不知道这是局部逻辑还是公共逻辑

如果你还没确认这个方法、组件、字段是不是被多个地方共用，不要直接改。

### 4. 改动会影响接口、数据库或状态流转

这类改动一旦做错，通常不是改回一行代码就能解决的。

### 5. 相关文档明显缺失或过期

如果系统说明、差异记录、模块文档明显对不上当前代码，先补理解，再决定要不要开发。

### 6. 任务描述太短，但影响看起来很大

比如一句“把权限逻辑调整一下”“把订单流程改一下”。

这种需求表述很短，但风险非常高，不能按普通小修复对待。

## 四、需要先看哪些文档

推荐最少按这个顺序看。

### 第一步：项目入口文档

先看：

- `performance-management-system/docs/README.md`
- `performance-management-system/docs/17-项目执行流程（精简版）.md`

目的：先确认本项目唯一事实源和当前执行顺序，不要直接陷入局部代码。

### 第二步：模块文档

看当前模块对应的：

- 需求说明
- 技术设计
- API 文档
- 数据库文档
- 开发实施文档

目的：知道这个模块原本怎么设计、为什么这么实现。

### 第三步：检查清单

看：

- `docs/checklists/01-开发前检查.md`
- 本文档 `docs/checklists/核心模块修改前检查.md`

目的：在真正改代码前做最后一次收口。

## 五、短 checklist

下面这份短清单，适合开工前快速过一遍。

```md
## 核心模块修改前 8 项检查

- [ ] 我已经确认这次改动涉及的端、模块、角色和主流程
- [ ] 我已经确认它是不是核心模块或公共结构
- [ ] 我已经看过项目入口和执行流程，知道当前唯一事实源和阶段顺序
- [ ] 我已经看过当前模块相关文档，不是只凭文件名猜实现
- [ ] 我已经确认这次改动是否影响接口、数据库、状态流转或权限
- [ ] 如果当前信息不足，我会先补检查或补文档，不直接写代码
- [ ] 我知道改完后哪些文档需要同步更新
```

## 六、给团队和 AI 的实用建议

### 建议 1：先判断，再开发

长期维护项目里，很多错误不是“代码写错”，而是“改错地方”。

所以先判断模块属性、影响范围、参考依据，往往比直接写代码更重要。

### 建议 2：核心模块任务默认不是小任务

哪怕需求看起来只是一句话，只要碰核心模块，就不要按普通小修复处理。

### 建议 3：先恢复项目理解，不要只靠临时上下文

AI 特别容易因为当前对话太短，只抓住表面需求。

所以遇到核心模块修改，必须先恢复：

- 系统全貌
- 差异背景
- 模块设计
- 当前项目真实现状

### 建议 4：当前项目优先，base 其次

如果当前项目已经偏离 base，不要机械按 base 回退。

应该先理解为什么偏离，再决定是继续沿用、局部整理，还是真的要回归模板。

---

## 七、一句话记住

**改核心模块前，先恢复理解、先做检查、先确认依据；不要只因为“需求来了”就直接改代码。**
