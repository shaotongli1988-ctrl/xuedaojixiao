# 2026-04-19 目标&计划首版目标运营台设计

## 背景

`目标&计划` 已在导航层成为独立一级业务域，但当前实际承载页仍复用 `/performance/goals` 的目标地图页面。用户本轮进一步明确，希望把该模块从“目标列表页”升级为“部门日运营目标系统”：

1. 每天上午开会制定工作计划和目标。
2. 每天下午下班前填写对应实际数据。
3. 系统按日进行统计、横向同事对比和纵向个人对比。
4. 每天下午自动形成部门工作汇报。
5. 员工个人目标需要量化，并与自己的实际结果对比。

本设计只负责把首版闭环和页面结构冻结清楚，不直接进入代码实现。

## 设计目标

首版 `目标&计划` 固定按“目标运营台”方向设计，而不是传统计划表或单纯统计看板。唯一主线是：

`规则配置 -> 目标下发 -> 员工回填 -> 自动统计 -> 自动日报 -> 留痕推送`

系统必须同时服务三类角色：

1. 员工：查看目标、填写当日实际完成值、补充个人目标。
2. 主管：配置部门参数、下发周/月/日目标、查看团队对比和日报。
3. HR/管理员：配置公司级基础规则和模板底座，做跨部门管理。

## 本轮冻结结论

### 首版范围

1. 同时覆盖员工录入和主管汇总双端闭环。
2. 时间粒度固定为 `日录入，周/月汇总`。
3. 上午目标由主管下发，支持预填，减少重复输入。
4. 下午员工只填各目标项的实际完成数值。
5. 目标来源固定为 `部门公共模板 + 员工个人补充目标`。
6. 横向对比范围固定为同部门人员，且普通员工可见实名榜单。
7. 横向对比口径当前按 `公共目标 + 个人补充目标都进入统计`。
8. 纵向对比同时覆盖 `自己和自己比` 与 `自己和目标比`。
9. 当天截止前未填写的数据按 `0` 记入，不保留“待补填”状态。
10. 周目标和月目标由主管先设定，系统拆为每日建议值。
11. 排行展示固定为 `完成率榜 + 绝对产出榜` 双榜并存，不强行合成单一总分榜。
12. 部门日报由系统自动生成，并同时支持系统内留痕和群推送。
13. 日报默认自动发送，但主管可在发送前拦截或延期。
14. 部门的下发、填报和发送时间点由部门自行配置。
15. 基础规则由 HR/管理员配置，主管可调整本部门参数。
16. 员工新增的个人目标无需主管确认，直接进入统计和对比。
17. 部门公共目标不可由员工修改，但员工可维护自己的个人补充目标。
18. 同部门横向对比首版不按岗位或模板分组，直接出全部门统一榜单。

### 非目标

1. 本阶段不进入代码实现、接口开发和数据库建模。
2. 不改现有导航 IA 结论，不新增一级菜单。
3. 不把 `班主任化` 合并进目标运营台，不动其现有二级目录。
4. 不在首版引入复杂审批流、补填流、申诉流和人工校正流。
5. 不扩展到跨部门对比、匿名排行、岗位维度分组对比。
6. 不在首版做自动评分模型、复杂积分引擎和 AI 目标建议。
7. 不绑定真实外部消息通道实现细节，只冻结“系统内 + 群推送”产品口径。

## 业务定位

`目标&计划` 首版不应被理解为员工个人待办工具，也不只是原有 `目标地图` 的增强版。它的定位固定为：

`围绕部门日经营节奏运转的目标运营台`

其管理动作是：

1. 主管根据周/月经营目标，每天上午向部门和员工下发日目标。
2. 员工在下午截止前回填当天实际完成值。
3. 系统自动完成统计、排行、趋势和目标差异分析。
4. 系统自动生成部门日报并完成留痕和推送。

## 模块分层

首版按 5 层职责拆分，避免总览页变成无边界的大杂烩。

### 1. 规则层

给 HR/管理员使用，负责：

1. 公司级目标模板底座。
2. 默认时间节点。
3. 默认日报推送方式。
4. 统计展示基础口径。

不负责每天的部门运营动作。

### 2. 部门运营层

给主管使用，负责：

1. 本部门参数调整。
2. 本部门模板和周/月目标维护。
3. 每天目标生成与下发。
4. 日报拦截、延期和结果确认。

这是首版的主管主入口。

### 3. 员工执行层

给员工使用，负责：

1. 查看主管下发的当天目标。
2. 补充个人目标项。
3. 填写每个目标项的实际完成值。
4. 查看自己的目标达成和历史趋势。

不负责部门级汇总和日报配置。

### 4. 统计对比层

由系统自动完成，负责：

1. 日/周/月汇总。
2. 完成率榜和绝对产出榜。
3. 个人历史趋势。
4. 目标差异分析。
5. 个人、团队、部门多视角统计快照。

### 5. 汇报发布层

由系统自动生成，主管可拦截，负责：

1. 自动汇总部门日报。
2. 在系统内留痕。
3. 向外部群推送。
4. 记录发送时间、状态和拦截动作。

## 页面结构

### 导航结构

首版继续服从当前导航 IA：

1. 一级入口：`目标&计划`
2. 二级入口只保留：
   - `目标&计划总览`
   - `班主任化`

其中 `目标&计划总览` 是目标运营台的真实承载入口，当前可继续复用 `/performance/goals`。

### 总览页结构

`目标&计划总览` 首版不再拆很多二级菜单，而是在一个总览页内做 4 个工作区。

#### 1. 今日运营区

面向日闭环的核心动作区。

主管看到：

1. 今日目标是否已下发。
2. 已填报人数、未填报人数。
3. 今日完成率榜。
4. 今日绝对产出榜。
5. 当日日报生成和发送状态。

员工看到：

1. 今日主管下发的目标。
2. 截止时间。
3. 已填写和未填写状态。
4. 个人快捷填报入口。

#### 2. 目标对比区

面向统计分析的核心区。

1. 横向对比：同部门实名榜，双榜并列。
2. 纵向对比：今日/昨日、本周/上周、本月/上月趋势。
3. 目标对比：个人和团队的实际值 vs 目标值差异。

主管看团队视角，员工看个人视角，同页按角色裁剪。

#### 3. 目标配置区

面向规则和目标制定。

1. HR/管理员看到公司级基础规则与基础模板。
2. 主管看到本部门参数、本部门模板、周/月目标和日报推送群。
3. 员工不承担此区主要配置动作。

#### 4. 日报汇报区

面向下午汇报结果交付。

1. 今日部门日报草稿。
2. 异常人员与记 0 人员列表。
3. 周/月目标进度摘要。
4. 发送记录。
5. 拦截、延期、重发入口。

### 员工填报承载策略

首版不强制单独做复杂的员工专属页面。建议：

1. 在总览页内提供显著的 `我的今日目标` 卡片。
2. 通过抽屉、弹层或轻量分区承载员工当日填报。
3. 后续仅在员工使用频率被验证足够高时，再独立成专门页面。

这样可以优先保证主管运营闭环先跑通，而不是过早拆出大量低复用页面。

## 日常数据闭环

首版按 7 步固定业务流程。

### 1. 基础规则准备

1. HR/管理员配置公司级基础模板和默认规则。
2. 主管配置本部门时间点、推送群和部门目标模板。

### 2. 周/月目标下发

1. 主管先定义周目标和月目标。
2. 系统将周/月目标拆出每日建议值，作为日目标预填基础。

### 3. 当日目标生成与下发

1. 到部门设定时间后，系统生成当日目标草稿。
2. 主管可确认或微调。
3. 正式下发后，公共目标值固定，由主管控制。

### 4. 员工下午回填实际值

1. 员工在截止时间前填每个目标项的实际完成数值。
2. 员工可新增和维护自己的个人目标项。
3. 个人目标项无需主管确认，立即进入统计。
4. 截止未填的目标项按 `0` 入账。

### 5. 系统自动统计

系统自动形成：

1. 今日完成率榜。
2. 今日绝对产出榜。
3. 个人实际 vs 个人目标。
4. 今日/昨日、本周/上周、本月/上月趋势。
5. 团队和部门汇总数据。

### 6. 系统自动生成部门日报

日报内容至少包含：

1. 部门目标达成摘要。
2. 完成率榜和绝对产出榜摘要。
3. 异常人员与记 0 人员。
4. 周/月目标进度。

### 7. 发送与留痕

1. 正常情况下系统自动发送日报。
2. 主管可在发送前拦截或延期。
3. 一份日报保存在系统内可追溯。
4. 一份日报推送到部门群或指定群。

## 角色边界

### 员工

可以：

1. 查看自己被下发的目标。
2. 录入实际完成值。
3. 新增和维护个人补充目标。
4. 查看本部门实名横向榜。
5. 查看个人纵向趋势和目标差异。

不能：

1. 修改部门公共目标值。
2. 配置部门规则。
3. 拦截或发送日报。

### 主管

可以：

1. 配置本部门参数。
2. 制定周/月目标。
3. 下发和微调日目标。
4. 查看团队全量统计与实名榜单。
5. 拦截、延期和重发日报。

### HR/管理员

可以：

1. 配置公司级基础规则和模板底座。
2. 跨部门查看和治理规则一致性。
3. 做基础参数平台的维护者。

本阶段不要求 HR/管理员承担每天的部门运营动作。

## 关键对象建议

首版设计阶段建议把数据模型收敛到 6 类核心对象，后续实现时也应尽量围绕它们展开。

### 1. 目标模板

用于描述部门公共模板和可补充目标的基础定义。

### 2. 目标计划单

用于描述某天、某周或某月发给谁、包含哪些目标项和目标值。

### 3. 目标项

最小目标单位。必须显式区分来源：

1. 部门公共目标
2. 个人补充目标

### 4. 结果填报

记录员工在具体日期对具体目标项填写的实际完成值。

### 5. 统计快照

记录特定时间粒度下的完成率、产出值、排名和趋势结果。

### 6. 日报记录

记录日报内容、发送状态、推送对象、拦截动作和发送时间。

## 时间节点规则

首版时间节点固定为 `部门可配置`，至少覆盖三类时间：

1. 当日目标草稿生成或下发时间。
2. 员工填报截止时间。
3. 日报自动发送时间。

部门级时间优先于公司默认时间。HR/管理员配置默认值，主管维护本部门参数。

## 统计展示规则

### 双榜并存

首版固定同时展示：

1. 完成率榜：以 `实际值 / 目标值` 为核心口径。
2. 绝对产出榜：以当日实际完成总量为核心口径。

不强行把双榜合成一个“综合总榜”。

### 横向对比

1. 范围固定为同部门。
2. 首版默认实名展示。
3. 首版不按岗位或模板分组。

### 纵向对比

至少覆盖：

1. 今日 vs 昨日
2. 本周 vs 上周
3. 本月 vs 上月
4. 实际值 vs 当前目标值

## P1 风险与强制约束

### 风险结论

当前冻结口径同时满足以下条件：

1. 员工个人补充目标无需审批。
2. 个人补充目标直接进入统计。
3. 普通员工可见实名横向榜。
4. 全部门统一榜单，不按岗位或模板分组。

这会带来显著的公平性争议风险。如果直接输出单一总榜或单一总分，容易出现：

1. 不同员工自定义不同难度目标，横向不可比。
2. 公共目标完成一般，但靠个人追加目标拉高总产出。
3. 榜单结果无法被主管和员工一致解释。

### 首版强制约束

为避免带着未关闭的 `P1` 口径进入实现，设计上必须补以下规则：

1. 榜单至少拆成 `公共目标贡献` 和 `个人补充目标贡献` 两个可解释维度。
2. 页面上必须能看出个人结果来自哪一类目标来源，而不是只给一个混合总量。
3. 部门日报中必须单独标记个人补充目标对日报结果的贡献，避免管理层误读为公共目标执行结果。

若后续实现阶段不能满足上述 3 条，本主题不得作为“无未关闭 P1”放行。

## 复用与兼容策略

### 当前复用策略

1. 导航上继续复用当前 `目标&计划 -> 目标&计划总览` 承载关系。
2. 页面路由短期继续复用 `/performance/goals`。
3. 首版更偏向在现有目标地图承载页上演进为目标运营台，而不是先新建多套路由。

### 兼容边界

本设计阶段不冻结以下技术细节：

1. 是否完全复用现有 `goal` 资源作为唯一后端主资源。
2. 是否需要为日报、统计快照、周/月计划拆出新资源。
3. 是否需要新权限键或复用现有权限键扩展。

这些问题留到实施设计和实现计划阶段处理，但不得违背本页业务边界。

## 实施优先级建议

若后续进入实现计划，建议按以下主链优先级推进：

1. 主管总览和目标下发入口。
2. 员工当日填报入口。
3. 自动统计与双榜展示。
4. 自动日报生成与发送控制。
5. 规则和模板配置。

不建议先做大量低频配置页或复杂的员工独立页面。

## 验收标准

首版设计通过后的实现验收至少应覆盖下列结果。

### 业务闭环验收

1. 主管可配置本部门时间节点并下发周/月/日目标。
2. 员工可查看下发目标并填写当日实际完成值。
3. 未在截止时间前填写的数据按 `0` 进入统计。
4. 系统可生成完成率榜和绝对产出榜。
5. 员工可查看同部门实名横向榜和个人纵向趋势。
6. 系统可自动生成部门日报并在系统内留痕。
7. 系统可按设定时间自动推送日报，主管可拦截或延期。

### 解释性验收

1. 页面上能区分公共目标与个人补充目标的贡献。
2. 日报中能解释榜单和汇总结果的来源构成。
3. 横向榜单和纵向趋势的统计口径对用户可解释。

### 范围边界验收

1. `班主任化` 仍保持独立二级目录，不被吞并到目标运营台。
2. `目标&计划` 一级导航不继续增胖为多个二级路由集合。
3. 首版不强制要求复杂审批、补填和申诉能力上线。

## 待进入下一阶段前确认的技术问题

本设计已经冻结业务方向，但在进入实现计划前，仍需专门做技术方案确认：

1. 当前 `goal` 资源是否足以承载“日/周/月计划 + 日报 + 统计快照”。
2. 权限键如何从现有 `performance:goal:*` 扩成主管/员工/管理员三层动作。
3. 现有 `/performance/goals` 页面是增量改造还是切成壳层 + 子组件。
4. 外部群推送采用什么基础设施落地，以及失败重试如何处理。

这些问题属于实现计划阶段，不属于本轮业务冻结阻塞。

## 结论

本轮设计正式把 `目标&计划` 首版冻结为一个 `部门日运营目标系统`。它的核心不是管理静态目标条目，而是围绕部门每日经营节奏完成：

`目标制定 -> 结果回填 -> 对比分析 -> 自动汇报`

后续若进入实现，必须继续遵守本设计中对角色边界、时间规则、榜单口径和 `P1` 公平性风险的约束，不能只做表面 UI 改名或简单列表增强。
