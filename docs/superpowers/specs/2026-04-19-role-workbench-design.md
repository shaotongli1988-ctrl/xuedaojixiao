# 2026-04-19 角色工作台设计

## 背景

当前系统里的业务页面已经不少，但入口分散在不同模块中。对用户来说，最大的问题不是“没有页面”，而是：

1. 不知道今天先处理什么。
2. 知道要处理，但不知道应该进哪个模块。
3. 同样叫“工作入口”，员工、经理、HR 实际需要看的内容完全不同。

本轮讨论后的目标不是再造一套新业务系统，而是在现有模块之上增加一个统一的 `角色工作台`，把真正需要个人处理的事项收口成一个更顺手的入口。

## Before Starting Record

- Scope:
  - 设计一个与现有业务模块并列的 `角色工作台`。
  - 明确首页结构、任务卡口径、角色差异、跳转策略和首批接入范围。
- Non-goals:
  - 不在本阶段重做现有业务页面。
  - 不在本阶段建设统一消息中心。
  - 不在本阶段建设新的审批平台或统一流程引擎。
  - 不在本阶段替代原有模块入口。
- Affected modules:
  - 工作台入口与首页壳。
  - 东西领用相关入口。
  - 数据填报相关入口。
  - 行政协同相关入口。
  - 现有“我的事项/我发起的事项”承载页面。
- Acceptance criteria:
  - 用户打开首页能快速看懂“现在该干什么”。
  - 不同角色使用同一套页面骨架，但内容明显不同。
  - 首批接入事项都能形成“出现任务 -> 点进去处理 -> 状态变化 -> 返回可见结果”的闭环。
  - 工作台上线后不影响原有模块入口使用。
- Impact surface:
  - 导航入口。
  - 首页信息架构。
  - 任务聚合规则。
  - 角色裁剪规则。
  - 页面跳转与筛选拼装规则。
- Source of truth:
  - 以本设计文档和本轮用户确认结论为准。
- High-risk surfaces:
  - 角色可见范围。
  - 待办聚合准确性。
  - 跳转是否能到达可处理位置。
- Required delivery evidence:
  - 设计稿冻结。
  - 首批接入清单明确。
  - 页面区块与任务模型明确。
  - 验收标准明确。
- Key assumptions and constraints:
  - 当前已有业务页可复用。
  - 首版优先做轻聚合，不强行统一底层流程。
  - 首批只接“责任人明确、动作明确、状态明确、能跳转”的事项。

## 决策结论

本轮已冻结以下结论：

1. 方案方向固定为 `角色工作台`，不是“个人中心”也不是“新业务模块”。
2. 工作台采用 `角色主框架 + 权限控制内容显隐` 的混合策略。
3. 工作台作为 `并列入口` 存在，先不替代老页面。
4. 首页主分组方式固定为 `按动作类型分组`，不按业务域分组。
5. 首版动作桶固定为：
   - `待填写`
   - `待提交`
   - `待审批`
   - `待确认`
6. 首版目标角色固定为：
   - `员工`
   - `经理`
   - `HR`
7. 首版内容范围固定为：
   - `动作型事项`
   - `我的结果`
8. 首页每条任务卡必须统一包含：
   - 动作类型
   - 业务标签
   - 标题
   - 时间信息
   - 状态补充
   - `去处理` 按钮

## 产品定位

`角色工作台` 的定位固定为：

`面向个人日常动作分发的统一处理入口`

它只负责三件事：

1. 把真正需要当前用户处理的事项聚起来。
2. 按角色和动作类型排好优先级。
3. 把用户快速送到现有可处理页面。

它不负责：

1. 重新承载业务详情页。
2. 重做底层流程。
3. 统一改造所有模块的数据模型。
4. 取代原系统所有入口。

一句话讲，工作台是 `分发层`，不是 `第二套业务系统`。

## 为什么按动作类型分组

首页固定按“人现在要做什么”来分，而不是按“这事属于哪个部门模块”来分。

原因有三个：

1. 用户进首页时，最关心的是“我先做哪件事”，不是“我该进哪个系统”。
2. 一个角色当天的待办往往跨多个业务域，如果先按业务域分，会让用户继续来回跳模块。
3. 后面新增新业务时，只要能挂进现有动作桶，就不用重做首页结构。

所以首页主视图只回答一个问题：

`你现在有哪些事要动手？`

## 页面骨架

首页固定分成 6 个区块，但同一块里显示什么，按角色裁剪。

### 1. 角色头部

作用是让用户一进来就知道自己现在的压力点。

建议承载：

1. 当前角色身份。
2. 待处理总数。
3. 今日截止或逾期摘要。
4. 一个最重要提醒。

示例：

1. 员工：你有 5 项待处理，2 项今日截止。
2. 经理：你有 8 项待审批，3 项团队事项待跟进。
3. HR：你有 12 项待收集，4 项超期未补。

### 2. 待我处理

这是首页主区域，也是工作台最核心的区块。

固定按 4 个动作桶分组：

1. 待填写
2. 待提交
3. 待审批
4. 待确认

每组只展示最急、最重要的几条，并提供 `查看更多`。

这里不放纯通知、不放说明文案、不放统计图表充数，只放 `下一步真要动手的事`。

### 3. 我的事项

这个区块承载“我发起的”“我提交过的”“我关心结果的”事项。

它不是当前动作区，而是结果回看区。

建议承载：

1. 我发起的东西领用。
2. 我提交的数据填报。
3. 我发起的协同事项。
4. 我的绩效或其他个人结果进度。

### 4. 角色专区

这是同一套骨架下做角色差异化的核心位置。

示例：

1. 员工：
   - 常用填报入口
   - 个人材料入口
   - 近期个人提醒
2. 经理：
   - 团队待办
   - 超期名单
   - 团队数据概览
3. HR：
   - 收集进度
   - 缺交名单
   - 异常数据
   - 批量处理入口

### 5. 快捷入口

放高频但不一定是待办的主动入口。

示例：

1. 发起领用
2. 发起协同
3. 查看制度
4. 查看我的记录

这里解决的是“我主动想进某个功能”，不是系统推送的待办。

### 6. 风险提醒

这块只做少量高价值提醒，不做消息中心。

建议只承载：

1. 今日截止
2. 已逾期
3. 被退回
4. 缺关键材料

它的目的不是信息全，而是避免漏事。

## 任务卡模型

工作台首版不按“模块卡片”设计，而按“任务卡片”设计。

每一条任务卡固定使用统一结构：

1. `动作类型`
   - 待填写 / 待提交 / 待审批 / 待确认
2. `业务标签`
   - 例如：东西领用、数据填报、行政协同、绩效
3. `标题`
   - 用户一眼能看懂要处理什么
4. `时间信息`
   - 截止时间，或今日截止 / 已逾期 / 本周内
5. `状态补充`
   - 例如：缺 2 项、退回 1 次、等待上级确认
6. `处理按钮`
   - 统一文案为 `去处理`
7. `跳转信息`
   - 直达具体处理页，或带筛选条件跳列表页

这样做的价值是：

1. 首页只讲动作，不讲系统结构。
2. 新业务未来只要接入任务池，不需要重做页面框架。
3. 同一角色看到的待办能跨业务统一排序。

## 角色差异策略

首版固定支持三类角色：`员工 / 经理 / HR`。

策略不是做三套页面，而是：

`同一套页面骨架，不同的任务内容和角色专区`

### 员工

员工视角只聚焦“我自己要做的事”：

1. 我来填写。
2. 我来提交。
3. 我来确认。
4. 我查看我自己的结果。

员工首页要尽量少判断、少绕路，点进去就能处理。

### 经理

经理视角聚焦“我来决策、我来催办、我来看团队进展”：

1. 待审批。
2. 待确认。
3. 团队谁没填、谁超期、谁退回。
4. 我负责范围内事项的结果进度。

经理首页重点不是自己填很多内容，而是快速发现卡点并作决策。

### HR

HR 视角聚焦“我来收、我来审、我来补、我来收口异常”：

1. 待收集。
2. 待审核。
3. 待补录。
4. 异常项与整体结果。

HR 首页要强调批量处理和异常收口能力。

## 首批接入范围

首批只接 `有明确责任人、有明确动作、有明确状态、能直接跳转处理` 的事项。

不满足这四条的，不进第一批。

### 员工首批接入

建议先接：

1. 东西领用
   - 发起申请
   - 补资料
   - 确认领用
   - 确认归还
2. 数据填报
   - 待填写
   - 待提交
   - 被退回重填
3. 行政协同
   - 分到我的待办
   - 需要我确认的事项
4. 我的结果
   - 我的申请进度
   - 我的填报状态
   - 我的协同状态

### 经理首批接入

建议先接：

1. 审批类事项
   - 下属提交后需要经理审批
2. 确认类事项
   - 需要经理确认的领用、流程节点、协同节点
3. 团队填报跟进
   - 谁未填
   - 谁退回
   - 谁临近截止
4. 团队结果
   - 我发起或我负责范围内事项的进度

### HR 首批接入

建议先接：

1. 收集类事项
   - 谁未提交
   - 哪些数据缺失
2. 审核类事项
   - 提交后需要 HR 校验或审核
3. 补录类事项
   - 需要 HR 修异常、补数据、补材料
4. 结果类事项
   - 整体进度
   - 异常名单
   - 退回情况

## 首批不接入的内容

为了避免首版做成大杂烩，以下内容明确不进第一批：

1. 统一消息中心。
2. 统一审批平台。
3. 纯查询页。
4. 纯介绍页。
5. 纯统计展示但没有明确动作的页面。
6. 无法准确定位责任人或无法准确跳转的模糊事项。

## 跳转策略

工作台不负责自己处理业务，只负责把人送到正确位置。

固定按以下 3 条执行：

### 1. 能直达处理页，就直达

如果当前事项能定位到具体单据、具体表单、具体协同页，则 `去处理` 直接进入实际处理页面。

这是首选策略，因为最省用户操作。

### 2. 不能直达处理页，就带筛选条件跳列表页

如果现有系统只能先进入列表页，那么工作台必须自动带好筛选条件，例如：

1. 只看待我审批。
2. 只看待我填写。
3. 只看待我确认。
4. 只看我发起的。
5. 只看已逾期。

这样用户虽然先进列表，但不需要再自己找一遍。

### 3. 没有动作的内容，不进入“待我处理”

没有明确下一步动作的内容，只能进入：

1. 快捷入口
2. 我的事项
3. 角色专区

不能混进首页主待办区。

## 接入节奏

首版建议按 3 个阶段推进：

### 第 1 批

每个角色先接 2 到 3 个最刚需、最容易闭环的入口，优先证明：

1. 任务能出现。
2. 点进去能处理。
3. 处理完状态会变化。

### 第 2 批

补齐：

1. 我的事项
2. 团队结果
3. 更多同类型待办接入

### 第 3 批

再补：

1. 风险提醒增强
2. 异常预警增强
3. HR 批量处理能力

这样推进的好处是：

1. 先把工作台做成“真能用”。
2. 避免一开始就想统一全系统。
3. 降低错误聚合和错误跳转的风险。

## 与现有系统关系

工作台和现有页面的关系固定为：

1. 工作台是并列入口，不替代原有业务入口。
2. 现有模块继续保留原路径和原入口。
3. 工作台优先复用现有处理页、列表页和状态页。
4. 首版不强行统一现有模块内部的数据结构和状态流。

这意味着首版主要是 `入口层和分发层优化`，不是业务重构。

## 验收标准

首版是否成立，按以下 6 条验收：

### 1. 首页可读性验收

用户打开工作台后，应在 5 秒内看懂：

1. 我有多少待处理事项。
2. 哪些是今天最急的。
3. 我下一步该点哪里。

### 2. 跳转有效性验收

所有首批接入任务都必须满足：

1. 点 `去处理` 能进入可处理位置。
2. 不出现进入后还要自己重新搜索一次的情况。

### 3. 角色差异验收

员工、经理、HR 必须满足：

1. 页面骨架相同。
2. 看到的任务内容明显不同。
3. 角色专区内容明显不同。

### 4. 首页纯度验收

首页 `待我处理` 区只放真动作，不被以下内容污染：

1. 无动作通知。
2. 介绍文案。
3. 无法处理的统计堆砌。

### 5. 首批闭环验收

至少要证明首批接入事项存在闭环：

1. 任务出现。
2. 用户能处理。
3. 处理后首页状态变化。
4. 用户能在 `我的事项` 中看到结果。

### 6. 兼容性验收

工作台上线后：

1. 原有模块入口仍可正常使用。
2. 原有页面不因工作台接入而失效。
3. 工作台只是更高效入口，不是强替代。

## 第一版成功标准

如果第一版上线后能达到下面三点，就算方向跑对了：

1. 员工少找页面。
2. 经理少漏审批、少漏跟进。
3. HR 少漏收集和异常事项。
