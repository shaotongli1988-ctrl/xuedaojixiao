# docs 文档入口

这个目录是项目文档总入口。

如果你是第一次接手这个基座，不要一上来把所有文档都看完，先按顺序看。

## 推荐阅读顺序

### 第一组：新人先看

1. `01-新人先看.md`
   - 新人最快入口
   - 告诉你先看哪个 skill
   - 告诉你小修复、新功能、跨端任务怎么走

2. `02-功能开发怎么走（超短版）.md`
   - 功能开发 1 分钟速看

3. `04-怎么跟AI说需求.md`
   - 不会写提示词时，直接复制模板用

### 第二组：开始做功能前再看

4. `03-功能开发怎么走.md`
   - 完整版功能开发流程

5. `05-示例：新增一个普通功能怎么走.md`
   - 用“优惠券管理”这个真实例子，带你走一遍从接需求到交付

6. `06-阶段化开发怎么协作.md`
   - 告诉团队以后默认怎么和 AI 分阶段协作
   - 重点看什么时候要人工确认、改需求时怎么回改文档

7. `07-任务执行卡模板.md`
   - 给每个阶段化任务配一张统一执行卡
   - 重点看当前阶段、待确认点、下一步动作怎么记录

### PostgreSQL / 后端开发必看

- `00-system/PostgreSQL后端开发规范.md`
  - 仅在项目数据库使用 PostgreSQL 时启用的后端开发规范
  - 改 entity、controller、service、分页、排序、QueryBuilder 前建议先看

### 第三组：开发过程中随时看

8. `checklists/`
   - `01-开发前检查.md`：开发前检查
   - `核心模块修改前检查.md`：修改核心模块、高风险流程、公共结构前先看
   - `02-提交前检查.md`：提交前检查
   - `03-交付前检查.md`：交付前检查

### 第四组：项目长期文档

9. `00-system/`
   - 放系统长期说明
   - 适合看系统全貌、模块边界、功能入口
   - `系统总说明怎么写.md`：教你怎么写一份让团队成员和 AI 都能快速恢复上下文的系统总说明
   - `项目与base差异怎么记录.md`：教你怎么记录当前项目已经偏离 base 的关键差异

10. `01-modules/`
   - 放模块级文档
   - 适合看某个具体功能是怎么设计、怎么开发、怎么交付的

## 常见场景怎么找文档

### 1. 我是新人，刚接手项目
先看：

- `01-新人先看.md`
- `02-功能开发怎么走（超短版）.md`
- `04-怎么跟AI说需求.md`
- `05-示例：新增一个普通功能怎么走.md`
- `06-阶段化开发怎么协作.md`
- `07-任务执行卡模板.md`
- `03-功能开发怎么走.md`
- `checklists/01-开发前检查.md`

### 2. 我要做一个小修复
先看：

- `01-新人先看.md`
- 对应端的 skill
- `checklists/02-提交前检查.md`

### 3. 我要做一个新功能
先看：

- `01-新人先看.md`
- `02-功能开发怎么走（超短版）.md`
- `03-功能开发怎么走.md`
- `05-示例：新增一个普通功能怎么走.md`
- `06-阶段化开发怎么协作.md`
- `07-任务执行卡模板.md`
- 对应端的 skill
- 模块文档目录：`01-modules/模块名/`
- `checklists/01-开发前检查.md`

如果是长期维护项目里的复杂功能、高影响修改、核心模块修改，再加看：

- `00-system/系统总说明怎么写.md`
- `00-system/项目与base差异怎么记录.md`
- `checklists/核心模块修改前检查.md`
- `skills/cool-project-stack/references/code-organization.md`（代码组织、模块复用、中文注释要求）

### 4. 我做的功能会拆成多期，文档怎么放
先看：

- `03-功能开发怎么走.md`
- 模块文档目录：`01-modules/模块名/`
- 对应模块下的 `README.md`
- 当前版本对应的需求 / 设计 / 实施文档
- 当前版本对应的任务执行卡

记住：

- 多期功能默认放在同一个模块目录下，不要把每一期散落到 `docs/` 根目录
- 模块 `README.md` 必须负责串联版本规划、阅读顺序、当前版本入口
- 写模块入口时，优先参考模板：`skills/cool-project-stack/templates/module-readme-template.md`
- 版本文档按版本拆，长期共用内容留在模块级文档
- 新版本如果基于旧版本增强，优先写“继承什么、调整什么”，不要整份复制重写

### 5. 我要准备交付
先看：

- `checklists/03-交付前检查.md`
- 当前模块下的 `07-test-checklist.md`
- 当前模块下的 `08-delivery.md`

## 文档怎么分

### `01-新人先看.md`
给新人看的最短入口。

### `02-功能开发怎么走（超短版）.md`
给新人 1 分钟速看，先知道接到功能后第一步做什么。

### `03-功能开发怎么走.md`
告诉你接到一个功能后，第一步做什么、第二步做什么、什么时候先写文档、什么时候可以直接改代码。

### `04-怎么跟AI说需求.md`
给不会写提示词的人直接复制用，照着填需求就行。

### `05-示例：新增一个普通功能怎么走.md`
给新人看真实案例，知道普通功能在真实工作里怎么推进。

### `06-阶段化开发怎么协作.md`
告诉团队成员和 AI 做功能时，默认会按什么阶段推进、哪些地方要人工确认、改需求时为什么要先回文档。

### `07-任务执行卡模板.md`
给团队一张可直接复制的执行卡模板，用来记录当前阶段、待确认点、下一步动作和回改范围。

### `checklists/`
给开发过程用的短清单，避免漏步骤。

### `00-system/`
放系统级长期文档：
- 系统目标
- 模块划分
- 功能入口
- 长期维护说明
- 当前项目与 base 的关键差异怎么判断

### `01-modules/`
放模块级文档：
- 模块 README（背景、版本规划、阅读顺序、当前版本入口）
- 长期约束说明
- 稳定后的核心实体说明或模块总览
- 各版本的需求、设计、数据库、API、开发实施、测试、交付文档
- 各版本或各任务自己的执行卡

如果一个功能会拆成多期，默认放在同一个模块目录下，由模块 `README.md` 串联所有版本，不要把每一期文档散落到别处。

## 一句话记住

- 看不懂先看 `01-新人先看.md`
- 开工前看 `01-开发前检查.md`
- 提交前看 `02-提交前检查.md`
- 交付前看 `03-交付前检查.md`
